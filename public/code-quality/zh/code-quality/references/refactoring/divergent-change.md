# Divergent Change

## 它是什么

Divergent Change 指的是这样一种 smell：同一个模块反复因为很多互不相关的原因而被修改。每当数据库 schema 变动，你都要改这个模块；每当报表格式变动，你又改它；每当业务规则迁移，你还是回到同一个文件里，去碰另一部分。这个模块已经变成一个交汇点，几种彼此独立的关注点凑巧共存于此。一个文件，很多不相关的修改理由。

它之所以是 _change preventer_，是因为这些混杂的关注点让每次修改都比应有的更危险：改报表格式的代码时，你得先穿过持久化代码和验证代码；某个关注点上的改动还可能误伤另一个关注点。这个模块也更难理解，因为你要想“这个模块是干什么的”，必须同时在脑子里放好几个互不相干的主题。

## 信号

检测问题时要问的是：_这个模块有多少种不同的修改理由？_ 如果你看 commit history（或者只是想象未来会有哪些请求），发现这个文件会因为“新的税务规则”“新的导出格式”“新的存储后端”“新的验证策略”而被改，那就是四种理由、四个关注点、一个模块。再看文件内部的信号：某个 class 或 module 的不同部分之间没有共同的数据或逻辑，方法分成几组，但这些组彼此从不调用，也不碰同一批字段。

它和 [shotgun-surgery.md](./shotgun-surgery.md) 的对照很清楚：那里是一种改动打到很多模块；这里是很多种改动打到一个模块。它们互为镜像。

## 它在告诉你什么

Divergent Change 在提醒你 **拆分职责**。这个模块在做好几件事，而每一件都应该放到自己的模块里，这样它们才能独立、隔离地变化。通常的 refactoring 是：

- 先用 [extract-function.md](./extract-function.md) 把有凝聚力的片段隔离出来，然后再做 Extract Class / Extract Module，让每个关注点有自己的归宿。
- 用 [move-function.md](./move-function.md) 把行为移到拥有该关注点的模块里。
- 把纠缠在一起的层分开 - 例如，把同时混了业务规则、持久化和格式化的模块拆成三个，这样格式化变更就只会触碰格式化模块。

拆开之后，每个结果模块都只沿着一个变化轴变化：你只会在持久化变化时改持久化模块，只在规则变化时改规则模块。这就是凝聚性的实际意义。

## 与 SRP 和 Shotgun Surgery 的关系

Divergent Change 本质上就是 **Single Responsibility Principle** 的 smell 版。Uncle Martin 对 SRP 的表述 - “一个模块应该只有一个修改理由” - 正好就是 Divergent Change 违反的性质：这个模块有很多修改理由。所以修法也就是 SRP 的修法：找出驱动代码变化的不同角色或力量，并给它们各自的模块。见 [../design-principles/solid.md](../design-principles/solid.md)。

它也是 **Shotgun Surgery 的反面**，两者需要一起看，因为它们的修法方向相反：

- **Divergent Change** → 一个模块、很多修改理由 → **拆开**。
- **Shotgun Surgery** → 一种修改理由、很多模块 → **收拢**。

过度纠正任一方向，都会制造另一种 smell。拆得太狠，一个改动就会开始需要在新碎片之间四处修改（Shotgun Surgery）。收得太狠，合并后的模块又会因为很多不同理由而变化（Divergent Change）。两种 smell 指向的目标其实一样：模块边界应该跟真实的变化轴对齐，这样一种变化就对应一个编辑点。拿不准时，就先找出这段代码历史上真正因为哪些原因变化过，让这些理由去定义边界。
