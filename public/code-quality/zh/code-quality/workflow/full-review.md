# 完整审查（Full Review）——代码质量

重量级、系统性的审查。**仅由用户触发**——切勿自行进入此模式。比快速审查更慢、更彻底，采用分阶段阅读和自我验证。

## 触发条件（Trigger）

仅当用户明确要求"完整审查"、"架构审查"、"设计审查"或"重构评估"时。

## 步骤（Steps）

### 1. 上下文摄入（Context intake）

广泛阅读之前，先记录：

- 审查目标——实际评判的内容（模块、子系统、重构计划、配置）。
- 必读内容——目标的核心代码、配置或测试。
- 可选上下文——可能解释某个决策的相邻代码。
- 未解决的问题——不确定、可能需要向用户确认的内容。

### 2. 分阶段阅读（Staged reading）

按问题领域加载参考文档，而非一次性全部加载。先读取最近的 `AGENTS.md`/`CLAUDE.md`，然后是偏好（`.agents/preferences/code-quality.md` 或 `.agents/preferences/code-quality/index.md`）。仅在具体信号指向时，才拉取原则、模式、重构或范式文档。分阶段阅读代码——先读目标，仅在发现项需要时再扩大范围。

### 3. 系统性矩阵（Systematic matrix）

逐项检查以下维度，为每条发现项记录证据：

- 变更方向和成本——接下来可能发生什么变更，当前结构使其变得多昂贵。
- 原则张力——DRY vs KISS、抽象 vs 重复、灵活性 vs YAGNI。指出权衡，而非教条地选择。
- 模式合理性——对于每个命名模式，确认它所管理的变化点确实存在。
- 坏味识别——过长函数、重复知识、基本类型偏执、依恋情结、霰弹式修改、发散式变更、薄包装。
- 范式匹配——命令式/面向对象/函数式核心/数据导向/状态机是否匹配问题，还是与其对抗。
- 测试覆盖——行为是否被充分固定以安全重构。
- Agent 配置——若适用，评判 `AGENTS.md`/`SKILL.md`/工作流配置是否存在矛盾、死规则和清晰度问题。

### 4. 自我验证高严重度发现项

对于每条高严重度的发现项，重新检查证据，说明置信度，并主动考虑误报的可能性：该结构是否可能是故意的，或受限于你尚未看到的变更原因？对未经受住此检验的发现项，降级或删除。

### 5. 确认暂停（Confirmation stop）

在推荐或执行任何跨文件重构、架构迁移或批量变更之前，停下来询问用户。呈现方案及其成本，让用户决定。

## 输出格式（Output Format）

```text
发现项

- [严重度, 置信度] 路径:行号 标题
  事实：可观察的证据。
  影响：变更成本、可读性、正确性、可测试性。
  判断：原则、模式、坏味、范式不匹配或配置坏味。
  证据：支持、反证以及剩余的不确定性。
  建议：最小充分的变更。
  验证：命令/检查，或说明为何不需要。

开放式问题
- 需要用户或项目确认的事项。

备注
- 被降级、有意为之或由工具处理的项，附上原因。
```

按类别分组发现项。明确说明你降级了哪些项以及原因。

## 停止规则（Stop Rules）

- 未经明确要求，不要修改代码。
- 跨文件重构、迁移和批量变更前须确认暂停。
- 不要为了满足某个原则而制造发现项，也不要以相似性为由进行抽象。
- 不要将偏好表述为普适的工程真理。
- 在每条发现项中区分事实、判断、偏好和建议。
