# 测试原则

测试套件同时以两种方式证明自己的价值：捕捉回归，以及记录代码应该做什么。两者都遵循同一条纪律——**让测试与行为耦合，与结构解耦**。Kent Beck 说得直接：测试应该对*行为*变化敏感，对*结构*变化不敏感。下面的每条原则都是从这一句话衍生出来的。

本文档语言无关。它陈述原则和取舍；违反原则的反模式有专门的目录 [test-smells](test-smells.md)，语言特定的机制（pytest fixture、参数化、mock）放在对应的工程技能中。

## 测试行为，而非实现

测试应该断言一个单元*从外部看做了什么*——返回值、抛出的异常、记录的副作用——而不是它内部怎么做的。原因在于成本：可观察行为是你承诺的契约，所以以此为准的测试只在违背承诺时才失败，保留行为不变的重构则继续保持通过。断言内部调用、私有字段或协作者调用顺序的测试会在每次重构时失败，同时证明不了任何正确性——它只是把实现复述给自己看。

这是对"AI 写的测试改了代码就得重写一大堆，而且大部分是些不该测的细节"这个抱怨的直接回答。那种痛苦几乎总是[脆弱测试](test-smells.md)和[变更检测器测试](test-smells.md)——镜像结构而非行为的测试。解决方法是"断言结果，只在真正的边界用 mock"，而不是"少写测试"。

**边界情况。** 偶尔行为*就是*一个内部选择——你必须验证读请求走了缓存而不是数据库，或重试确实发生了。这些测试是合理的，但应该很少，并且以保证命名（"重复读走缓存"）而不是以方法命名。如果大部分测试都需要探入内部，说明设计在要求把依赖作为参数注入，而不是在内部直接获取。

## 期望属性：好测试优化什么

Kent Beck 的 *Test Desiderata* 列出了测试可以具备的十二条属性。它们不是一张需要全部满足的清单——有几条相互冲突——而是一套用来判断某个测试应该优化什么的词汇：

- **隔离**——测试互不影响；任意顺序结果一致。
- **可组合**——可以独立测试某一个维度。
- **确定性**——相同代码、相同结果、每次运行都一样。
- **快速**——成本够低，可以频繁运行。
- **易写**——相对于被测代码，编写成本低。
- **可读**——读者能看出测了什么以及为什么重要。
- **行为敏感**——对行为变化敏感；有 bug 应该击穿测试。
- **结构不敏感**——保留行为不变的重构不应该击穿测试。
- **自动化**——无需人工介入即可运行。
- **具体**——失败指向单一原因。
- **可预测**——通过就说明代码可以安全发布。
- **鼓舞信心**——通过时建立真正的信心。

这些属性相互冲突，这正是重点。*可预测*把你拉向真实的全栈测试；*快速*和*具体*把你拉向小型隔离测试。面向程序员的单元测试有意牺牲一部分*可预测*和*鼓舞信心*，来换取*易写*、*快速*和*具体*；端到端测试则做相反的取舍。知道一个测试*为了*哪些属性，就不会要求每个测试都满足全部十二条。

## 覆盖率是找盲点的，不是目标

行覆盖率衡量哪些行被执行了，而不是有没有做检查——一个调了函数但不做断言的测试报告完全覆盖，却什么也没验证。Fowler 的立场是可操作的：覆盖率对*负面*信号有用（完全没被任何测试触及的代码是真正的缺口），但作为*正面*目标则具有误导性——高百分比说明不了断言质量。一旦覆盖率百分比成了目标，它就会催生出为了涨数字而写的低价值测试。

更有力的信号是**基于突变的**：如果你把 `+` 改成 `-`、翻转一个边界、或删掉一行，有没有测试失败？存活的突变是真正的漏洞——一个没有被任何断言固定的行为——这种漏洞是"被覆盖但没断言"的行永远暴露不了的。你不需要专门的突变测试工具也能这么想；写测试的时候问一句"这个测试能抓住哪种突变？"就足以发现什么也没检查的断言。

这样一来，目标就从"覆盖每一行"变成了"**用尽可能少的测试固定每一个重要的行为**"。两个测试通过不同的表层数据走过同一路径，往往只提供一个测试的保护量，维护成本却是两个。当两个案例杀死相同的突变，而且两者都没有记录对方遗漏的独特场景时，保留更清晰的那个，合并或删除另一个——这就是用更少的测试实现相同覆盖率的方法。合并前的检查与下文相同：相同的突变分数只是重叠的信号，不能单凭它证明两个测试可以互换。

## 更少更强的测试

测试多不等于更安全。每个测试固定一个独特行为时是资产，按比例反之为负债。三个习惯保持精简：

- **每个测试一个行为。** 断言一个保证的测试因一个原因失败，标题就是原因。走过五个行为的测试（Eager Test，是[晦涩测试](test-smells.md)的一种形式）会模糊地失败，且抗拒改动。宁要多而小的行为测试，不要少而杂的——但"小"指*行为范围窄*，不是*跟某个函数的内部实现耦合*。
- **参数化等价案例。** 一个行为在多个输入上成立时，把案例写成数据，而不是复制粘贴。每个案例单独报告，失败时直接告诉你哪个输入出问题，案例本身像表格一样清晰。这是边界和等价类测试的正确工具；案例需要不同 setup 或不同断言时不适用——强行塞进一个参数化函数体只会藏起分支逻辑。
- **删死重要讲证据。** 一个不给其他测试已经覆盖的逻辑增加任何保护的测试，只有维护成本。这种情况确实会发生：Bas Dijkstra 对一组 LLM 生成的测试做突变测试时（n=23），有 4 个测试对行覆盖*和*突变覆盖都毫无贡献——这是一个具体案例的数字，不是可以推广的固定比例。但"杀死相同突变"不等于冗余的确证：删之前确认这个测试不提供独特的行为描述、不提供更好的失败定位、也不覆盖其他测试遗漏的某个突变。突变工具不能枚举所有可能的缺陷，所以相同的突变分数是重叠的强信号，而不是保证。确认冗余之后再删，不要光靠分数吻合就删。

## 测试中 DAMP 优先于 DRY

生产代码倾向 DRY：消除重复的*知识*，让一条规则只在一个地方维护。测试代码倾向另一个方向——**DAMP**（Descriptive And Meaningful Phrases，描述性且有意义的表达）——这是有意为之。原因很简单：*测试没有测试*，它的正确性靠人来读，所以可读性比去重更重要。一个直接把 setup 内联、直接断言期望的测试可以一眼看懂；一个把所有东西都路由到共享辅助函数、循环和遥远的 `setUp` 的测试，读者得先在脑子里拼装场景才能判断。

这不为复制粘贴开绿灯。精确的调和方式是：消除损害*维护*的重复（值对象构建器、去除噪音的辅助函数），容忍有助于*阅读*的重复（测试体里写清楚具体的输入和预期输出）。Meszaros 的[测试代码重复](test-smells.md)针对的是结构型重复——一个行为变化迫使在多处编辑；DAMP 针对的是让每个测试自包含的表层重复。两者一致多于冲突：都希望一个行为的变化只影响少量测试，都希望读者不用离开当前测试就能理解它。

## 隔离和确定性没得商量

一个测试的通过或失败取决于顺序、时机、环境或前一个测试留下的状态，那它不只是自己失败——它腐蚀了人们对整个套件的信任。不被信任的套件会被忽略，这比没有套件更糟。Fowler 说得直接：根除非确定性，不要容忍它。反复出现的根因是测试间共享可变状态、依赖真实外部资源（网络、时钟、文件系统、远程服务）、不受控的异步操作、以及未设种子的随机性。

解决之道：每个测试用干净的新状态，任意运行顺序结果相同；在系统边界用测试替身替换慢或非确定性的依赖；控制时钟和种子，而不是读真实值；绝不让不稳定测试躺在主套件里"以后修"——隔离或修掉，因为一个不稳定的测试就会教会所有人无视失败。Google 的 Small/Medium/Large 测试规模方案在小型测试里*禁止*危险资源（网络、sleep、多线程），这让它们从构造上就快、可并行、不依赖顺序。

## 不要测框架、配置或琐碎代码

每个测试永远消耗注意力，所以一个测试只有在能抓住*你自己的*逻辑里的缺陷时才值得。三类通常做不到：

- **框架和库的行为。** 测试一个经过充分测试的库能不能排序、或 ORM 能不能保存一行，只是重复验证别人的保证，升级时还会炸。在边界上测试*你对库的使用*，而不是库本身。
- **配置和常量。** 读个配置值然后断言它等于同一个字面量，或者断言刚声明的字段存在——这是[变更检测器测试](test-smells.md)——复制了源，源一变它就炸，什么都没抓住。测试*配置驱动的行为*（`timeout` 设成 0 是不是真的抛异常？），而不是值本身。
- **琐碎的机械代码。** Getter、薄薄的透传包装器、格式化器或类型检查器已经保护的数据容器，很少值得专门写测试。如果辅助函数有真正的逻辑值得测，通过*使用它的行为*来测，而不是隔离测——一个工具函数的孤立微测试会把套件耦合到系统其他部分可能根本不依赖的细节上。

误报边界同样重要：有真正可复用逻辑的工具函数（解析器、日期计算、验证规则）*值得*直接测，因为里面的 bug 会扩散。反模式是为了*凑覆盖率*而测那些机械保证的琐碎代码和内部辅助函数，而不是真正承重的逻辑放在哪就测哪。

## 测试即文档

写好的测试是项目能拥有的最可靠的文档之一——它很难在不和代码脱节的情况下保持准确——但弱断言、错误的预期、或跟被保护代码一起同步误改的测试，仍可能在保持通过的同时悄悄漂移，所以这是强倾向而非保证。想了解某个函数用法的读者，应该能打开它的测试，看到真实的调用和预期结果。这是一个设计目标，不是副产品：**Arrange-Act-Assert** 结构让每个测试读起来是准备、执行、验证；每个测试一个行为，让失败指向一件事；名字讲清楚保证（`rejects_expired_token`，而不是 `test_validate_2`）。用接近真实环境的数据——像 `"abc"` 或 `123` 这样的输入可能会让一个真实值本该失败的测试通过。这样写的测试同时就是可运行的示例，这也是测试驱动开发能在一次过程中同时产生设计工具和活文档的深层原因。
