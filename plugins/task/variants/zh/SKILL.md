# Task

使用五个 Task 工具管理持久项目状态。将 Task 文件视为用户拥有的状态，将确定性 Core 视为受管元数据、关系、生命周期、发现、锁定和 WAL 追加的权威。

## 调用

以当前宿主暴露的 Task 工具作为主要接口。在 Codex 和 Claude Code 中优先使用注册的 MCP 工具；在 Pi 中使用原生 Task 工具。当等效宿主工具可用且正常工作时，不要通过 shell 调用 `task-core invoke`。

仅当等效宿主工具不可用或损坏时，或对没有对应 Task 工具的管理命令：`init`、`check`、`rename` 和 `version`，才将 CLI 作为回退使用。在回退执行五种常规操作之一前说明原因。

## 解析 Task

每个新会话开始时无绑定。不要从上一会话推断持久当前 Task。

在每次 Task 工具调用时传入当前绝对工作区 `cwd`。打包的 MCP 服务器从其插件目录启动以保证运行时可移植；因此进程 cwd 不是项目上下文。切换工作树或目录后重新解析 `cwd`。

1. 当用户给出 ID、确切名称、路径、目录名或分支时，调用 `task_find` 或 `task_read`。
2. 当用户仅说"继续"时，使用 cwd、分支、工作树变更和 `task_find` 作为证据。
3. 如果解析有歧义，展示候选项并询问用户。不要按相似度静默选择。
4. 优先读取 `summary`。仅在需要旧/完整 WAL 内容时使用 `detailed`。在需要状态和关系检查时使用 `metadata`。
5. 使用宿主文件工具按需读取返回的绝对 `task_dir` 下的普通文件。Core 有意不盘点 Task 材料。

同一会话中的连续轮次可将已解析的 Task 保留在上下文中；不要持久化该绑定。

## 创建

顶层创建遵循项目 `creation_policy`：

- 在 `strict` 下，仅当用户在当前对话中明确请求或确认创建时才创建；仅对此事实传递 `user_confirmed: true`。
- 在 `permissive` 下，为明显持续性、多阶段或跨会话的工作创建，并在同一轮次告知用户。不要为快速编辑、一次性回答、想法、普通待办事项或尚未选定的工作创建。
- 如果 Core 返回 `project_not_initialized`，询问是否运行 `task-core init`。永远不要在创建中隐藏初始化。
- 在已分配的 Task 树内创建子任务不需要新的顶层确认。对应当全有或全无的同级任务使用单次批量创建。

将 Task body 写为当前事实，而非时间线。将计划、分析、检查点和移交放在 `task_dir` 下的普通文件中，并从 `TASK.md` 链接重要入口点。

## 更新与记录

仅对分支、关系增量、浅层 `extra` 变更和一次生命周期操作使用 `task_update`。使用宿主文件工具编辑 `TASK.md` body 和普通材料。存在等效 Core 操作时，永远不要直接编辑 Core 管理的 frontmatter。

对持久工作活动使用 `task_log`：发现结果、决策、修正、可恢复的里程碑、验证和已验证的协作结果、有意义的用户/外部编辑，以及改变下一步的阻塞项。每个持久结果形成后立即追加，在开始下一个独立工作线之前；绝不推迟到后续里程碑、测试或会话结束。研究或修正结论、其后的实现里程碑、以及其后的验证结果是不同边界上的不同事件。仅合并同一次语义事件中共同形成的事实；不要为了减少调用次数将后续里程碑或验证结果折叠回前面的记录。不要记录命令流、常规读取、临时计划、进行中状态或 Core 已记录的机械变更。会话结束检查点仅填补空白；如果它将成为本会话的唯一记录，说明你跳过了事件边界记录，因此先分别补齐那些不同事件。通过追加而非重写历史 WAL 来修正。

从当前可用的最具体信息设置 actor：先是精确的运行时 `model.id`，然后是 Agent 自身上下文中的精确模型，再是 Agent 能合理说明的最具体模型或模型系列，最后是 `<host>:unknown`。优先使用 `codex:<model>`、`claude:<model>`、`pi:<model>` 或 `codex:<subagent>:<model>` 等形式。没有理由时不要降级已知的精确模型；actor 是基于现有证据尽力准确填充的字段上下文，而非经过身份验证的运行时证明。

所有生命周期转换都需要原因。仅在所有后代和依赖关闭后才正常关闭；仅在用户明确请求时使用 force 并保留被跳过的检查。仅归档已关闭的 Task。仅当用户当前明确同意后取消归档，然后如果需要继续工作则重新打开。

仅通过 `task-core rename` 重命名。先运行 `task-core rename <ref> <name> --dry-run`。如果存在未解析的引用，展示它们且仅在用户同意后才使用 `--allow-unresolved` 继续。

## 分配边界

运行时分配是上下文相关的，不持久存储：

- 分配给 Task X 的 Agent 可写入 X 及其后代；父级和同级为只读。
- 分配给顶层 Task 的 Agent 可写入整个 Task 树。
- 分配给子任务的子代理不写入父级 WAL。父 Agent 记录观察到的结果。
- 此边界适用于 Task 数据，而非普通项目工作文件。

在结束未完成的工作前，追加一条简洁的 WAL 记录，涵盖变更内容、当前阻塞项或决策，以及下一步有用操作。仅当细节不适合一条有用的 WAL 记录时，才创建普通的移交/检查点文件。
