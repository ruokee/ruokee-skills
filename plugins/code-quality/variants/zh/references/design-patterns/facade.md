# 外观模式（Facade）

## 意图（Intent）

为复杂子系统提供一个单一的、简单的接口。外观定义了一个更高级别的入口点，使子系统更易于使用，同时不阻止需要更精细控制的调用方访问底层部件。

## 解决的问题

子系统积累了许多需要按正确顺序协调的类和步骤：获取客户、从订单行创建发票、发布事件、更新账本。如果每个调用方自己编排这些步骤，则协调逻辑被重复，调用代码耦合到内部结构，并且更改工作流意味着编辑每个调用点。外观命名了常见的用例（"为这个订单开具发票"）并在一个地方拥有编排逻辑。

## 结构和参与者

- **外观（Facade）**：暴露一小套高级操作，并委托给子系统对象，对它们的交互进行排序。
- **子系统类（Subsystem classes）**：执行实际工作；它们不了解外观，且保持可直接使用。

外观本身不添加任何新功能——它将现有部件组合成一个便捷的表面。调用方依赖外观；外观依赖子系统。

## Python 惯用实现

模块本身就是一种自然的外观：包的 `__init__.py` 加上精心策划的 `__all__` 暴露少数入口点函数，而实现存在于私有的子模块中。

当用例共享状态或依赖时，客户端类效果很好：

```python
class BillingClient:
    def __init__(self, customers, invoices, events) -> None:
        self._customers = customers
        self._invoices = invoices
        self._events = events

    async def invoice_order(self, order: Order) -> Invoice:
        customer = await self._customers.get(order.customer_id)
        invoice = await self._invoices.create(customer, order.lines)
        await self._events.publish(InvoiceCreated(invoice.id))
        return invoice
```

一个好的外观是一个*深模块（deep module）*：一个小接口面对大量内部复杂性。这种深度使其值得拥有——它隐藏的是真正的工作，而不仅仅是几行代码。

## 何时使用

- 多步骤工作流或第三方 SDK 在多个地方以相同方式使用，你想要一个命名的入口点。
- 你想将调用方与子系统的内部结构解耦，使其能够在稳定表面后演进。
- 你在分层系统，希望每一层向上层呈现最小接口。

## 何时有害

- **隐藏调用方真正需要的复杂性。** 如果调用方必须控制重试、分页、部分失败或事务边界，外观简化了这些反而会迫使它们绕过或变通它。暴露调用方必须推理的内容。
- **浅外观。** 如果它仅将一个调用转发给一个子系统方法，那么它增加了一层和一个名称，却没有隐藏任何东西。代价（一次额外的跳转、一个需要查找的地方）超过了收益。
- **上帝对象。** 外观发展到覆盖系统中的每个操作，成为一个堆积场，耦合了不相关的用例并积累了所有依赖。

## 失败模式

- 外观在其签名中泄露了子系统类型，导致调用方仍然耦合到内部，抽象变成幻觉。
- 错误处理将不同的子系统故障折叠为一个不透明的异常，使调用方无法做出适当响应。
- 外观成为唯一允许的路径，阻止了合法的进阶使用，而本应是便捷的默认路径，同时直接访问仍然可用。

## 与其他模式的关系

[adapter.md](adapter.md) 改变接口以匹配调用方的期望；外观为多个对象定义*新的、更简单的*接口——适配器包装一个事物以适配，外观包装多个事物以简化。外观通常位于 [abstract-factory.md](abstract-factory.md) 之前，或协调由工厂构建的对象。好的外观背后的"深模块"理念与设计原则参考中的信息隐藏原则相关联。比较中介者模式（Mediator），它也集中交互，但让被协调的对象通过它回话，而外观是一个单向的简化入口点。
