# 安全重命名 Task

在每次 Task 重命名前阅读此参考文档。重命名是项目级低频 CLI 操作，而非 `task_update` 字段变更或仅 frontmatter 编辑。

## 理解重命名不变性

重命名保留：

- 不可变的 UUIDv7；
- 顶层创建日期分区；
- 同级/日序号；
- 父级拓扑；
- 状态、归档、关系、body、材料和历史 WAL。

重命名更改：

- `TASK.md.name`；
- 规范化 slug 变化时的叶子 slug；
- Core 能精确确定的项目引用；
- 当前 WAL（追加一条重命名记录）。

它不会在旧路径留下别名或符号链接。它不会重写历史 WAL。如果新名称产生相同的 slug，仅显示名称和 WAL 会变化。

新名称遵循常规 Task 名称约束：NFC、已修剪、无 NUL/换行、最多 40 显示列和 96 UTF-8 字节。Core 派生 slug；不要指定 slug。

## 准备工作现场

在 dry-run 前：

1. 通过完整 ID 或清晰的路径/名称唯一解析目标 Task。
2. 确认重命名在范围内并理解名称应更改的原因。
3. 检查 Git 和工作树状态，使无关编辑保持可见。
4. 识别 Core 无法扫描的外部系统或未跟踪材料。
5. 避免同时运行另一个项目级 Task 操作。

不要先编辑 `TASK.md.name` 或移动目录。这会在不更新引用的情况下造成不匹配。

## 运行 dry-run

从项目/工作树目录运行：

```bash
task-core rename <task-ref> <new-name> --dry-run
```

对名称和路径使用正确的 shell 引号。结构化计划包括：

- Task ID；
- 旧/新名称；
- 旧/新路径；
- 引用更新计数；
- `manual_review` 条目。

在任何写入前审查确切的计划。Dry-run 不会预留目标；Core 在锁下执行期间重新扫描和重新验证。

## 解释引用更新

Core 扫描项目根目录和完整的 Task 根目录，排除 `.git`、`.cache` 和历史 WAL 内容。它考虑支持的文本文件和符号链接。

它可以更新旧绝对路径的精确引用，以及当 Task 在项目内时的精确项目相对路径。解析到被重命名 Task 的符号链接可以被重新定向。

Core 不保证对每种可能的引用格式进行语义解析。动态路径构造、裸 slug、模糊文字、编码路径、二进制文件、不支持的文件类型、外部仓库和远程系统可能仍然未解析。

将 `reference_updates` 视为 Core 计划进行的精确替换，而非所有引用已被找到的证明。

## 处理手动审查

当 `manual_review` 非空时，默认执行会停止并返回 `rename_manual_review_required`。

对每个项目：

1. 打开文件或符号链接而不更改它。
2. 确定它是指此 Task、另一个具有相同 slug 的对象，还是历史文字。
3. 决定是在重命名前更新、重命名后更新，还是有意保留它。
4. 记录 Core 无法执行的任何外部/手动后续操作。

仅在用户显式接受后，才继续处理未解决的项目。该同意授权：

```bash
task-core rename <task-ref> <new-name> --allow-unresolved
```

它不授权删除有歧义的引用或无关编辑。在询问前展示剩余的审查项目和预期后果。

## 执行重命名

如果 dry-run 没有未解决的项目：

```bash
task-core rename <task-ref> <new-name>
```

添加 `--actor` 并提供当前可用的最具体模型信息，遵循与 `task_log` 相同的 actor 规则（基于现有证据尽力准确填充）。

Core 获取项目锁和所有顶层 Task 锁（以稳定顺序），然后重新解析 Task、重新扫描引用、更新受管名称、在需要时移动叶子、更新精确引用，并追加一条重命名 WAL 记录。

不要仅因为路径更改而运行第二次重命名。使用返回的权威 `new_path` 和不可变 ID 进行后续操作。

## 验证结果

成功之后：

1. 通过不可变 ID 读取 Task，确认名称、路径、父级、状态和关系。
2. 从项目目录运行 `task-core check`。
3. 检查返回的 `manual_review` 列表，即使未解析的延续已被授权。
4. 搜索项目和 Task 材料中的过时旧绝对/相对路径和有意的裸引用。
5. 确认符号链接目标及重要的 Markdown/wiki/引用链接。
6. 检查 Git 状态和 diff；确保没有无关文件被更改。
7. 确认重命名 WAL 记录了旧/新名称、路径、更新计数和手动审查计数。

不要编辑仍提及旧路径的历史 WAL。它是历史证据。

## 处理失败

- `directory_exists`：选择更清晰的名称或解决实际的目标冲突；不要自动删除目标。
- `rename_manual_review_required`：检查并在 `--allow-unresolved` 前获得用户同意。
- `external_write_race`：重新读取已更改的文件并在协调编辑后重复 dry-run。
- `lock_target_changed` 或身份歧义：再次通过 UUID/路径解析当前 Task；不要猜测。
- 成功但 `committed: true` 且 `wal_write_failed`：重命名已发生。通过 ID/新路径验证，修复 WAL，追加缺失的记录；不要重复重命名。
- 进程/OS 失败后的意外部分工作现场：停止，盘点旧/新路径和所有计划的引用更新，并阅读[诊断与修复](diagnostics-and-repair.md)。Core 没有通用重命名回滚。
