# 代码风格

Python 项目中的风格清晰地分为两层：工具可以机械地决定和执行的部分，以及需要人类对含义进行判断的部分。将两者混为一谈会将审查注意力浪费在格式化工具已经解决的问题上，更糟糕的是，让真正重要的东西——名称、边界、抽象层次——因为大家都在争论行长度而被忽视。本文档就是关于划定这条界限，只在有意义的地方投入判断力。

## PEP 8 基线

PEP 8 是 Python 的默认风格基础：布局、缩进、导入、命名、注释和一些编程建议。它是共享的基线，让不熟悉的 Python 代码也能阅读。但它不是一份完整的工程规范——它没有说明类型边界、异常语义、工具或架构，而且它的许多具体格式规则早于现代的自动格式化工具。采用 PEP 8 的正确方式是"默认遵循，允许本地一致性和可读性胜出"，而不是"将每条历史格式条款都转录为硬性规则"。

## 自动化边界

风格管理中最有用的一步是将所有机械性工作委托给工具，停止讨论它们。

格式化工具（Ruff 的格式化器或 Black）拥有布局：换行、引号风格、空白、尾随逗号、括号位置。这些没有语义内容，所以唯一重要的是一致性，而格式化工具免费提供这一点。一旦格式化工具在 pre-commit 和 CI 中运行，格式化就不再是审查话题——没有什么可争论的。

Linter（Ruff）拥有机械可检测的内容：未使用的导入、导入排序、明显的死代码、现代目标版本已取代的过时语法，以及大量的易错模式目录。Linter 通过有选择性地启用规则家族来配置，而不是全部启用；每个添加的规则都应该根据误报成本来证明其存在的价值。工具的具体细节——哪些规则家族、如何配置——属于工具参考；这里重要的是原则：如果一个规则可以决定它，人类就不应该在审查中决定它。

留给人类审查的是一切工具*无法*判断的东西：名称是否传达意图、模块是否具有单一职责、抽象是否值得、代码是否表意清晰。这才是风格审查注意力应该放的地方。

## 命名

名称是杠杆率最高的风格决策，因为它们被阅读的次数远多于编写的次数，并且没有工具能够评估它们。除了 PEP 8 的机械约定（函数和变量用 `snake_case`，类用 `PascalCase`，常量用 `UPPER_CASE`），判断在于名称是否在使用点传达了意图。

一个名称应描述事物在领域中*是什么*或*做什么*，而不是它的类型或实现。`users` 优于 `user_list`；`deadline` 优于 `dt2`；`is_expired` 优于 `flag`。作用域应指导长度：循环索引可以是 `i`，模块级函数则不行。避免编码读者已能从注解中看到的类型信息，避免节省几个字符却让每次阅读多花片刻解码的缩写。布尔值最好读作谓词（`is_`、`has_`、`should_`）；有副作用的函数最好读作动词。

## 模块边界

一个模块应该有一个可以用一句话陈述的存在理由。当一个模块积累不相干的职责——请求处理旁边是数据库访问，再旁边是格式化辅助函数——每次修改都触及它，每次导入都拖入整堆东西。工具无法做出的判断是模块中的东西是否属于一起：它们是否因相同的原因而改变，它们是否处于相同的抽象层次，寻找其中一个的读者是否会期望找到其他东西。内聚性是标准，而不是文件大小。一个专注的 400 行模块比一个名为 `utils` 的 50 行大杂烩更健康，后者被所有东西导入却没有任何东西拥有它。

## 抽象层次

好的代码在任何单个函数内以一致的层次阅读：高层编排函数应调用命名良好的步骤，而不是将编排与字节操作交织在一起。混合层次——一个既决定业务策略又操作字符串索引的函数——迫使读者不断切换思维齿轮。纠正方法不是"更多抽象"而是*一致*的抽象：将底层细节提取到一个名称后面，这样高层函数读起来像一系列意图。这是一个判断调用，在另一个方向也有真正的失败模式（过早抽象、没有回报的间接性），这正是它无法自动化的原因。

## 显式性

PEP 20 的"显式优于隐式"是一种风格价值观，而不是一条 lint 规则，它体现在数十个小的决策中：将依赖作为参数传递而不是去取全局变量，返回命名类型而不是调用者必须记住如何解包的裸元组，引发特定异常而不是裸的 `Exception`，将魔数命名为常量。显式性用一点简洁换取大量可读性，在接口边界处几乎总是值得的。反作用是，显式性走向极端就变成了噪音——将好的名称或清晰的类型已经显而易见的东西再拼写出来毫无意义。判断标准是让某件事显式是否能*减少*读者的认知负担；当它能时，就优先选择。
