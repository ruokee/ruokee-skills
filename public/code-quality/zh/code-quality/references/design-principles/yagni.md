# YAGNI — You Aren't Gonna Need It

## 它是什么

YAGNI 的意思是：不要为一个你只是想象中的未来去构建能力。这个原则由 Martin Fowler 强化，针对的是 _预设功能_ 和 _投机性抽象_——现在就加上的功能或灵活性，只是因为你猜它以后会有用。它讲的是时机：等真实需要到来时再做，而不是提前做。

YAGNI 经常被误引为反对测试、错误处理或认真设计。不是这样。Fowler 明确说过，YAGNI 并不反对让代码保持易于改变的测试、重构和技术健康。事实上，YAGNI _依赖_ 这些东西。

## 为什么投机式构建的成本比看上去更高

Fowler 将一个预设功能的成本拆成四部分：

- **构建成本**——把一个还不需要的东西做出来的时间，这些时间本可以用在真正需要的事情上。
- **延期成本**——你真正需要的功能会更晚交付，因为你忙着做投机性的东西。
- **持有成本**——未使用的能力会让整个系统对所有人、每天都更难理解和修改，直到它最终被用上或被移除。
- **修复成本**——当投机性功能猜错方向时（这种情况通常发生），你还得再花一次代价去修或删。

持有成本是人们最容易忘掉的。一个没被使用的扩展点并不是免费的地盘；它是理解上的永久税。每个读者都要猜它是干什么的，每次重构都要保住它，每次测试都要考虑它。

## YAGNI 禁止什么

- 在还没有第二个真实实现之前，就构建 plugin system、base class、registry 或 generic framework。
- 为一个假想中的第二个实现增加 interface 或 abstraction。
- 添加当前没有任何 caller 使用的 field、parameter、config switch 或 extension hook。
- 把函数泛化到处理从未出现过的输入。

## YAGNI 不禁止什么

这部分最容易被误解。YAGNI 说的是 _投机性功能_，不是说不要工程纪律。它并不意味着可以省略：

- **测试**——测试保护行为，让你可以安全地推迟决策。
- **错误处理**——针对确实会发生的失败，尤其是边界处的失败。
- **类型安全**——公共 API 和 module 边界上的 type hint 会降低未来修改成本，而不是增加它。
- **资源清理**——文件、连接、锁和事务的 context manager 属于正确性，不是投机。
- **重构**——保持代码可塑性，正是 YAGNI 能成立的前提。
- **已发布接口的 API 稳定性**——对于公共 API、持久化 schema 或外部 protocol，提前稳定边界并不是投机；因为之后修改它的代价是真实且已知的。

判别标准是：这项工作是在服务一个 _已知且当前存在_ 的需求（正确性、安全性、真实消费者），还是在为一个 _想象中的未来_ 需求做准备？前者是好的工程；只有后者才是 YAGNI 所反对的。

## YAGNI 需要代码可塑

只有当以后改变想法的成本很低时，YAGNI 才安全。如果代码没有测试，而且又难以重构，那么把决策推迟就会变成永久空缺——等真的需要时，你可能已经不敢再加。于是 YAGNI、[tdd.md](./tdd.md)（或事后测试）和重构就组成了一组：测试和重构让代码保持可塑，而可塑的代码让“以后再决定”成为理性的策略，而不是债务借口。

## 在 Python 中

- 先用普通 function、显式参数、`dataclass` 或简单 mapping 满足真实调用点。
- 在抽象之前，先等第二个真实变化点出现；在提取之前，想象一下未来重构会有多便宜——通常便宜到足够等一等。
- 对于公共 API、持久化 schema 和外部 protocol，提前稳定边界是可以的；那不是投机功能。
- 删除一个未使用的 extension point，比保留一个“也许以后有用”的 abstraction 更符合 YAGNI。

## 与其他原则的关系

- [kiss.md](./kiss.md)：YAGNI 去掉投机性特性；KISS 去掉你保留下来的特性中的不必要复杂度。
- [rule-of-three.md](./rule-of-three.md)：这是“等到它真的需要了再做”的去重版本。
- SOLID 的 Open/Closed Principle（见 [solid.md](./solid.md)）会推动你做扩展点；YAGNI 则说只有在真实 variation 出现时才去做。
