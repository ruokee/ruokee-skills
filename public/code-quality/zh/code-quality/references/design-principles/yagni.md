# YAGNI — 你并不会需要它（You Aren't Gonna Need It）

## 它是什么

YAGNI 说：不要为仅凭想象构建未来所需的能力。这个原则经 Martin Fowler 提炼，针对的是*推定功能（presumptive features）*和*投机性抽象（speculative abstractions）*——现在添加的功能或灵活性，仅仅因为你猜测以后会有用。这是关于时机的论断：在真正需要时再构建，而不是提前。

YAGNI 经常被曲解为反对测试、错误处理或精心设计的论据。事实并非如此。Fowler 明确表示 YAGNI 并不反对重构、测试和保持代码易于修改的那些技术健康实践。事实上，YAGNI *依赖于*这些东西。

## 为什么投机性构建的成本比看上去更高

Fowler 将推定功能的成本分为四部分：

- **构建成本（Cost of build）**——花在构建尚不需要的东西上的时间，从真正需要的东西那里夺走的。
- **延迟成本（Cost of delay）**——你实际需要的功能因为忙于投机性功能而延迟交付。
- **持有成本（Cost of carry）**——未使用的能力使系统更难理解和修改，*对每个人、每一天*，直到它最终被使用或移除。
- **修复成本（Cost of repair）**——当投机性功能猜错了（通常情况），你需要再次付出代价来修复或移除它。

持有成本是人们忘记的那部分。一个未使用的扩展点不是免费的不动产；它是对理解力的永久征税。每个读者都要琢磨它是干什么用的，每次重构都要保留它，每个测试都要考虑它。

## YAGNI 禁止什么

- 在还没有超过一个真实实现之前，构建插件系统、基类、注册表或通用框架。
- 为假设的第二个实现添加接口或抽象。
- 添加没有当前调用者使用的字段、参数、配置开关或扩展钩子。
- 泛化函数以处理从未出现的输入。

## YAGNI 不禁止什么

这是最常被误解的部分。YAGNI 是关于*投机性功能*，而不是关于工程纪律。它不豁免以下内容：

- **测试（Tests）**——它们保护行为，让你能安全地推迟决策。
- **错误处理（Error handling）**——针对实际发生的失败，尤其是边界处。
- **类型安全（Type safety）**——公共 API 和模块边界上的类型提示能降低（而非增加）未来变更的成本。
- **资源清理（Resource cleanup）**——用于文件、连接、锁和事务的上下文管理器是正确性，而非投机。
- **重构（Refactoring）**——保持代码的可塑性正是 YAGNI 可行的原因。
- **已发布接口的 API 稳定性（API stability for published interfaces）**——对于公共 API、持久化 schema 或外部协议，提前设计稳定边界不是投机；以后修改它的成本是真实且已知的。

区分标准：这项工作服务于*已知的、当前的*需求（正确性、安全性、真实的消费者），还是为了对冲*想象的、未来的*需求？前者是好的工程；只有后者才是 YAGNI 所禁止的。

## YAGNI 需要可塑的代码

YAGNI 只有在以后改变主意成本低廉时才是安全的。如果代码没有测试且抗拒重构，那么推迟决策就会变成永久性的缺口——当真正需要添加能力时，你会因害怕而不敢动手。所以 YAGNI、[tdd.md](./tdd.md)（或事后测试）和重构构成一个组合：测试和重构保持代码可塑，而可塑的代码让"以后决定"成为理性策略而非借口的托词。

## 在 Python 中

- 先用普通函数、显式参数、`dataclass` 或简单映射来满足真实的调用点。
- 在第二个真实变化点出现之前等待抽象；在提取之前，设想一下未来的重构会有多便宜——通常便宜到值得等待。
- 对于公共 API、持久化 schema 和外部协议，提前稳定边界是可以的；那不是投机性功能。
- 移除未使用的扩展点比保留"可能有用"的抽象更符合 YAGNI。

## 与其他原则的交互

- [kiss.md](./kiss.md)：YAGNI 移除投机性功能；KISS 移除你保留的功能中的不必要复杂度。
- [rule-of-three.md](./rule-of-three.md)：专门针对去重的"等到真正需要时"形式。
- SOLID 的开闭原则（参见 [solid.md](./solid.md)）倾向于扩展点；YAGNI 说只在真正出现变化时才构建它们。
