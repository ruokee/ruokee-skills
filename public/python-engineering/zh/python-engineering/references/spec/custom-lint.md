# Custom Lint Rules

自定义 lint rule 是一种项目专属检查：当代码违反了某条通用工具不知道的约定时，它会让构建失败。它是把一条规则从“记住它”变成“强制执行”的强大手段 - 但它同时也是持续维护成本，并且可能产生误报，削弱整个 lint 套件的可信度。大多数约定不应该变成自定义规则。真正需要做的是识别那一小部分值得做成规则的约定，把它们设计到几乎不会误报，并把它们放到运行成本低的地方。

## 何时值得做自定义规则

通用 linter（Ruff）已经覆盖了大量社区通用模式 - 未使用的 import、过时语法、常见 bug 形态。自定义规则是给那些没有现成规则表达、但又值得机械执行的项目专属约定准备的。满足以下条件时，才值得做：

- **规则是项目专属的。** 禁止某个 import path、依赖方向边界（“domain layer 不能 import web layer”）、framework 约定（“每个 handler module 都暴露一个 `router`”）、组织内部的命名或放置规则。这些都是通用工具无法知道的本地知识。
- **违反频率高或代价足够大，值得自动化。** 如果约定很少被破坏，而且 review 时很容易发现，那么做成规则就是过度设计。应当自动化那些反复出现的，或者违反后会造成真实损失的。
- **检查是机械性的。** 它可以仅从代码结构判断，不需要理解意图。“这个 module 有没有 import `app.web`？”是机械的；“这个抽象是否合适？”不是，也永远不可能成为 lint rule。
- **误报率可以压到接近零。** 一个会对合法代码发声的规则，会训练大家忽略或 suppress 它，从而污染整个 lint gate。如果不能把规则做得足够精确，它就还没准备好成为规则。

当某条约定不满足这些条件 - 需要判断、很少出现、无法做得足够精确 - 它就应该留在 review checklist 或文档里，而不是进 linter。

## 规则如何设计

好的自定义规则必须是 **机械的、静态的、低误报的**，并且像其他代码一样被文档化和测试。

- **静态且结构化。** 规则应从源代码或 AST 判断，而不是运行代码。静态检查速度快、安全且确定；需要 runtime 行为的规则，本身就选错了工具。
- **精确优于宽泛。** 针对具体违规点精细定位。标记“看起来危险的 import”的规则会误伤；标记“从 `app/` 外部 import `app.internal`”的规则则是精确的。优先选择一个能抓住真实问题的窄规则，而不是一个会顺带打中无辜代码的宽规则。
- **清晰、带前缀的错误码。** 给规则一个带项目前缀的稳定代码（例如 `PRJ001`），这样它才能被引用、按需带理由地禁用，并在输出中一眼识别。附带 message 应当说明哪里错了，以及该怎么做。
- **有文档且经过测试。** 自定义规则就是代码：它需要测试来证明自己既能在违规时触发，也能在合法代码上保持沉默，还需要一行文档说明其用途。没有文档、只有作者自己懂的规则，会变成所有人的隐形陷阱。

## Ruff 与 Flake8 plugin

实现项目专属规则有两种实际机制。

**Flake8 plugin** 是发布自定义检查的传统方式：它是一个小型 Python package，注册一个 AST visitor，并以你的前缀输出错误码。它成熟、文档完善，而且正是为这种用途设计的 - 定制的、项目本地的规则，逻辑完全可控。代价是为了托管这些自定义检查，需要额外运行 Flake8。

**Ruff** 负责通用 lint，而且速度很快，但在 Ruff 里编写真正自定义的项目专属规则，比编写一个 Flake8 plugin 更受限制。多数项目的务实做法是：让 Ruff 负责所有标准规则族，再用一个小型 Flake8 plugin 承载 Ruff 无法表达的少量项目专属规则。不要再用 Flake8 重复检查 Ruff 已经覆盖的规则 - 那是重复劳动，也是冲突配置。如果未来某条自定义规则的逻辑变得可以用 Ruff 表达（或者 Ruff 原生支持了），迁回 Ruff 是合理的；选择应当服务于规则的可维护性，而不是工具纯洁性。各工具的具体信息见工具参考（`../tooling/ruff.md`、`../tooling/flake8-plugin.md`）。

## 规则运行在哪里

自定义规则应当和其他 lint 一样，运行在两个 gate 上：pre-commit 和 CI。

- **pre-commit** 会在每次提交前运行规则，在最早且成本最低的时点捕获违规。这正是快速、确定性的自定义检查最有价值的地方 - 作者会立刻看到失败，而不是等到 review。
- **CI** 会再次运行同一规则，作为权威 gate，因为 pre-commit 可能被跳过、未安装，或者在不同机器上表现不一致。真正重要的规则，无论本地环境如何，都必须在 CI 中运行。

要把规则做得足够快以适合 pre-commit；如果某个自定义检查很慢或需要网络访问，就应只放在 CI 中。两道 gate 的关系见 [pre-commit reference](../tooling/pre-commit.md)：pre-commit 是早、快、可跳过的一层，CI 是最终、完整、权威的一层。
