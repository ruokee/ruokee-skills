# 三的原则（Rule of Three）

## 它是什么

三的原则是关于*何时*抽象的启发法：第一次直接编写，第二次容忍有限的重复，只有到第三次出现——或者当变化方向清晰理解后——才提取共享抽象。它是 [dry.md](./dry.md) 的实际刹车。

数字三并非魔法。真正的信号是"我现在理解这些案例是如何变化的"。两个实例很少能告诉你这一点。三个通常可以，因为第三个点揭示了哪些部分是稳定的、哪些部分是不同的，从而知道分割点在哪里。

## 为什么要等待

只有两个示例时，你看不到变化的轴。你提取的任何抽象都是对将要变化的内容的猜测，而对未来的猜测通常是错误的。当你猜错时，共享函数通过增加标志、模式参数、回调和特殊情况分支来生存——每个新需求都会在从未为其塑造的结构上增加一个旋钮。代码最终比它所取代的重复更糟糕，你最终不得不展开它：内联抽象、恢复重复，然后从三个具体案例中重新推导出真正的分割点。

三个具体实例让你提取的参数集反映的是实际变化而非想象的变化。抽象的形态是被发现的，而非被发明的。

## 与 DRY 的关系——互补而非矛盾

表面上看，DRY 和三的原则似乎是矛盾的：DRY 说要消除重复，三的原则说要等待。它们通过记住各自针对的目标来调和。

- DRY 针对的是*知识*重复——同一规则在多处表达。
- 三的原则推迟的是*形态*相似的抽象——看起来相似但可能不共享知识的代码。

所以它们实际上并不冲突。DRY 仍然要求立即折叠真正重复的领域规则。三的原则只管辖更难的案例：那些彼此相似但共同本质尚未得到证明的片段。在这种情况下，等待第三个实例可以保护你免受错误抽象的影响。

## 何时覆盖它

当知识既明显相同又是高风险时，不要机械地等待三次。在以下情况下覆盖三的原则：

- 重复的内容是安全检查、权限规则、金钱计算、协议合约、数据一致性不变量，或 schema 的唯一真实来源。此处分歧是危险的，而且你已经知道这是一条规则。
- 模式是明显且稳定的——一个众所周知的惯用模式，其变化轴从第一个或第二个实例就清晰可见，且案例之间没有实际分歧的可能。

简而言之：当共性不确定时等待；当共性确定且漂移成本高时行动。

## 在 Python 中

- 第一次出现：直接编写内联代码。
- 第二次出现：允许少量复制粘贴；抵制提取的冲动。
- 第三次出现，或清晰的变化方向：提取——通常是带有明确参数的模块级函数、表驱动映射或小型 `Protocol`，不一定是类。
- 当你后来意识到抽象得太早时，优先选择内联/扁平化/再次复制，然后寻找真正的变化轴。这是正常的重构操作，而非失败。

## 与其他原则的交互

- [dry.md](./dry.md)：三的原则是安全阀，防止 DRY 产生过早的、过度参数化的抽象。
- [yagni.md](./yagni.md)：两者都抵制在需要之前构建结构；三的原则是专用于去重的版本。
- [kiss.md](./kiss.md)：等待通常能让代码保持更简单，因为两个清晰的副本胜过一团纠结的通用函数。
