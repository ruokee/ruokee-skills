# 自定义 Lint 规则

自定义 lint 规则是一种项目特定的检查，当代码违反了通用工具不知道的约定时，它会导致构建失败。它是使规则被*强制执行*而不是被*记住*的强大方式——但它也是一项持续的维护成本，并可能成为误报的来源，从而侵蚀对整个 lint 套件的信任。大多数约定不应成为自定义规则。这里的技能是识别应该成为自定义规则的狭窄集合，设计它们使其几乎从不误报，并将它们放置在能廉价运行的地方。

## 何时需要自定义规则

通用 linter（Ruff）已经覆盖了大量社区范围的模式——未使用的导入、过时的语法、常见的错误形态。自定义规则是为没有现成规则表达且重要到需要机械强制执行的项目特定约定而准备的。证明其合理性的条件：

- **规则是项目特定的。** 禁止的导入路径、依赖方向边界（"领域层不得导入 Web 层"）、框架约定（"每个处理模块都暴露一个 `router`"）、组织命名或放置规则。这些编码了通用工具无法拥有的本地知识。
- **违规足够频繁或代价足够高，需要自动化。** 如果约定很少被违反且容易在审查中发现，则规则属于过度工程。自动化那些重复出现或其违规造成实际损害的情况。
- **检查是机械性的。** 它只需从代码的结构就能决定，无需理解意图。"这个模块是否从 `app.web` 导入"是机械性的；"这个抽象是否合适"则不是，也永远不能成为 lint 规则。
- **误报率可以保持接近零。** 在合法代码上触发的规则会让每个人都学会忽略或压制它，这会毒害整个 lint 门控。如果你无法使规则精确，它就不适合成为规则。

当一个约定未能通过这些测试时——它需要判断、它很少见、它无法做到精确——它属于审查清单或文档，而不是 linter。

## 如何设计规则

一个好的自定义规则是**机械性的、静态的、低误报的**，并且像任何其他代码一样被记录和测试。

- **静态的和结构性的。** 从源码或其 AST 决定规则，而不是通过运行代码。静态检查快速、安全且确定性强；需要运行时行为的规则是错误工具。
- **精确优于宽泛。** 紧密地针对特定的违规。一个标记"看起来有风险的导入"的规则会误报；一个标记"任何从 `app/` 外部对 `app.internal` 的导入"的规则是精确的。优先选择捕获真实情况的窄规则，而不是同时误伤无辜者的宽规则。
- **清晰的、带前缀的错误代码。** 给规则一个带有项目前缀的稳定代码（例如 `PRJ001`），以便它可以被引用、在附有理由的情况下有选择地禁用，并在输出中被识别。随附的消息应说明什么错了以及应该怎么做。
- **记录和测试。** 自定义规则也是代码：它需要一个证明它在违规时触发并在合法代码上保持沉默的测试，以及一行说明其目的文档。一个只有其作者理解的、没有文档的规则会成为其他人的隐形陷阱。

## Ruff 与 Flake8 插件

存在两种实用机制用于项目特定的规则。

**Flake8 插件**是交付自定义检查的既定方式：它是一个注册了 AST 访问器并在你的前缀下发出错误码的小型 Python 包。它成熟、文档完善，并且正是为此而构建的——定制的、项目本地的规则，完全控制逻辑。代价是仅为了托管这些自定义检查而让 Flake8 与 Ruff 一起运行。

**Ruff** 拥有通用 linting 功能且速度很快，但在其中编写真正的自定义项目特定规则比编写 Flake8 插件更受限。大多数项目最终采用的实际分离方案是：让 Ruff 处理所有标准规则家族，并使用一个小的 Flake8 插件来承载 Ruff 无法表达的那几个项目特定规则。不要运行 Flake8 来重新检查 Ruff 已覆盖的规则——那是重复工作和冲突的配置。如果自定义规则的逻辑后来可以在 Ruff 中表达（或者 Ruff 增加了原生支持），将其迁移回来是合理的；选择服务于规则的可维护性，而不是工具的纯洁性。每个工具的具体细节在工具参考中（`references/tooling/ruff.md`、`references/tooling/flake8-plugin.md`）。

## 规则运行的位置

自定义规则属于与其余 linting 相同的两个门控：pre-commit 和 CI。

- **pre-commit** 在每个提交之前运行规则，在最早的廉价时刻捕获违规。这是快速、确定性的自定义检查发挥作用的地方——作者在审查之前立即看到失败。
- **CI** 作为权威门控再次运行相同的规则，因为 pre-commit 可以被跳过、卸载或因机器而异。重要的规则必须不管本地设置如何在 CI 中运行。

保持规则足够快速以用于 pre-commit；缓慢或需要网络访问的自定义检查只应放在 CI 中。两个门控之间的关系在 [pre-commit 参考](references/tooling/pre-commit.md) 中有介绍：pre-commit 是早期、快速、可跳过的层，而 CI 是最终、完整、权威的层。
