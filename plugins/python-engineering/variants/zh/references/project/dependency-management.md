# 依赖管理

每个依赖既是便利，也是一项持续的负债：它必须被解析、锁定、安装、审计，并最终升级，而每个依赖都会扩大项目可能出问题的面。依赖管理是一门纪律，它声明每个依赖*位于何处*、*为什么*存在、*约束的紧密度*，以及是否锁定了完整的解析结果。目标是在必须的地方可重现的安装，针对每个用例最小化依赖，并诚实地反映运行时真正需要什么而开发者只需要什么。

## 运行时、开发、可选和工作空间内部依赖

对于任何依赖，第一个决定是它属于哪个分类，因为不同的分类面向不同的受众。

运行时依赖放在 `[project.dependencies]` 中。它们是包无法导入或运行所必需的库——它们为每个最终用户和每次部署安装。进入门槛是真正的必要性：包确实离不开它，而且重新实现它不合理。

开发依赖——linter、格式化工具、类型检查器、pytest、覆盖率、文档构建工具——只在处理项目时需要，在运行时从不使用。它们属于 `[dependency-groups]`（PEP 735）下的依赖组，通常分为 `dev`、`test` 和 `docs` 等连贯的组。将它们排除在运行时的集合之外，可以防止已部署的服务拖入一个它永远不会调用的测试框架。

可选依赖放在 `[project.optional-dependencies]` 中，支持并非每个用户都需要的特性——数据库驱动、YAML 解析器、文档附加功能。每个 extra（附加组）应对应一个连贯的特性边界，用户可以通过名称选择加入（`pip install mypkg[redis]`），而不应成为"某些人可能想要的东西"的垃圾场。

工作空间内部依赖是[工作空间](structure.md)成员之间的边：一个成员依赖于同一仓库中的另一个成员。这些依赖遵循声明的依赖*方向*，必须指向成员的稳定公共 API，绝不能深入其内部。

## 锁文件策略

锁文件记录了精确的解析图——每个传递依赖的精确版本——使得安装可以逐字节重现。是否提交锁文件取决于项目*是什么*，而不是通用规则。

应用和服务提交其锁文件（`uv.lock` 或等效文件），因为重点是部署与测试时相同的已解析环境。CI *从*锁文件安装，因此生产环境与开发环境一致。库通常*不*提交锁文件：下游消费者根据自身的约束解析自己的依赖图，而仅针对一个冻结的解析结果进行测试的库会遗漏其实际声称支持的版本范围内的破坏。工作空间使用一个覆盖所有成员的单一根锁文件，每个成员自身的约束在其自己的 `pyproject.toml` 中表达。

当锁文件变更出现在 diff 中时，值得仔细阅读而不是随意批准：意外的主版本跳升、新的传递依赖或移除的包可能伴随不相关的更改悄悄混入。升级应来自明确的命令和审查，而不是无声地与其他编辑一起漂入。

## 版本约束

每个约束的张力在于灵活性和可重现性之间。兼容发布的约束（`>=2.1,<3` 或 `~=2.1`）让解析器选择补丁和次版本更新，这使项目保持最新，并避免与其他包共存时产生冲突——这是库的正确默认值，因为库的约束被所有下游继承。精确固定（`==2.1.4`）保证一个特定版本，这是你希望应用可重现部署的方式，但在库中这是不友好的：它强制所有消费者都固定该版本，并且经常成为无法满足的解析的根源。

一般规则是尽可能宽地约束。在可重现性是目标的地方精确固定（应用，尽可能通过锁文件而不是说明符）；在包将与其他包组合的地方使用兼容范围（库）。比必要更紧的锁定应附上理由，因为未来的读者否则将不得不猜测它是在防范真实的不兼容还是只是过时的谨慎。

## 重量和合理性

每个新依赖都应通过一个合理性门槛：能否通过标准库、一个更小的库或几行本地代码来满足需求？标准库是默认依赖；只有在标准库机制确实不足时才越过它。为了一个简单的工具函数拖入一个庞大的框架，是用巨大的安装体积、更大的攻击面以及持续的升级负担，去换取一个简短的辅助函数就能覆盖的东西。

成本不仅仅是下载大小。每个依赖都是一个供应链入口、一个需要检查的许可证，以及一个维护承诺——一个无人维护或维护薄弱的包在停止接收修复时就会变成项目的问题。某些重量是不可避免且正确的：数值和数据处理工作正当地依赖于大型、维护良好的包，这并非坏味道。判断标准是相称性，而不是为极简而极简。

## 单体仓库中的依赖方向

在工作空间或单体仓库中，内部包之间的依赖构成一个图，而这个图必须保持有向无环图。共享和底层包（领域模型、工具类）被依赖；应用和服务依赖*于*它们，而不是反过来。当两个成员开始相互导入时，它们之间的边界已经失效，它们实际上是一个包穿着两个名字。

每个成员应只声明自己的直接依赖。依赖传递依赖——因为另一个成员碰巧引入了它——的包有一个未声明的依赖，一旦中间包移除它就会出问题。每个成员显式声明的纪律，加上一个单一的共享锁文件，是让工作空间保持连贯而不是放大耦合的关键；这方面的结构侧在 [structure](structure.md) 中介绍。
