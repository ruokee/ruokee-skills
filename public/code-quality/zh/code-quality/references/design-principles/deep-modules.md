# 深模块与信息隐藏（Deep Modules and Information Hiding）

本文档借鉴了 John Ousterhout 的《软件设计哲学》（*A Philosophy of Software Design*）。其核心思想
是一种通过比较接口成本与隐藏内容的价值来判断抽象质量的方法。

## 深度：接口成本 vs 实现价值

每个模块都有两个面：**接口（interface）**（调用者必须理解才能使用的内容）和**实现（implementation）**
（接口后面的所有内容）。Ousterhout 将模块质量定义为两者之间的比率：

- **深模块（Deep module）** 具有隐藏了大量有用复杂度的简单接口。调用者学习很少，获得很多。
  垃圾回收器、设计良好的文件系统 API 或 `requests.get(url)` 是深的——调用简单，背后内容丰富。
- **浅模块（Shallow module）** 的接口几乎和实现一样复杂。调用者为了使用它几乎支付了理解内部
  的全部成本，因此该模块几乎没有提供净收益。一个签名复述其主体的单行包装器是极端情况。

模块的价值不在于它的大小。深模块内部可以很大；重要的是接口相对于它所吸收的内容来说是小的。
这重新定义了抽象的目标：不是"隐藏代码"，而是"让调用者忽略他们不需要推理的复杂度"。

## 信息隐藏（Information Hiding）

深度背后的机制是信息隐藏：每个模块封装一个设计决策——尤其是可能变化或难以正确实现的决策——
这样其他模块就不会依赖于它。当被隐藏的决策发生变化时，变更保持在局部。

其反面是**信息泄露（Information leakage）**：当一个设计决策出现在多个模块中时，一个变更迫使
所有模块都进行编辑。一个对读取方和写入方都已知的存储格式、每个调用者都可见的线协议细节、
跨层复制的错误映射约定——这些都是泄露。泄露是诸如[霰弹式修改](references/refactoring/index.md)
等坏味道背后的深层原因，而信息隐藏是 [DRY](./dry.md) 如何应用于设计决策（而不仅仅是代码）的体现：
知识存在于一个模块中。

## 什么构成了好的抽象

- 它隐藏了真正值得隐藏的东西——一个困难的算法、一个不稳定的依赖、一个协议细节、一组错误和版本差异。
- 其接口用调用者已经思考的术语来表达，而不是用实现术语。
- 它不强迫调用者知道正确的调用*顺序*或内部状态才能正确使用它。需要这样的知识本身就是一种泄露的复杂度。
- 常见情况调用简单；罕见情况是可能的但不使常见路径复杂化。

一个常见的反面模式是**透传方法（pass-through method）**——一个除了用相同签名调用另一个方法外
什么都不做的方法。它增加了接口面和间接层，却什么也没隐藏，使模块变得更浅。同样适用于重命名
单个表达式的薄包装函数：参见 [DRY](./dry.md) 关于浅辅助函数的讨论。

## 与 KISS 和接口设计的关系

深模块使 [KISS](./kiss.md) 更加具体。"保持简单"并不意味着每个函数都必须短；它意味着最小化
读者必须在脑中容纳的复杂度。少数几个具有干净接口的深模块留给读者的总复杂度比许多迫使读者不断
在文件间跳转的浅辅助函数更少。为了追求短函数和小文件本身而这样做，会产生浅模块和*更多*需要学习的
接口——与简单的本意相反。

深度也缓和了[接口隔离](./solid.md)的小接口本能：目标是接口*相对于它们交付的内容*而言是小的，
而不是将接口切得如此之薄以至于调用者必须组装许多接口才能做任何事。

## 原则被误用时的情况

- 将信息隐藏视为"让每个字段私有并添加 getter/setter"。机械地暴露每个字段的访问器什么也没隐藏，
  只是增加了表面。用于简单数据载体的透明 dataclass 是好的；当存在不变量或可能的变更时才封装。
- 隐藏调用者合理需要的东西——必要的配置、有意义的错误或真实的性能成本。隐藏成本并不会移除它；
  它会在以后给调用者带来惊喜。
- 将一个内聚的深模块拆分成浅碎片，以满足文件大小或函数长度的偏好。

## 在 Python 中

- Python 没有强制的 `private`。使用命名约定（`_internal`）、模块结构、`__all__`、故意设计的
  公共 API、`property` 和 `Protocol` 而非访问修饰符来表达边界。
- 优先让模块的内部稍微长一些但局部清晰，而不是将逻辑分散到读者需要追踪的十几个浅辅助函数中。
- 适配器层是天然的深模块：它通过一个小的接口隐藏第三方库的细节，同时仍然暴露调用者必须考虑的
  错误和性能特征。这是[防腐层](./ddd.md)和[适配器](references/design-patterns/adapter.md)模式
  的结构基础。
