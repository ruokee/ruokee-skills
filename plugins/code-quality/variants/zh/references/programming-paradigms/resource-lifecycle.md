# 资源生命周期设计（Resource Lifecycle Design）

资源（Resource）是任何必须获取并在之后释放的东西：文件句柄、套接字、锁、数据库连接或事务、临时目录、线程或任务、子进程。资源生命周期设计是关于为每个资源回答以下问题：谁创建它，谁关闭它，以及它在每条路径（包括异常路径）上如何被释放。大多数资源泄漏和"连接池耗尽"事件都是从未被显式回答的生命周期问题。

## 所有权

每个资源需要正好一个所有者：负责释放它的代码。模糊的所有权是泄漏（每个人都以为别人会关闭它）和关闭后使用（一个持有者关闭了它，而另一个仍然需要它）错误的根源。

最清晰的规则是：创建资源的代码拥有它并关闭它，并且在其控制的范围内执行此操作。当一个函数仅在其自身持续时间内需要资源时，它应该在本地创建、使用和释放它。当一个资源必须比单个函数存活更久时，所有权上移至一个更长生命周期的持有者——一个应用程序对象、一个上下文、一个池——并且该持有者的生命周期成为资源的生命周期。将打开的资源传递给*不*拥有它的函数是可以的，只要约定清晰：被调用者使用它，调用者关闭它。

函数签名可以使所有权显式化。接受已打开资源的函数借用它；打开自己资源的函数拥有它。混合两者——有时打开，有时接受——正是所有权变得模糊且资源泄漏的地方。

```python
def write_report(out: TextIO, rows: list[Row]) -> None:
    # 借用 out：调用者打开它，调用者关闭它
    for row in rows:
        out.write(format_row(row))


def save_report(path: str, rows: list[Row]) -> None:
    # 拥有文件：在同一个作用域中打开、使用、释放
    with open(path, "w") as out:
        write_report(out, rows)
```

借用函数从不调用 `out.close()`——那将释放它不拥有的资源并使调用者感到惊讶。在接口中保持借用与拥有的一致性，使得泄漏易于推理。

## RAII 和上下文管理器

RAII（Resource Acquisition Is Initialization，资源获取即初始化）将释放绑定到作用域的结束，而不是你可能忘记的手动清理调用。Python 通过上下文管理器协议和 `with`/`async with` 来表达这一点：资源在进入时获取，在退出时释放，无论代码块是正常返回还是抛出异常。

```python
with open(path) as f:
    data = f.read()
# f 此处已关闭，即使 read() 抛出了异常
```

不要依赖 `__del__` 或 CPython 引用计数来释放文件、锁、事务或连接。析构函数的时机在不同实现中各不相同，在引用循环下会失效，并且在异常路径上不可靠。使释放显式且绑定到作用域。

## 作用域绑定 vs 动态资源集合

对于固定的、静态已知的资源集合，嵌套的 `with` 语句（或带有多个管理器的单个 `with`）是最清晰的表达方式。当资源的*数量*是动态的——为每个输入路径打开一个文件，获取一组可变的连接——使用 `contextlib.ExitStack`（或 `AsyncExitStack`）而不是手写嵌套的清理栈：

```python
from contextlib import ExitStack


def read_all(paths: list[str]) -> list[str]:
    with ExitStack() as stack:
        files = [stack.enter_context(open(p)) for p in paths]
        return [f.read() for f in files]
```

`ExitStack` 保证每个进入的资源以相反顺序被释放，即使其中一个获取中途失败。

## 异常路径

当出现问题时清理必须运行，而这正是最容易遗漏的情况。优先使用 `with` 而不是手动 `try/finally`，因为管理器将清理封装一次且正确。当你确实要写 `try/finally` 时，释放操作放在 `finally` 中，而不是在 `try` 块之后。自定义上下文管理器的 `__exit__` 不得吞没异常，除非抑制是其明确且文档化的目的——从 `__exit__` 返回真值会静默地隐藏错误。

## 异步资源所有权

异步资源（连接、会话、异步生成器）遵循相同的所有权规则，但使用 `async with` 和 `__aenter__`/`__aexit__`。两个额外的危险：持有资源的异步生成器可能被挂起且永不恢复，因此使用 `contextlib.aclosing()` 来保证其清理运行；由任务拥有的资源仅在该任务被正确等待或取消时才会被释放，因此后台任务需要显式的生命周期管理（参见 [async-concurrency.md](./async-concurrency.md)）。

## 应用程序启动和关闭

生命周期最长的资源——连接池、客户端、线程池、缓存——由应用程序本身拥有。在启动期间获取它们，在关闭期间以相反顺序释放它们。框架生命周期钩子（ASGI 生命周期、应用工厂、依赖注入作用域）是正确的地方。避免在导入时获取这些资源：导入时的副作用使模块无法安全地导入用于测试或工具，并将资源生命周期与导入顺序而不是应用程序生命周期绑定（参见 [imperative.md](./imperative.md) 关于入口点结构）。

## 部分初始化和释放顺序

一个微妙的失败模式出现在几个资源按顺序获取而稍后的获取之一失败时。已经打开的资源仍然必须被释放，按相反顺序，即使设置从未完成。手写代码往往在这方面出错——"进行了一半"的清理路径是最不可能被测试的路径。

这是优先使用 `with` 和 `ExitStack` 而不是手动设置的另一个原因：它们释放恰好成功进入的资源，按相反顺序，无论设置在何处失败。按依赖顺序获取（先连接，然后是依赖连接的事务），自动的反序释放会正确拆除：先事务，然后连接。当你必须显式地安排释放顺序时，规则是后获取的先释放，因为后面的资源可能在其自身的清理过程中依赖前面的资源仍然存活。

## 池化和租约模式

当获取成本高昂时（数据库连接、HTTP 会话），池拥有一组长生命周期的资源，并在一个工作单元期间将它们*租用*给调用者。租约（而非资源）是调用者获取和释放的东西——通常通过一个上下文管理器实现，该管理器在进入时检出资源，在退出时将其返回到池。规范是相同的：租用的资源有一个清晰的作用域，它在每条路径（包括异常）上返回池。泄漏的租约比泄漏的文件更糟糕，因为它永久性地缩小池直至耗尽。

```python
def handle_request() -> Result:
    with pool.connection() as conn:   # 进入时租用，退出时归还
        return run_query(conn)
```

相同的 RAII 规范从单个文件句柄扩展到应用程序范围的池：命名作用域，使释放自动化，永远不要依赖垃圾回收器为你做这件事。
