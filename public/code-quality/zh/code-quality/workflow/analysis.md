# Analysis Mode — Code Quality

建议性模式。用户是在问“该怎么做”，不是在问“哪里错了”。没有 diff 可判。帮助梳理设计或重构决策，并收敛到一个方案。

## 触发条件

- 设计讨论：“我该怎样组织这个？”、“这个行为应该放哪里？”
- 模式选择：“Strategy 还是直接写个函数更合适？”、“这里值得用 Factory 吗？”
- 重构规划：如何拆分模块、如何安排安全重构顺序、是否该重构。
- 范式选择：imperative vs functional core、OO vs data-oriented、什么时候 state machine 才值得引入。
- 原则应用：DRY、KISS、YAGNI 或 SOLID 在具体情境下如何适用。

## 步骤

1. 理解设计问题及其上下文——预期会发生什么变化、当前哪里痛、有哪些约束。如果缺少关键事实，就提问。
2. 读取与主题匹配的相关参考文档（`references/design-principles/`、`design-patterns/`、`refactoring/`、`programming-paradigms/`），让讨论使用项目的真实语言。
3. 展示选项与取舍。大多数设计问题都有多个可辩护答案；展示现实可行的方案，以及它们在变更成本、可读性和复杂度上的代价。
4. 用真正适用的原则、模式和范式来组织讨论，但只有在 variation point 真实存在时才引入某个模式。
5. 收敛。针对这个上下文推荐一个方案并说明原因，或者提出一个聚焦的问题。不要只罗列。

## 输出

采用对话式、以推荐为导向的表达。不使用僵硬的 findings 格式。说明可选方案，指出决定性的取舍，然后给出推荐。保持与问题规模相称。

## 停止规则

- 只提供建议，不修改文件。
- 不要在不适用时强行套用原则或模式——没有模式往往才是正确答案。
- 不要仅因为相似就建议抽象。
- 由用户决定。如果讨论转为实际工作，建议切换到 fast 或 full review，而不是悄悄改模式。
