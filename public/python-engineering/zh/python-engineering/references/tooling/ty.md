# ty

ty 是 Astral（Ruff 和 uv 背后的团队）开发的基于 Rust 的快速类型检查器和语言服务器。其设计目标是速度和紧密的编辑器集成：一个足够快的类型检查器，可以通过 LSP 在每次按键时运行，再加上一个适合 CI 门禁的 CLI。

## 速度优势（Speed Advantage）

决定性特征是增量速度。Rust 核心使 ty 能够在 mypy 或 pyright 所需时间的一小部分内重新检查大型项目，这保持了编辑器反馈循环的紧凑性，并使 CI 类型门禁变得廉价。快速反馈是早期采用它的全部理由：运行它的成本足够低，不会拖慢日常工作。

## 检查内容（What It Checks）

ty 执行静态类型检查器预期的工作：项目范围内的完整类型覆盖、接口边界、`Any` 泄漏、类型收窄、可达性分析以及不可达代码检测。它既服务于编辑器（通过 LSP），也服务于命令行，因此开发者在行内看到的相同检查也会在 CI 中运行。

```bash
ty check
```

## 成熟度与局限性（Maturity and Limitations）

ty 比 mypy 和 pyright 更新，因此处于早期采用路径上。它尚未证明能覆盖旧检查器处理的所有类型场景，并且对每个第三方桩文件（stub）、边缘情况推断以及最新的 Python 语义的覆盖需要项目级别的验证，而非假设。将其视为高潜力且有能力，但在具体项目上验证其行为，而不是盲目信任。

## 与 mypy 和 pyright 的关系

ty 与 [mypy](mypy.md) 和 [basedpyright](basedpyright.md) 竞争作为主要类型检查器。权衡是具体的：ty 提供今天的速度和编辑器响应性，而 mypy 和 pyright/basedpyright 提供更成熟的生态系统，并且在 basedpyright 的情况下提供更严格的默认配置。选择 ty 作为默认门禁是基于这样的判断：快速反馈在当前项目中超过了成熟度差距；在发布库、外部协作或迁移时，旧的检查器仍然可以作为比较层使用。

## 配置与采用（Configuration and Adoption）

在 `pyproject.toml` 的 `[tool.ty]` 表下配置 ty，并明确定义规则严格性，而不是一次启用所有实验性诊断。早期采用的风险控制方法是固定工具版本、保留项目级配置、在 CI 中验证，并在特定类型问题需要第二意见时保留 mypy 或 basedpyright 作为交叉检查。锁定版本，这样工具更新不会悄然改变触发的诊断。

## 它能捕获的与可能遗漏的

ty 捕获广泛的静态类型错误：参数不匹配、错误的返回值、`None` 处理、收窄失败和不可达分支。它可能遗漏的场景涉及不成熟的第三方桩文件、最新的语言语义，或者旧检查器花费多年时间强化的推断角落。当 ty 和成熟检查器在困难案例上存在分歧时，应进行调查，而不是假设 ty 是错误的或正确的。
