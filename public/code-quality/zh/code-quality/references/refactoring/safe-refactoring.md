# 安全重构（Safe Refactoring）

## 行为等价是契约

重构的唯一承诺，正如 [fowler-refactoring.md](./fowler-refactoring.md) 所定义的，是**外部可观察行为不会改变**。相同的输入产生相同的输出；调用方依赖的副作用仍然发生；原本引发的错误仍然引发。内部结构你可以随意更改，只要外部没有人能看出区别。本文档中的一切内容都是为了让你在保持这一承诺的前提下进行结构变更——并且关键是*证明*你保持了它，而不是仅仅希望如此。

"可观察"在这个定义中确实起作用。私有方法调用的确切序列不可观察。返回值、写入的文件、发出的事件、引发的异常、消费者解析的日志内容——这些是可观察的。在靠近边界重构之前，要清楚什么是实际上可观察的，因为那是你不可逾越的界线。对于公共 API、序列化模式或 CLI 参数，可观察表面很广，约束也很严格。

## 在动结构之前先有测试

你无法保留你无法观察到的行为，而没有快速检查的方法也无法确认保留。因此重构的前提条件是一个安全网：一套能够覆盖你即将重组的行为的测试套件，并且快得足以在每一步小步骤后运行。如果该套件存在且为绿色，重构风险很低。如果不存在，你要么先构建它，要么不重构。

这不是可选的严谨。重组行为未经验证的代码是编辑加祈祷，这也是重构引入缺陷的最常见方式。整个方法依赖于反馈循环是可信的。

## 特征测试（Characterization Tests）

当你需要重构没有测试的代码——典型情况是遗留代码或代理快速生成的代码——你首先编写**特征测试**。特征测试不是断言代码*应该*做什么；它钉住代码*当前*做什么，包括任何怪癖。你运行代码，观察输出，并编写一个恰好断言该输出的测试。该测试现在刻画了现有行为。

其目的不是正确性——现有行为甚至可能有轻微错误——目的是一个绊线。一旦当前行为被钉住，你可以自由重构，任何变红的测试都告诉你重组改变了某些东西。如果你后来决定旧行为是一个缺陷，修复它是一个单独的、有意的行为变更，附带其自己的测试更新，而不是在重构的掩护下溜进来。在 Python 中，快照/批准测试和捕获代表性输出是快速刻画变换密集型代码的实用方法。

## 小的可逆步骤

每一步应该足够小，以至于显而易见正确且容易撤销。重命名一个符号；运行测试。提取一个函数；运行测试。移动它；运行测试。这样做有效的原因：

- 失败定位到你刚做的一个变更，因此调试很简单。
- 你持续处于可发布状态，可以随时停止。
- 撤销单个小步骤代价很小；撤销一小时的混乱变更则不然。

频繁提交。每个重构步骤（或每小组合相关步骤）一个干净的提交意味着 `git` 本身是你回滚策略的一部分。

## IDE 和工具支持

自动化重构（重命名符号、[提取函数](./extract-function.md)、[内联](./inline-function.md)、[移动](./move-function.md)）比手动编辑更安全，因为工具机械地跨所有引用执行变换，避免了手动编辑引入的拼写错误和遗漏调用点。在可用时使用它们。

在 Python 中，要意识到工具支持比静态类型语言弱。动态分发、`getattr`、基于字符串的引用、monkey patching 和鸭子类型意味着重命名工具无法确定地找到每个引用。因此 Python 重构更依赖其他检查：类型检查器（mypy/pyright）捕获损坏的签名，`rg` 查找工具可能错过的文本引用，测试套件作为最终仲裁者。将自动化的 Python 重构视为快速的第一次通过，然后验证，而不是保证。

## 手动重构的风险场景

某些情况会击败通常的安全机制，需要额外小心：

- **反射、动态属性访问、字符串键分发。** 工具和 grep 都会漏掉这些。广泛搜索并依赖运行时测试。
- **公共 API 和序列化格式。** 可观察表面延伸到了你的代码库之外，到达你无法测试的消费者。将变更视为行为变更，而不是重构。
- **并发和顺序。** 重排操作可能会改变行为，即使每一步在单线程测试中看起来是保持行为的。
- **带有你无法轻松在测试中重现的副作用的代码**（外部服务、时间、随机性）。先将副作用隔离在一个接缝之后，使核心变得可测试。

## 回滚策略

重构应该始终有一个干净的出口。开始前，确保你的工作树已提交，以便有一个已知良好的点可以返回。采用小步骤并频繁提交，以便回滚单元很小。如果某一步失败且原因不明显，撤销该步骤而不是尝试向前修补——纪律是回到绿色状态，而不是在损坏状态之上堆积修复。而且，如果你中途发现你所朝向的抽象是错误的，正确的做法通常是[内联](./inline-function.md)回之前的结构并重新找到真正的接缝，而不是继续推进。一个可以干净放弃的重构才是安全的重构。
