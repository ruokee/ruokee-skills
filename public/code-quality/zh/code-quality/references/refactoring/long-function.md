# 过长函数（Long Function）

## 什么是过长函数

过长函数是最常见的臃肿体，也是最容易被误判的。这个坏味不在于按行数衡量的长度——一个函数"太长"是因为它**混合了抽象层级、承担了多个职责、或者有需要完整阅读才能理解的子部分**。长度是提示其中某个问题正在发生的线索，而不是问题本身。一个四十行的函数如果是一个单一抽象层级的清晰序列，就没问题；一个十二行的函数如果交织了高层策略和底层字节操作，则不行。

可靠的测试是看你能不能诚实地命名这个函数。如果名称是内部所有内容的真实概括，函数就是内聚的。如果诚实的名称应该是"验证订单，*并且*计算总额，*并且*格式化收据，*并且*发送邮件"，那它在做多件事，这些"并且"标记了它想要拆分的地方。

## 什么使函数过长

- **混合抽象层级。** 函数体在"做什么"（编排业务流程）和"怎么做"（字符串格式化、索引运算、字典操作）之间跳跃。读者必须不断转换思维高度。每个底层块都想成为一个命名函数，以便高层流程能清晰阅读。
- **多个职责。** 不同的阶段——解析、然后验证、然后转换、然后持久化——打包在一个函数中。每个阶段都是提取的候选，尤其是当它们通过少数几个局部变量通信时，这些变量形成了自然的接口。
- **难以命名的子部分。** 一个明显的信号是引入代码块的注释：`# 现在计算折扣`。注释在命名代码拒绝命名的概念。这个代码块通常是一个待发生的提取函数（Extract Function），注释将成为函数名。
- **深度嵌套和长条件语句。** 箭头形代码，守卫子句或提取谓词可以展平逻辑并揭示意图。

## 提取信号

在以下情况下使用 [extract-function.md](./extract-function.md)：

- 你写了注释来解释接下来的几行——提取它们，让名称承载注释。
- 一组行使用了一个紧密的局部变量集群，而函数的其余部分不碰这些变量——这个集群就是一个函数边界。
- 你发现自己需要滚动才能将整个函数记在脑中。
- 某个阶段本可以隔离测试，但因为被埋没而无法做到。

## 何时长度是可接受的

并非每个长函数都是坏味，强制拆分可能通过将线性故事分散到许多微小的函数中（读者必须重新组装）而使代码更糟。在以下情况下长度是可接受的：

- **它是同一抽象层级上的一个清晰序列。** 一个做十个步骤的函数，按顺序，在同一层级，没有分支复杂性，像散文一样从上到下阅读。拆分它只会增加跳转成本。参见与 [thin-wrapper-function.md](./thin-wrapper-function.md) 的张力——提取一个只使用一次、没有抽象边界的步骤，通常会产生一个比原来的长度更糟的浅层辅助函数。
- **配置或数据。** 长的字面量表、大的映射、一组常量——这里的长度只是数据量，而不是逻辑复杂性。
- **测试设置和测试体。** 测试即使很长，也常常以显式的 setup-act-assert 形式读起来更好；过度提取测试辅助函数会隐藏测试实际在验证什么。
- **一个 `match` 或多分支分发**，其中每个分支很短且结构是扁平的。

## 在 Python 中

- 将阶段提取为模块级函数或方法；它们共享的局部变量变为参数和返回值，这通常能澄清数据流。
- 守卫子句（`if not valid: raise`）能有效地展平嵌套并且可读性好。
- 注意每个提取的函数都会增加一个调用帧；在真正的热循环中这是可观的，尽管对普通代码来说清晰性胜出。相关权衡在 [thin-wrapper-function.md](./thin-wrapper-function.md) 中描述。
- 列表推导式或生成器可以替代在循环中累加的数据块，缩短函数而不隐藏任何东西。

## 与其他坏味和重构的关系

过长函数是 [extract-function.md](./extract-function.md) 的主要触发器。它经常与过长参数列表和数据泥团一起出现——当提取产生的函数需要相同的五个参数时，这些参数在要求成为一个参数对象。它也可能是一个缺失抽象的症状：在代理辅助代码中，函数变长是因为每个新需求都被追加为另一个分支，使函数成为变更的倾倒场；这更接近 [divergent-change.md](./divergent-change.md)，修复方式是找到变化轴而不是继续提取。
