# KISS — 保持简单（Keep It Simple）

## 它是什么

KISS 说：选择满足实际目标的最简单设计。陷阱在于将"简单"理解为"简短"。简单不是用代码行数来衡量的。它是用一个读者为了理解和安全地修改代码而必须在脑中容纳的概念数量来衡量的：依赖的数量、隐藏的控制流数量、状态的广度，以及变更需要传播多远。

一个更具操作性的表述：最小化*不必要的*复杂度——你添加的复杂度，而不是问题本身固有的复杂度。

## 必要复杂度 vs 意外复杂度

这个区分是整个原则的核心。

- **必要（本质）复杂度**来自问题本身。一个正确的分布式共识协议是复杂的，因为共识本身就很难。一个税务引擎是复杂的，因为税法本身就复杂。你无法消除这些。
- **意外复杂度**来自解决方案：投机性的抽象层、没人要的配置旋钮、为单个调用点构建的框架、只为看起来灵活而存在的间接层、需要十分钟才能读懂的聪明单行代码。

KISS 针对的是意外复杂度。它不要求你假装一个困难的问题是简单的。当领域确实复杂时，目标是将该复杂度*包含*在一个边界良好、接口简单的模块内部（参见 [deep-modules.md](./deep-modules.md)），而不是将其稀薄地散布在整个系统中。

## 简单是"更少的概念"，而非"更少的行数"

两个清晰、略微重复的副本通常比一个带有三个布尔标志和一个回调的通用函数更简单。一个扁平的语句序列通常比一串迫使读者在文件中跳来跳去的小型辅助函数更简单。一个普通的 `dict` 查找通常比一个带有插件协议的注册表更简单。当你压缩行数却倍增概念时，你让代码变得更短也更难——这与 KISS 背道而驰。

这就是为什么 KISS 与天真的 [dry.md](./dry.md) 以及泛泛的"小函数"偏好存在张力：两者都可能用抽象的、更难掌握的结构来换取具体、可读的重复。

## 过度工程化的症状

- 只有一种实现的抽象、接口或基类。
- 没有当前调用者使用的配置选项、标志或扩展点（这也是 [yagni.md](./yagni.md) 的违规）。
- 唯一理由是"灵活性"或"以防以后需要替换"的间接层。
- 为服务单一具体用途而构建的框架或插件系统。
- 许多单行包装函数，重命名了表达式但没有隐藏复杂度。
- 聪明的元编程、动态分发或通用机制，而直接的调用就可以完成。

## 何时复杂度是合理的

KISS 不是偷工减料的借口。添加的复杂度只有在服务于以下目标时才有其价值：

- **正确性（Correctness）**——处理问题实际具有的真实边缘情况、并发风险或不变量。
- **性能（Performance）**——在热路径上经过测量的必要优化（复杂度被局部化并记录原因）。
- **安全性（Safety）**——系统边界的输入验证、资源清理、对实际发生的失败的错误处理。

以"简单"为名去除这些不是 KISS，而是制造脆弱代码。其技巧在于区分那些为正确性、性能或安全性买单的复杂度和那些只为了看起来高大上的复杂度。

## 在 Python 中

- 默认使用清晰的数据结构、直接的控制流、标准库以及少量命名良好的函数。
- 当复杂度不可避免时，将其隔离在模块、适配器或深模块（deep module）内部，使调用方保持简单。
- 不要为单个调用点创建抽象，也不要把所有东西塞进一个巨型函数——两者都是向相反方向对简单的背离。
- 让格式化工具和 linter 处理风格问题，以便人类注意力集中在边界和行为上。

## 与其他原则的交互

- [yagni.md](./yagni.md)：YAGNI 移除投机性功能；KISS 移除你所构建功能中的不必要复杂度。二者共同抵制过度工程化。
- [deep-modules.md](./deep-modules.md)：长期来看，处理真正复杂度的简单方式是将其隐藏在简单接口之后，而不是散布浅辅助函数。
- [dry.md](./dry.md)：当去重添加的概念比移除的还多时，KISS 胜出——保留清晰的重复。
