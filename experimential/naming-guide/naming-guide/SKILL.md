# 命名指南

这是一个指南式 skill。它保留清晰阶段，但核心不是机械执行流程，而是帮助 Agent 进入正确的命名心智：先理解名字要承担的体验和边界，再发散候选，最后把取舍交给用户。

## 命名心智

好名字通常不是功能说明书。直白名字容易让人第一眼觉得安全，例如 `agent-launcher`、`worker-runner`、`profile-manager`，但它们也很容易被同类项目淹没，缺少让用户记住的心理位置。

命名不是最后贴在项目上的标签，而是用户反复输入、搜索、转述、写进文档和代码里的长期资产。一个名字每被使用一次，都会积累一点认知。如果名字只有“解释功能”的能力，却没有辨识度、输入手感和用户喜欢，它很难在长期使用里留下位置。

命名时要先创造，再判断。过早评判会把候选压回熟悉、安全、平庸的区域。隐喻、复合词、生造词、跨领域词、第一眼不太习惯的词，都应该先允许进入候选池，再根据场景检查风险。

分歧不是自动坏信号。一个名字需要解释、有人第一眼不习惯，可能说明它有张力和辨识度；真正需要严肃处理的是误导型风险，例如暗示不存在的能力、错误的技术边界、强生态冲突或真实的 import/package/config 混淆。

不同命名表面承担的任务不同。产品名可以创造记忆位置，CLI 名要顺手，package 名要能安装和导入，模块名要让维护者理解边界。不要用同一套标准硬评所有名字，也不要假设一个名字必须覆盖所有表面。

## 总原则

命名的第一判断人是用户。Agent 的职责是把命名问题整理清楚，把候选发散充分，把各候选的适用条件和代价说清楚。

优先级从高到低：

1. 用户喜欢。
2. 日常输入顺手。
3. 有辨识度。
4. 大致贴合功能和边界。
5. 避开著名项目、强冲突和高频泛词。
6. 发音、是否合成词、是否需要解释。

最后一组只能作为提示，不能在用户没有明确重视时作为否决理由。

## 禁止事项

- 用户没有明确要求建议时，不写“推荐”“首选”“最佳”“胜出”“我倾向”“默认选”“最稳妥”“更值得选”“建议直接用”等排序式、结论式或暗示式偏向表达。
- 用户没有确认前，不写“最终选择”。
- 不用直白功能名冒充完整探索，例如只给 `agent-launcher`、`worker-runner`、`profile-manager`。
- 不因为需要解释、生造、复合词、发音不稳定就过早否决。
- 不把 registry 精确名占用当成 package 候选失败；先检查后缀、scope、组织前缀、拆分 package/bin 名或协商可能。
- 命名记录文档中已经和用户确认过的前置章节，不要因为后续分析而回头重写。
- 不要因为后续发现某个候选不够好，就删除或改写 `候选名称` 里的原始候选；把新判断写进 `命名分析`。
- 只有用户明确要求修改前文时，才回头改 `背景`、`命名简报`、`边界` 或已有候选。
- `候选名称` 中已有条目的顺序、来源、原始拼写和批次默认不可改；重复、拼写问题、占用问题和弱候选判断都写入 `命名分析`。

## 阶段 1：确定命名对象

先收集或从上下文推断：

- 命名对象：产品、项目、文件夹、repo、package/crate/npm/Python distribution、模块、CLI、API、类型、配置键、数据库资源或领域概念。
- 使用者：用户、开发者、维护者、调用方、CLI 高频使用者。
- 功能边界：它是什么，不是什么，拥有什么，不拥有什么。
- 命名表面：名字会出现在哪些地方。
- 用户偏好：喜欢、讨厌、语言、长度、是否接受隐喻/生造词、是否优先 CLI 手感。
- 工作模式：只探索候选、比较候选、维护命名记录、记录最终决策、执行重命名。

能读文件解决的不要问用户。

根据用户请求选择工作模式：

- 只探索候选：输出命名简报、边界、候选路线和候选表格，不自动创建完整命名记录。
- 比较候选：只比较已有候选或本轮候选，不追加 `决策`。
- 维护命名记录：按阶段 8 的追加式文档流程写入或更新命名记录。
- 记录最终决策：只有用户明确确认名字后，才在文档末尾追加 `决策`。
- 执行重命名：先确认命名记录中的最终选择，再修改代码、路径或文档。

## 阶段 2：写命名简报

生成候选前，写一段自然语言简报。简报不是候选理由，也不是最终结论，而是把命名问题交给另一个人时，对方仍能继续思考的上下文。

```markdown
我需要为一个 <对象> 取名。它用于 <场景>，面向 <用户>，核心能力是 <能力>。它不是 <相邻系统>，名字不应暗示 <误导含义>。这个名字需要能用于 <表面>。当前偏好是 <偏好>，需要避免 <禁用词>。
```

产品、项目、CLI 名再补充：

- 希望用户记住什么感觉。
- 希望与哪些相邻名字拉开距离。
- 希望显得稳、快、轻、私有、可控、锋利、安静，还是别的气质。
- 如果这个名字不直接解释功能，它还可以通过什么体验、隐喻或使用场景成立。

## 阶段 3：按表面设置权重

|表面|看重什么|少看重什么|
|-|-|-|
|产品/品牌名|喜欢、辨识度、记忆点、长期气质|不要求完全自解释|
|项目文件夹名|短、无空格、路径顺手、可检索|不要求公开品牌感|
|repo 名|可分享、可搜索、与项目对应|私有 repo 可更个人化|
|package/crate/npm/Python distribution|合法、可安装、规范化、导入路径、冲突处理|不要求精确名必须可用|
|模块名|边界准确、目录清楚、维护者易懂|不追求品牌感|
|类型/类名|抽象层级准确、调用处自然|不追求独特|
|函数/方法名|动作和副作用清楚|不追求品牌感|
|配置键|稳定、低歧义、便于检索|不追求短到难懂|
|CLI 命令|短、顺手、和子命令组合自然|不要求解释完整功能|
|API/数据库资源|领域一致、长期兼容、单复数清楚|少用隐喻|

如果多个表面都重要，拆开讨论。repo、package、library crate、bin/CLI 和配置目录可以不同名。

## 阶段 4：先发散路线

先列 3 到 6 条路线，每条路线给 2 到 4 个候选。路线之间要真正不同，不要只是同一批功能词的同义替换。

候选路线可以包括：

- 直白描述路线。
- 对象隐喻路线。
- 空间/容器路线。
- 动作路线。
- 工程/工艺路线。
- 领域术语路线。
- 超短 CLI 路线。
- 生造词/品牌路线。
- 缩写路线。

发散时主动尝试这些视角：

- 竞品会怎么叫。
- 禁止使用最直白功能词后还能怎么叫。
- 如果完全不提当前行业热词，还能叫什么。
- 从不相关类别借词，例如工艺、空间、地形、材料、工具。
- 把两个弱相关词组合后，是否能产生更强的记忆点。
- 如果每天在 CLI 输入几十次，还是否顺手。

输出格式：

```markdown
## 候选路线

### <路线名>

适合：

- ...

风险：

- ...

候选：

|候选|适配点|代价|适合表面|
|-|-|-|-|
|`...`|...|...|...|
```

## 阶段 5：测试衍生形式

不要只看裸词。根据对象测试真实使用形态：

- CLI：`name run`、`name status`、`name logs`。
- Rust：repo、package、library crate、bin、`NameConfig`、`NameState`、`NameBackend`。
- Python：distribution、module/import、CLI、class、config key。
- npm：package、scope、import path、bin。
- 产品：导航、标题、文档术语、API resource、内部模块。

候选在某个表面弱，不等于整体失败；记录它适合和不适合的表面。

## 阶段 6：风险分类

先区分风险类型，再决定是否要弱化或放弃候选。

分歧型风险：

- 需要解释。
- 第一眼不习惯。
- 风格偏隐喻、生造或复合。

分歧型风险不一定坏，可能代表辨识度。对个人项目、产品名和 CLI 名，用户喜欢和愿意长期使用通常比“所有人第一眼都无分歧”更重要。

误导型风险：

- 暗示不拥有的职责。
- 暗示错误技术能力。
- 与著名项目或标准命令强冲突。
- package/import/config 会造成真实混淆。

误导型风险需要认真处理。

package/crate/npm/Python distribution 的占用处理：

- 尝试 `-rs`、`-py`、`-cli`、`-tool` 等后缀。
- 尝试 scope 或组织前缀。
- 拆分产品名、repo 名、package 名和 CLI/bin 名。
- 对冷门或疑似废弃包，记录联系发布者协商的可能。

## 阶段 7：中立比较

探索阶段只用中立比较。比较的目标不是替用户收敛成唯一答案，而是让用户看到不同偏好会把决策推向哪里。

```markdown
|候选|适合什么偏好|主要代价|适合表面|
|-|-|-|-|
|`...`|偏短、偏品牌感|需要解释|产品 / CLI|
|`...`|偏清楚、偏工程|较普通|package / module|
```

可以写条件判断：

- 如果更重视 CLI 手感，哪组更贴近。
- 如果更重视 package 发布，哪组更贴近。
- 如果更重视辨识度，哪组可以继续看。

条件判断必须绑定用户偏好，不得写成单一结论。不要把“更贴近某个偏好”写成“应该选择某个候选”。

## 阶段 8：维护追加式命名记录

命名记录不是一次性生成的最终报告，而是随讨论追加演进的工作文档。默认采用追加式写法：先固定前置上下文，再确认候选线路，再收集候选名称，再追加分析，最后追加决策。

初始文档只写这些章节：

```markdown
# 命名讨论

## 背景

<为什么需要命名或重命名，旧名字的问题，已知约束。>

## 命名简报

<可复制给其他 AI 或同事征集名字的自然语言简报。>

## 边界

它是：

- ...

它不是：

- ...

它拥有：

- ...

它不拥有：

- ...

## 候选线路

<记录命名方向和分组，例如直白描述、对象隐喻、空间/容器、工程/工艺、超短 CLI、生造词/品牌等。这里描述路线，不直接替用户收敛到某个名字。>

## 候选名称

<按路线或来源记录候选。这里保留候选的原始讨论结果，不因为后续分析而删除。>
```

和用户反复确认 `背景`、`命名简报`、`边界`、`候选线路` 是否符合预期。只有用户明确表达“确认”“这样可以”“这一版固定”“按这个写”等意思时，才视为这些章节稳定。用户只是继续讨论、没有反对或临时表达偏好，不等于确认。

用户把命名简报拿去问其他 AI 或同事后，Agent 可以把新收集到的名字追加到 `候选名称`。追加时保留来源、批次和所属候选线路。不要为了整洁擅自重排、合并、改写、重命名或移出已有候选，例如：

```markdown
### 外部征集：<来源或日期>

|候选|候选线路|来源|简要说明|
|-|-|-|-|
|`...`|对象隐喻 / 超短 CLI / 其他|Claude / 同事 / 自己想到|...|
```

后续个人思考、讨论、调查和占用检查，只写入新增章节 `命名分析`。不要回头改前面的 `背景`、`命名简报`、`边界`、`候选线路` 或 `候选名称`，除非用户明确要求。

```markdown
## 命名分析

### <日期或讨论轮次>

- `name-a`：<新的判断、占用检查、风险、适用表面>。
- `name-b`：<新的判断、占用检查、风险、适用表面>。
```

如果分析发现某个候选很弱，也不要删除候选本身；在 `命名分析` 中说明它为什么变弱、在哪些表面不适合、是否仍有保留价值。

如果需要清理候选列表、合并重复项、规范化拼写或重排候选，先询问用户；获得明确要求后再做，并在 `命名分析` 或变更说明里记录清理原因。

最终经过用户决策后，在文档最后追加 `决策` 章节：

```markdown
## 决策

最终选择 `<name>`。

选择原因：

1. ...
2. ...

## 后续默认命名

|含义|名称|
|-|-|
|项目名|`...`|
|repo|`...`|
|package|`...`|
|CLI|`...`|
|配置目录|`...`|
|核心类型|`...`|
```

如果用户明确要求 Agent 给建议，可以给一个条件性建议，但必须写清楚建议依赖的偏好条件，并等待用户确认。建议也先写入 `命名分析`；只有用户确认后，才写入 `决策`。

## 结束检查

- 有命名简报。
- 有表面权重。
- 有多条路线。
- 创造和判断分开。
- 探索阶段没有“推荐/首选/最佳”。
- 用户喜欢、输入手感、辨识度被放在高优先级。
- registry 占用被视为可处理约束，而非自动否决。
- 没有因为名字需要解释、是复合词或第一眼有分歧就过度否决。
- 已确认的前置章节没有被无授权回写。
- 后续调查和否定意见写在 `命名分析`，没有擅自删除或改写 `候选名称`。
- 最终决定来自用户确认。
