# 重复代码（Duplicated Code）

## 什么是重复代码

重复代码是同一事物在多处表达的坏味。它被列在"可有可无者"中，因为移除真正的重复通常能让代码更清晰、变更成本更低。但它也是最多被误读的坏味，因为表面症状——两段看起来相似的片段——并不能告诉你是否存在真正的问题。整个判断取决于一个关键区分。

## 知识重复 vs 巧合相似

这是关键的区分，在任何一个方向上犯错都会造成损害。

**知识重复**是同一个*决策*在多个地方表达：一条业务规则、一个验证条件、一个模式、一个协议细节、一个税款计算、一个状态转换。检验标准是反事实的：*如果其中一个副本背后的需求发生了变化，其他副本也必须变化吗？* 如果是，它们编码了一份知识，存在于多处意味着未来的编辑会更新一些而遗漏其他，导致系统自相矛盾。这值得移除。

**巧合相似**是今天由于无关原因看起来相似、但受不同力量驱动的代码。两个具有相同结构但检查不同概念的验证器；两个具有相似样板但逻辑无关的处理函数；看起来像其他测试设置但钉住不同场景的测试初始化。如果一个背后的需求变了，其他不会动。这些片段在有意义的意义上不是重复——它们仅仅是押韵——而统一它们会耦合本应独立演化的事物。

消除巧合相似的错误比留下真正的重复更隐蔽，因为它主动引入了耦合和一个错误的抽象，未来的变更必须与之抗争。请参见 [duplicated-code.md](./duplicated-code.md) 的兄弟文档 [dry](references/design-principles/dry.md) 了解其背后的原则：DRY 关乎知识，而非文本。

## 何时提取

当你确认片段编码了一份知识时提取。对于**高风险知识**——安全、权限、金钱、协议契约、数据一致性、模式的单一真相源——立即提取，即使只在第二次出现。这些领域出现分叉的成本太高，不能等待。

对于普通代码，你比较确定但变化方向尚不清晰时，**三次法则**是刹车：写一次，容忍两次，在第三次出现时——或当变化方向变得清晰时——提取。到第三次实例时，你通常能看到什么真正变化、什么保持不变，这样你就能选择正确的接缝而不是猜测。参见 [rule-of-three](references/design-principles/rule-of-three.md)。

## 何时保留

在片段是巧合相似、或者它们真正相关但尚未清晰且你还没达到第三次实例时，保留重复。过早的统一会选择错误的抽象，而错误的抽象比它所替换的重复更昂贵——你花钱构建它，然后在情况分歧时再花钱拆除它。容忍两份副本一段时间是便宜且可逆的；一个不想要的抽象两者都不是。

## 抽象稳定性要求

提取之前，抽象的形状必须相当稳定。如果你还不能干净地命名被提取的事物，或者你已经能预见到下一个案例需要不同的参数或分支，接缝还没准备好。一个好的提取有清晰的名字、很少的参数、单一变更原因，并且隐藏了真正的复杂性。如果候选者不满足这些，重复在告诉你底层的概念尚未稳定——等待。

## 错误的去重

常见的失败模式，尤其在代理辅助代码中，是**比重复更难读的参数化包装**。你将两个相似片段提取到一个函数中，但它们并不完全相同，于是函数增长出一个布尔标志来切换行为，然后一个模式参数，然后一个回调来容纳真正不同的部分，然后一个用于边缘 case 的特殊分支。结果是一个没人能读的函数，配置方式没人能追踪——比两个诚实的副本更糟。当你发现这样的函数时，修复方法是内联它，将代码复制回去，然后重新找到真正的变化轴。一个在两个行为之间选择的标志参数通常意味着该函数本应是两个函数。

## 在 Python 中

- 对于重复的模式、API 契约或模型，建立单一真相源：`dataclass`、`TypedDict`、Pydantic、OpenAPI 规范或代码生成——而不是三个手工维护的副本。
- 表驱动映射和分发字典（dispatch dicts）能合并真正重复的知识（每行一个 case），而无需发明类层次。
- 倾向于提取为模块级函数或小型策略函数，而不是一个唯一目的是共享代码的基类。
- 注意同一领域规则出现在模式、服务和测试夹具中——这是一种频繁的代理编码模式，其中三个"副本"悄无声息地分叉。

变换本身通常是 [extract-function.md](./extract-function.md)；反向操作，当你发现错误的去重时，是 [inline-function.md](./inline-function.md)。
