# 测试驱动开发（Test-Driven Development）

测试驱动开发（TDD）是一种工作流，其中测试推动开发反馈循环。你草拟一个行为列表，选择一个，
编写一个失败的、可运行的小测试，编写刚好足够的代码使其通过，然后重构。Martin Fowler 将该循环
总结为**红-绿-重构（Red-Green-Refactor）**，并强调重构步骤不是可选的装饰——它是设计真正改进的地方。

TDD 主要不是一种测试技术。它是一种碰巧留下测试的设计技术。先写测试迫使你在接口存在之前使用它，
这能尽早暴露尴尬的签名和不清晰的职责。

## 红-绿-重构循环

1. **红（Red）。** 为一个小的行为编写测试并观察它失败。失败确认测试确实在测试某些东西，
   而不是静默通过。
2. **绿（Green）。** 编写最简单的代码使测试通过。不是优雅的版本——是足够的版本。在这里
   偷工减料是可以的；下一步会清理。
3. **重构（Refactor）。** 在测试变绿并保护行为的情况下，改进结构：重命名、提取、消除重复。
   在整个过程中测试保持绿色。

纪律是保持步骤小，并在每一步之间运行测试。一个常见的失败是压缩循环——在一个测试下编写大量代码，
或者在行为尚不稳定的绿色步骤中偷带一个大的结构性变更。

## TDD 带来的好处

- **设计反馈。** 先写测试意味着在实现之前消费 API。痛苦的设置、太多的参数或难以构建的协作者
  表现为测试痛苦，这是关于设计本身的信号。这与 [dependency-inversion](./dependency-inversion.md)
  相联系：难以测试的代码往往是其依赖未被注入的代码。
- **行为规约。** 测试套件以可执行的形式记录了代码应该做什么，不会静默地过时。
- **回归保护。** 一旦行为被测试固定，未来破坏它的变更会响亮地失败。这种保护正是使激进
  的[重构](references/refactoring/index.md)和 [YAGNI](./yagni.md) 安全的原因——你可以推迟抽象，
  并在以后重塑而无需恐惧。

## 何时 TDD 有价值

TDD 在行为可枚举且错误答案的成本真实的情况下回报最高：

- 纯函数、解析器、转换和序列化逻辑。
- 具有清晰输入和预期输出的领域规则——定价、验证、状态转换。
- Bug 修复：先编写重现 bug 的失败测试，然后修复它。该测试成为永久的回归守卫。
- API 设计，先使用接口可以揭示它是否易于调用。

## 何时严格的 TDD 适得其反

TDD 假设你对期望的行为有足够的了解来编写测试。当你没有时，严格的测试优先会成为摩擦：

- **探索性代码和试验性项目（Spikes）。** 当接口甚至需求还不清楚时，先写试验性代码来学习，
  然后在形态稳定后添加特征描述测试（characterization test）或合约测试。强制测试优先在这里
  测试的是猜测。
- **UI 和视觉布局。** 大部分价值在于外观和交互，单元测试很难捕获这些。测试 UI 背后的逻辑，
  而不是像素定位。
- **胶水代码（Glue code）。** 主要委托给经过充分测试的库的薄接线代码，从仅重新断言库行为的
  测试中获益甚微。

将 TDD 视为高价值反馈策略，而不是每行代码的道德义务。目标是正常工作、设计良好的软件，
具有足够的行为覆盖率——而不是测试优先的仪式或覆盖率百分比的奖杯。一个没有有意义断言的测试
只增加覆盖率，别无其他。

## 与"事后测试"和行为覆盖率的关系

在代码之后编写测试不是罪过；重要的属性是行为被覆盖且测试断言某些真实的东西。TDD 比事后测试
多提供的是红色步骤带来的设计压力。许多团队将两者混合：TDD 用于核心逻辑和 bug 修复，
事后测试用于先探索的代码。两者都应针对可观察行为在合理的边界处，而不是内部实现细节，
这样测试才能在重构中存活。参见 [law-of-demeter](./law-of-demeter.md) 和
[dependency-inversion](./dependency-inversion.md) 了解为什么深入到内部（深层 mock、
patch 私有成员）的测试是关于设计的坏味道，而不仅仅是测试本身。

## 在 Python 中

- `pytest` 非常适合小步 TDD；保持 fixture 直接，避免在其周围构建不可见的框架。
- 对于原型、UI 和复杂的外部集成，先做试验性项目，然后补充特征描述测试或合约测试。
- `monkeypatch` 很方便但容易过度使用。优先在边界传入 fake 或 stub，而不是修补深层内部；
  需要深层修补是一个依赖本应被注入的信号。
