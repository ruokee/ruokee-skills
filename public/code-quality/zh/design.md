# Skill 设计

## 原始需求

本 Skill 是为了解决一个特定问题：AI 编码智能体在缺乏领域特定的工程判断力时产出平庸。像"遵循最佳实践"或"编写干净的代码"这类通用指令无法给智能体提供可操作的内容。这些 Skills 提供结构化知识，让智能体能够对代码质量做出真正的工程决策。

核心需求：

1. **渐进式披露（Progressive disclosure）** —— 顶层 `SKILL.md` 仅做导航。它告诉智能体存在什么内容以及何时读取。详细知识存在于叶子文档中，按任务信号按需加载。
2. **每个叶子必须把一个概念教透** —— 不是检查清单，不是摘要。叶子文档应解释：概念是什么、解决什么问题、典型实现方式、常见变体、高级用法、适用场景、不适用场景、以及如何评估现有代码是否正确使用。读完叶子文档的读者应获得真正的理解，而不仅仅是一套规则列表。
3. **多种工作流** —— 快速审查（fast review）在每日开发后触发用于自检。完整审查（full review）必须显式请求。分析模式（analysis）支持头脑风暴、机制设计、代码分析和重构规划。
4. **通用且可移植** —— 这些 Skills 不绑定到特定仓库或特定个人的偏好。示例应是通用的。项目特定的偏好从外部文件读取，绝不硬编码到 Skill 本身。

## 设计理念

### 知识优于规则（Knowledge Before Rules）

叶子文档是一个主题的完整参考，只是同时也支持审查。它不是审查清单。

Skill 的名称是 `code-quality`，而不是 `code-quality-review`。它描述模式、标准和实践。审查是使用这些知识的其中一种工作流；分析和设计是其他用途。

### 关注点分离（Separation of Concerns）

- **Skill 文档** 定义通用的工程知识和判断框架。
- **工作流文档（Workflow documents）** 定义操作模式：如何进行快速审查、完整审查或分析会话。
- **偏好（Preferences）** 定义项目和用户特定的选择。它们位于 Skill 之外，在 `.agents/preferences/` 或等效位置。
- **工具（Tools）** 处理机械性执行。如果格式化工具、linter 或类型检查器能检测到某个问题，Skill 不会将其作为提示规则重复。

### 渐进式披露

本 Skill 遵循**渐进式披露**。

```
SKILL.md          → 进入条件、模式、路由表、输出约定、停止规则
workflow/         → 操作模式定义（fast-review、full-review、analysis）
references/       → 按主题组织的领域知识
  domain/index.md → 目录边界和路由（用于人工维护）
  domain/leaf.md  → 一个概念的完整阐述
```

智能体先读取 `SKILL.md`，再路由到相关工作流和叶子文档。它从不一次性加载所有内容。

### 叶子质量标准

每个叶子文档必须支持读者：

- 理解概念是什么以及解决什么问题。
- 识别典型实现和常见变体。
- 判断概念何时适用、何时不适用。
- 评估现有代码是否正确使用它。
- 发现风险、反模式（anti-patterns）和误用情况。
- 区分误报（false positives）和真正的问题。
- 形成最小充分的推荐意见。

叶子文档由人类和 AI 智能体共同阅读。它们必须足够准确、完整和清晰，以适应任一读者群体。

### 偏好机制

Skill 支持外部偏好，但不包含个人或项目特定的观点。

发现顺序：

1. 检查当前项目目录下的 `.agents/preferences/<skill-name>.md`。
2. 如果未找到，检查 `.agents/preferences/<skill-name>/index.md`。
3. 检查用户级目录，如 `~/.codex/`、`~/.claude/` 或等效的智能体配置位置。
4. 如果没有偏好存在，则使用通用指导静默继续。

偏好可以定义：架构约束、审查优先级、或对其他 Skills 的引用。

偏好不能覆盖系统权限、安全规则或显式的用户指令。

输出必须区分偏好驱动的指导与通用的工程事实。

## 关键边界

- `design-principles/`（设计原则）讨论判断框架。`design-patterns/`（设计模式）讨论具体实现。原则说"依赖抽象"；模式说"这里是如何用 Factory 解决特定的创建问题"。
- `refactoring/`（重构）围绕 Fowler 的"坏味 + 保持行为不变的技术"。它不是"代码改进"的同义词。
- 智能体产生的薄包装（thin wrappers）、过早抽象（premature abstractions）和错误的 DRY 应属于 `refactoring/` 或 `design-principles/`，而非 `agentic-coding/`。`agentic-coding/` 目录仅覆盖配置层面的坏味。

补充说明：

- `programming-paradigms/state-machine.md` 是状态机参考。`design-patterns/state.md` 仅涵盖 GoF 状态模式（State Pattern）作为一种面向对象实现选项。
- `programming-paradigms/resource-lifecycle.md` 涵盖通用概念。语言特定的实现（如 Python 的 `with`）只是示例。

## 原则总结

1. 叶子文档完整地教授概念；它们不仅仅是检查清单。
2. 工作流文档定义操作模式；它们不混入叶子内容。
3. 偏好是外部的且明确标注；它们从不伪装成通用事实。
4. 工具处理机械性规则；Skill 处理判断。
5. 渐进式披露最小化上下文负担；智能体只读取任务所需的内容。
6. 示例是通用的且不涉及特定仓库代码。
7. Skill 可跨项目、跨智能体、跨用户移植。
