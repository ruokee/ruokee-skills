# Safe Refactoring

## 行为等价是契约

正如 [fowler-refactoring.md](./fowler-refactoring.md) 所定义的，refactoring 唯一的承诺是：**外部可观察行为不变**。同样的输入产生同样的输出；调用方依赖的副作用仍然发生；原本会抛出的错误仍然抛出。内部结构可以随你怎么改，只要外部看不出差别。本文的存在，就是为了让你在保留这个承诺的同时做结构修改 - 而且关键是，_证明_ 你保住了它，而不是靠猜。

“Observable” 在这个定义里是真正有分量的。私有方法调用的精确顺序不可观察。返回值、写出的文件、发出的事件、抛出的异常、某个消费者会解析的日志内容 - 这些才是可观察的。靠近边界做 refactoring 之前，先弄清楚到底什么是真的可观察，因为那就是你不能跨过去的线。对于 public API、serialization schema 或 CLI arguments，这条可观察边界会很宽，约束也会很紧。

## 先有测试，再动结构

你不能保证自己观察不到的东西不变，也无法在没有快速验证方式的情况下确认它没变。所以 refactoring 的前提是安全网：一套覆盖你即将重组的行为、并且足够快、可以在每个小步之后运行的测试。如果这套测试存在而且是绿的，refactoring 风险就低；如果不存在，就先把它建起来，否则不要动结构。

这不是“认真一点就好”的可选项。对未验证行为的代码做结构重组，本质上就是在改代码并祈祷，这也是 refactoring 最常见的 bug 来源。整个方法是否成立，全靠这个反馈回路是否可信。

## 特征测试

当你需要重构一段没有测试的代码 - 通常是遗留代码，或者是 agent 很快生成出来的东西 - 应该先写 **characterization tests**。这种测试不声明代码 _应该_ 做什么，而是把它 _当前_ 在做什么钉住，包括任何怪癖。你运行代码、观察输出，然后写一个断言，精确地锁定这个输出。这样测试就刻画了现有行为。

重点不是正确性 - 现有行为甚至可能有点错 - 重点是一个警报器。一旦当前行为被钉住，你就可以放心重构；任何变红的测试都在告诉你，重组改变了某些东西。如果你之后判断旧行为本来就是 bug，那么修它就应该是另一个独立的行为变更，并伴随自己的测试更新，而不是借着 refactoring 偷偷带进去。对于以转换为主的代码，在 Python 里，snapshot/approval testing 和捕获有代表性的输出是很实用的刻画方式。

## 小而可逆的步骤

每一步都应该足够小，小到明显正确、也容易撤销。改一个符号名；跑测试。提一个函数；跑测试。把它移到另一个 class；跑测试。这样做之所以有效，是因为：

- 一旦出问题，故障会局限在你刚刚做的那一步，调试就很简单。
- 你始终处于可交付状态，随时都可以停。
- 回滚一个小步骤的成本很低；回滚一个纠缠了一小时的改动堆却不低。

频繁提交。每个 refactoring 步骤一个干净 commit，或者一小组相关步骤一个 commit，这样 `git` 本身就成了你的回滚策略的一部分。

## IDE 和工具支持

自动 refactoring（重命名符号、[extract function](./extract-function.md)、[inline](./inline-function.md)、[move](./move-function.md)）比手工编辑更安全，因为工具会机械地处理所有引用，不会出现手工修改时容易引入的拼写错误和漏改调用点。只要可用，就尽量使用它们。

在 Python 里，要知道工具能力比静态类型语言弱。动态分发、`getattr`、基于字符串的引用、monkeypatch 和 duck typing 都意味着重命名工具无法百分之百找到所有引用。所以 Python 的 refactoring 更依赖其他检查：用类型检查器（mypy/pyright）捕捉断裂的签名，用 `rg` 搜索工具可能遗漏的文本引用，再把测试套件作为最终裁判。对 Python 自动 refactoring 的态度应当是：把它当作快速第一遍，然后再验证，而不是把它当作保证。

## 何时手工重构更危险

有些情况会击穿通常的安全机制，需要额外小心：

- **反射、动态属性访问、字符串键分发。** 工具和 grep 都会漏掉这些。要广搜，并依赖运行时测试。
- **公共 API 和序列化格式。** 可观察面已经扩展到你无法测试的外部消费者。应把这类变更当成行为变更，而不是 refactoring。
- **并发和顺序。** 重新排列操作会改变行为，即使单线程测试看起来每一步都保留了行为。
- **有些副作用你很难在测试里复现的代码**（外部服务、时间、随机性）。先把这些副作用隔离进一个 seam，让核心逻辑变得可测试。

## 回滚策略

Refactoring 应该永远有一个清晰出口。开始前，先确保工作区已经提交，这样你有一个已知的好状态可以退回。小步前进并频繁提交，这样回滚单位就很小。如果某一步失败，而且原因一时看不清，就回退那一步，不要试图在坏状态上继续补丁式推进 - 纪律的目标是回到 green，而不是在坏基础上堆更多修补。如果你中途发现自己朝着错误的抽象走去，通常最正确的动作是先 [inline](./inline-function.md) 回原来的结构，再重新寻找真正的 seam，而不是硬着头皮往前冲。一个你能干净放弃的 refactoring，才是安全的。
