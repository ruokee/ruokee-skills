# 分析模式（Analysis Mode）——代码质量

咨询模式。用户在询问*该做什么*，而非*哪里有问题*。无需评估 diff。帮助理性思考设计或重构决策并收敛到一个方案。

## 触发条件（Trigger）

- 设计讨论："我应该如何组织这个？"、"这个行为应该放在哪里？"
- 模式选择："哪个模式更适合——策略模式还是直接用函数？"、"这里值得用工厂模式吗？"
- 重构规划：如何拆分模块、编排安全的重构步骤，或决定是否应该重构。
- 范式选择：命令式 vs 函数式核心、面向对象 vs 数据导向、何时状态机值得使用。
- 原则应用：DRY、KISS、YAGNI 或 SOLID 如何作用于具体场景。

## 步骤（Steps）

1. 理解设计问题及其上下文——预期会发生什么变更、当前痛点是什么、存在哪些约束。如果缺少关键事实，请向用户提问。
2. 读取与主题相关的参考文档（`references/design-principles/`、`design-patterns/`、`refactoring/`、`programming-paradigms/`），使讨论使用项目真实的词汇。
3. 呈现选项及其权衡。大多数设计问题有多个合理的答案；展示现实可行的方案及其在变更成本、可读性和复杂性方面的代价。
4. 在原则、模式和范式真正适用的地方，以此框架展开讨论——但仅在模式所管理的变化点确实存在时才引用它。
5. 收敛。针对*当前*上下文推荐一个选项并解释原因，或提出一个聚焦的澄清问题。不要仅仅罗列选项。

## 输出（Output）

对话式，以推荐为导向。无需固定的发现项格式。列出备选方案，指出决定性的权衡，然后给出建议。输出与问题的规模成比例。

## 停止规则（Stop Rules）

- 仅提供咨询。不修改文件。
- 不要强行将原则或模式套用到不合适的场景——没有模式往往是正确的答案。
- 不要仅以相似性为由推荐抽象。
- 由用户做决定。如果讨论转向实际工作，建议切换到快速审查或完整审查，而非静默切换。
