# 告诉，不要问（Tell, Don't Ask）

告诉，不要问（Tell, Don't Ask）说的是：不要从对象中拉取数据并在外部对其做决策，
而是将决策推入拥有该数据的对象中。告诉对象你想要做什么；不要询问它的内部然后代表它行动。

该原则是关于*行为相对于其所依赖的数据的位置*。当数据和支配它的规则放在一起时，规则可以
依赖该对象维护的不变量，对这些规则的更改只有一个家。当调用者反复读取字段并在外部组合规则时，
逻辑散落各处，对象无法保护自身的一致性。

## 问题的形态

"问"式代码从外部读取状态并进行分支：

```python
# 问：调用者拉取数据并决策
if account.balance >= amount and not account.is_frozen:
    account.balance -= amount
    account.last_debit = now()
```

每个对账户扣款的调用者都必须记住同样的检查和同样的记账操作。如果出现新规则（透支限额、
冻结、审计日志），每个调用点都需要更新，任何遗漏的点都会使账户处于无效状态。

"告诉"式代码将决策移入内部：

```python
# 告诉：对象拥有规则及其不变量
account.debit(amount)   # 如果冻结或余额不足则抛出异常
```

现在账户实施自己的不变量。调用者表达意图。规则有一个家，通过此路径该对象永远不会被驱动到不一致状态。

## 与封装的关系

告诉，不要问是从调用者角度看的封装。封装说对象应该隐藏其内部状态；告诉，不要问说调用者
首先就不应该需要该状态——他们应该给对象一个命令并相信它能维护自身的不变量。两者相互强化：
一个暴露行为而非原始状态的对象保持其数据的私有性和意义。

它也与 GRASP 的[信息专家](./grasp.md)启发法（将职责分配给拥有完成该职责所需信息的类）
以及[迪米特法则](./law-of-demeter.md)（不要穿透结构来做外部决策）重叠。三者都将行为推向数据。

## 何时"问"是合适的

告诉，不要问是保护拥有行为的对象中不变量的指南。它不是对读取数据的禁止。当没有不变量需要保护时，
"问"是正确的：

- **查询和报表（Queries and reporting）。** 计算总数、聚合和汇总本质上是跨多个对象读取多个字段。
  强制将其转化为每个对象上的命令方法会扭曲模型。
- **序列化和 DTO。** 将对象转为 JSON、行或线格式是纯数据访问。数据传输对象或 API schema
  应直接暴露其字段。
- **UI 渲染。** 展示代码读取模型状态以进行绘制。那是它的工作；它不是在做模型应拥有的领域决策。
- **读模型（Read models）。** 在将读取与写入分离的系统中，读侧是故意面向数据和透明的。

区分标准：调用者是在*决定对象应拥有的东西*（业务规则、状态转换、不变量），还是在*报告/转换/展示*
  genuinely 属于调用者关注点的状态？前者是告诉，不要问的违规；后者是正常且健康的。

## 常见错误

- **消除所有 getter。** 一些读者将该原则理解为对象永远不应暴露数据。这会破坏报表、序列化和 UI，
  并产生尴尬的仅命令模型。查询是没问题的。
- **将行为塞进 DTO 和 dataclass。** 纯数据载体应该保持为数据载体。将领域命令添加到传输对象
  模糊了其边界，并将其耦合到它不应该知道的规则。只有数据有不变量需要保护时才将行为添加到数据。
- **将昂贵工作隐藏在看似无害的属性后面。** 一个触发 I/O 或大量计算的 `property` 会令期望
  廉价属性访问的调用者感到惊讶。告诉，不要问主张行为靠近数据，而不是将副作用伪装成属性读取。

## 在 Python 中

- 给领域对象语义命令方法：`invoice.mark_paid()`、`order.cancel()`、`account.debit(amount)`。
  这些方法在内部维护状态转换和不变量。
- 让数据载体、API schema、ORM 行和配置对象直接暴露属性。
- 使用 `property` 统一存储值和派生值，但要保持廉价且无副作用——不要在其后隐藏 I/O。
- 在函数式核心中，行为不必须是方法：一个接受数据并返回决策的命名函数
  （`def can_debit(account, amount) -> bool`）将逻辑和数据放在一起，而不强制面向对象。
  告诉，不要问是关于将规则和数据放在一起，而不是坚持使用方法。
