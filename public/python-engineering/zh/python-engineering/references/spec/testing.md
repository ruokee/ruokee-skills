# Testing

测试套件既是防回归的安全网，也是对代码应当如何工作的描述。两项职责都依赖同一条纪律 - 测试 _行为_，而不是实现。固定可观察行为的测试能在重构后继续存在，并记录意图；固定内部调用的测试则会在每次重构中断裂，而且几乎不记录任何有用信息。本文讨论的是如何组织测试，使它们同时做好这两件事。

## 测试组织

把测试放在顶层 `tests/` 目录中，与生产 package 分离，这样它们既不会被发布出去，也能像真实消费者那样使用 package（关于为什么 src layout 会强化这一点，见 [structure](../project/structure.md)）。测试文件和函数应按被测试的 _行为_ 命名，而不是按它们碰巧触达的实现文件命名 - `test_expired_token_is_rejected` 会告诉读者系统保证了什么；`test_validate` 只会告诉他哪个函数运行过。对于大型 library 或 framework，粗略镜像 package tree 有助于定位测试，但这种镜像只是导航辅助，而不是每个 module 都必须有并行测试文件的规则。测试应像生产代码一样认真对待：名字清晰、没有复制粘贴蔓延、意图明确。

## Fixture 设计

fixture 为测试提供一个准备好的 object 或环境，并可选地在之后清理它。好的 fixture 应该小而可组合，并以它们提供的内容命名（`temp_db`、`authenticated_client`）；其作用域也应尽量不比必要的大 - function-scoped fixture 会给每个测试提供新实例并保持测试独立，而 session-scoped fixture 则以隔离换取速度，只应保留给真正昂贵且只读的 setup。不要构造一个会生成整个世界的庞大 fixture；应该组合小 fixture，让每个测试只拿到自己需要的东西，读者也能从参数列表看出测试依赖什么。当 setup 需要 cleanup - 临时目录、打开的连接 - 应把它写成一个 yield fixture，使 cleanup 即使在测试失败时也会执行。

## 参数化

当同一行为应对许多输入都成立时，参数化测试可以把每个 case 作为数据表达出来，而不是复制多个测试 body：

```python
import pytest


@pytest.mark.parametrize(
    ("raw", "expected"),
    [
        ("  Alice ", "alice"),
        ("BOB", "bob"),
        ("", ""),
    ],
)
def test_normalize_username(raw: str, expected: str) -> None:
    assert normalize_username(raw) == expected
```

每个 case 都会被报告为独立测试，因此失败时会点名具体出错的输入。参数化适合边界和等价类测试 - 空输入、典型情况、边缘情况 - 它能把 case 以表格形式保留，而不是埋进循环里。当 case 需要真正不同的 setup 或断言时，它就不是好工具；把不同场景硬塞进同一个参数化 body，只会产生带分支逻辑的测试，比拆成多个测试更难读。

## 以行为覆盖取代行覆盖

行覆盖只衡量哪些行执行过，并不说明有没有真正检查任何东西 - 一个调用函数却不做断言的测试可以拿到 100% 覆盖，却什么都没验证。真正重要的是 _行为_ 覆盖：典型路径、边界、错误路径和不变量。断言应指向可观察结果 - 返回值、抛出的异常、记录的副作用 - 并确保重要 case（空值、边界、失败、并发）确实被覆盖。coverage 工具对找出“完全没有测试触达”的代码很有用，这确实是信号；但如果把它当作质量目标，它就会误导人，因为高百分比并不说明断言质量。应把它当作找盲区的工具，而不是宣布套件完成的证据。

## Mock 边界

mock 会用一个替身替换真实 dependency。应该在系统的 _边界_ 上 mock - network call、clock、filesystem、external service - 也就是那些真实对象慢、不可预测，或者在测试中承受不起副作用的地方。不要 mock 被测试的代码或它的近邻协作者；如果测试去 mock 它正在练习的对象，就会变成“代码按照测试告诉它的方式调用了 mock”的循环论证，一旦重构就会坏掉。mock 越多，测试对真实行为的证明就越少，因此应优先对便宜且确定性的东西使用真实对象，把 mock 留给真正的边缘。若某段代码不 mock 周围一切就很难测试，这通常是一个设计信号 - 依赖应该作为参数注入，而不是在内部直接去抓。

## 测试即文档

写得好的测试是项目里最可靠的文档，因为它不会在代码不变的情况下自行漂移。想了解某个函数如何使用的读者，应该能打开它的测试，看到真实的调用和预期结果。这可以有意识地做到：使用 arrange-act-assert 结构，让每个测试都像 setup、action、expectation；每个测试只验证一个行为，这样失败时就只指向一件事；名字要直接陈述保证。当测试以这种方式编写时，它们就同时也是示例，这也是 test-driven development - 先写测试来固定预期行为，再让实现通过 - 会在一次流程中同时产出设计工具和活文档的原因。用测试来驱动设计，也往往会推动代码走向更小、可注入的单元，而这正是生产代码更易维护的性质。
