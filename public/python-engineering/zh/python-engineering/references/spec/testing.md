# 测试

测试套件同时具备两个身份：捕获回归的安全网，以及代码应做什么的描述。这两个任务都由同一纪律来服务——测试*行为*，而非实现。固定可观察行为的测试经得起重构并记录了意图；固定内部调用的测试每次重构都会失败，且不记录任何有用的东西。本文档旨在组织测试，使它们能同时做好这两项工作。

## 测试组织

将测试放在顶层的 `tests/` 目录中，与生产包分离，这样它们永远不会被交付，并且它们以真实使用者会采用的方式使用包（参见 [structure](references/project/structure.md) 了解为什么 src 布局强化了这一点）。根据所测试的*行为*命名测试文件和函数，而不是它们恰好触及的实现文件——`test_expired_token_is_rejected` 告诉读者系统保证什么；`test_validate` 只告诉读者哪个函数运行了。对于大型库或框架，松散地镜像包树有助于定位测试，但镜像只是导航辅助工具，而非每个模块都必须有对应测试文件的规则。测试应得到与生产代码同等的关注：清晰的名称、无复制粘贴的蔓延、以及明显的意图。

## 夹具设计

夹具（fixture）为测试提供准备好的对象或环境，并可选择在测试后进行清理。好的夹具小巧、可组合、以其提供的内容命名（`temp_db`、`authenticated_client`），且作用域不超过必要的范围——函数作用域的夹具为每个测试提供新实例并保持测试独立，而会话作用域的夹具用隔离换取速度，应仅保留给真正昂贵且只读的初始化。抵制构建一个构造整个世界的巨型夹具；组合小夹具，使每个测试只引入它需要的内容，读者可以从参数列表中看到测试依赖什么。当初始化需要清理时——临时目录、打开的连接——将其表达为先 yield 再清理的夹具，这样即使测试失败也能执行清理。

## 参数化

当相同的行为应在多个输入上成立时，参数化测试将每个案例表达为数据而不是重复的测试体：

```python
import pytest


@pytest.mark.parametrize(
    ("raw", "expected"),
    [
        ("  Alice ", "alice"),
        ("BOB", "bob"),
        ("", ""),
    ],
)
def test_normalize_username(raw: str, expected: str) -> None:
    assert normalize_username(raw) == expected
```

每个案例被报告为一个独立的测试，因此失败会指出确切导致问题的输入。参数化是边界和等价类测试的正确工具——空输入、典型情况、边界——它使案例以表格形式可见，而不是埋藏在循环中。当案例需要真正不同的初始化或断言时，它就是不合适的工具；强制不同的场景通过一个参数化体会产生带有分支逻辑的测试，这比独立的测试更难阅读。

## 行为覆盖胜过行覆盖

行覆盖衡量哪些行运行了，而不是是否实际检查了什么——一个调用函数但不作任何断言的测试可以报告 100% 的覆盖率，却什么也没有验证。重要的是*行为*覆盖：典型路径、边界、错误路径和不变量。将断言对准可观察的结果——返回值、引发的异常、记录的副作用——并确保重要的案例（空、边界、失败、并发）确实被测试到。覆盖率工具在发现*没有*任何测试触及的代码时很有用，这是一个真实的信号；但当被当作质量目标时就具有误导性，因为高百分比并不说明断言质量的好坏。用它来发现盲点，而不是宣布套件完成。

## 模拟边界

模拟（mock）用一个替身替代真实的依赖。在*系统边界*处进行模拟——网络调用、时钟、文件系统、外部服务——真实的东西缓慢、不确定或具有你无法在测试中承担的副作用。不要模拟被测代码或其紧密协作者；模拟被测对象本身的测试最终只会断言你的代码按测试指示的方式调用了你的模拟，这是循环论证，且会在任何重构中失败。测试模拟得越多，它对真实行为的证明力就越弱，因此对于任何廉价且确定性的内容优先使用真实对象，将模拟保留给真正的边界。当一段代码不模拟其周围的一切就很难测试时，这通常是一个设计信号——依赖希望被作为参数注入，而不是在内部获取。

## 测试作为文档

编写良好的测试是项目拥有的最可靠的文档，因为它不与代码脱节就无法失败。想知道如何使用函数的读者应该能够打开其测试，看到带有预期结果的实际调用。这可以有意实现：arrange-act-assert 结构使每个测试读作初始化、操作、期望；每个测试一个行为，这样失败指向一件事；命名陈述了保证。当测试以这种方式编写时，它们同时充当了工作示例，这就是为什么测试驱动开发——先编写测试以固定预期行为，然后使其通过——在一次过程中既产生了设计工具，也产生了鲜活的文档。从测试驱动设计也倾向于将代码推向更小、可注入的单元，这正是使生产代码更易于维护的同一特性。
