# 霰弹式修改（Shotgun Surgery）

## 什么是霰弹式修改

霰弹式修改是这样一种坏味：一个概念上的变更迫使你在许多不同的文件或类中进行大量小修改。你决定添加一种新的支付方式、改变日期的格式化方式、或者重命名网络协议中的某个字段——然后突然间你需要改动十几个模块，每个都只有一小部分变更。漏掉一个，系统就微妙地损坏了。变更逻辑上是单一的，但在物理上是分散的。

痛苦在变更时感受到，这使它成为一种*变更阻止者*：代码在任何一个地方读起来可能都没问题，但演化的成本很高，并且不完整变更的风险始终存在。一个决策被涂抹到的地方越多，未来某个编辑更新了大部分而忘记其余的可能性就越大。

## 信号

当一次常规变更产生跨越看似无关文件的冗长、重复的 diff，或者当你对一次变更的心理检查清单是"记得还要更新 X、Y 和 Z"时，你就会注意到它。代码审查评论如"你也改了验证器/序列化器/文档吗？"是社交症状。一个实用的检测方法是进行变更并观察编辑如何分散：如果单个职责需要在七个地方编辑，那七个地方持有同一个概念的片段。

这通常是基本类型偏执和重复知识的下游效应：当一个领域概念（状态、比率、格式）没有单一归属时，每个使用它的地方都必须编码相同的知识，因此对该知识的每次变更都会波及每个地方。参见 [duplicated-code.md](./duplicated-code.md) 和 [primitive-obsession.md](./primitive-obsession.md)。

## 它在告诉你什么

霰弹式修改是**缺失抽象**或**糟糕内聚**的症状：本应在一处的东西散落各处。补救措施是将分散的片段收集到一个单一的模块、类或函数中，由其拥有该决策——这是该坏味的逆操作。常见的操作：

- [move-function.md](./move-function.md) 和移动字段将分散的行为和数据拉拢到一个所有者中。
- 引入缺失的类型（值对象、枚举、配置对象），使概念有一个归属，之后分散的用法降级为对该单一类型的引用。
- 用单一真相源替换重复的知识——一张表、一个分发映射、一个模式——这样变更就是一次编辑。

目标是未来此类变更只触及一个地方。你不会总是恰好得到一个，但从十二个地方减少到两个已经是风险的大幅降低。

## 与发散式修改的关系

霰弹式修改和 [divergent-change.md](./divergent-change.md) 是互逆的坏味，值得同时记住两者，因为修复方向相反。

- **霰弹式修改：** *一个变更 → 多个模块。* 一种变更散落在太多地方。修复方法是**汇聚**——将片段拉到一起，使变更加以本地化。
- **发散式修改：** *一个模块 → 多种变更。* 一个模块因多种无关原因被修改。修复方法是**拆分**——分离职责，使每个模块因一个原因而变化。

两者都是关于将模块边界与代码实际变化的轴线对齐。霰弹式修改说一个边界缺失了（概念没有归属）；发散式修改说一个边界在错误的位置（模块容纳了太多概念。统一的目标是单一职责原则，理解为"一个变更原因"——参见 [solid](references/design-principles/solid.md)。修复一个可能揭示另一个，因此在每次移动后重新评估，而不是在单次通过中过度汇聚或过度拆分。
