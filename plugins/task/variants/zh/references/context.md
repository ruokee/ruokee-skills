## 从无绑定开始

将每个新会话视为无绑定。不要从上一会话、项目、分支、工作树、近期 Task 或宿主状态恢复当前 Task。

在每次 Task 工具调用时传入当前绝对工作区 `cwd`。已安装的 MCP 服务器从其插件目录启动，因此进程 cwd 不是项目上下文。切换工作树或目录后重新计算 cwd。

用户的显式引用或唯一解析的请求仅建立当前对话的上下文。不要持久化该绑定。

## 从显式引用解析

当用户给出完整 ID、确切名称、路径、目录名或分支时：

1. 对唯一的 ID/路径/名称/目录引用直接使用 `task_read`，或在需要检查候选项时使用 `task_find`。
2. 对分支证据使用专门的精确 `branch` 过滤器；不要将其隐藏在模糊查询中。
3. 如果有多个 Task 匹配，展示简洁的候选项，包括名称、状态、路径、分支和完整 ID（按需）。
4. 让用户选择。永远不要按相似度或最近使用时间选择。
5. 在加载详细历史前先读取所选 Task 的 `summary`。

唯一操作接受精确引用；不支持 UUID 前缀。`task_find` 查询可能返回精确、前缀或子串匹配的证据，但搜索结果不是猜测相近 Task 的许可。

默认排除已归档的 Task。在用户引用某个已归档 Task 或搜索为历史性时，有意包含已归档 Task。

## 从工作现场证据恢复

当用户仅说"继续"或其他方式不提供持久引用时，按以下顺序使用证据：

1. 当前绝对项目/工作树路径；
2. 当前 Git 分支；
3. 未提交的变更和相关文件；
4. 来自 `task_find` 的已有 Task 候选；
5. 与可见工作现场匹配的名称和材料链接。

使用证据查找已有 Task，而非创建新 Task。如果强证据唯一支持一个 Task，则读取它并简要说明已解析的内容。如果证据是启发式的或仍有多个 Task 可能，在将其视为当前 Task 前先询问。

不要仅仅因为没有找到现有候选就创建 Task。顶层创建仍然遵循项目 `creation_policy` 和用户的真实意图。

已关闭的父级通常阻止普通的后代发现。完整的后代 ID 或显式路径可以穿透该边界。在修复或审查强制关闭的树时有目的地使用此方式；不要让已关闭的后代成为常规活动搜索的一部分。

## 选择读取视图

使用最小充分的视图：

- `metadata`：检查状态、受管字段、父/子级、依赖、相关 Task、警告和路径，不加载 body 或 WAL。
- `summary`：从元数据、当前 body 和最近 WAL 记录的第一段恢复正常工作。
- `detailed`：当摘要省略了必要的推理或移交细节时，检查完整的选定 WAL 记录。

WAL 字符和记录预算是独立的。截断响应意味着更旧或更长的活动仍在磁盘上。仅在当前决策需要时增加预算或读取特定 WAL 文件。

对损坏候选的显式路径读取可能返回带有 `managed_valid: false`、验证错误和可读内容的 body。将其视为诊断上下文，而非可写 Task。

## 读取普通材料

Core 返回权威的绝对 `task_dir`，但不盘点普通文件。读取 `TASK.md` 后：

1. 跟踪 body 或最近 WAL 中的重要材料入口点。
2. 仅检查与当前请求相关的文件。
3. 对代码事实使用项目/Git 状态，而非期望 Task 包含代码快照。
4. 将重要的计划、研究、检查点和输出放在普通文件中，而非无限扩展 `TASK.md` 或 WAL。

Task 材料归用户所有。保留现有组织和链接。不要强加固定的移交文件名或模式。

## 在同一对话中继续

Task 被唯一解析后，同一对话中语义连续的轮次可继续使用它而无需重复 ID。这是普通的语言上下文，而非持久的活动 Task。

在以下情况重新解析：

- 用户提到了另一个 Task；
- 主题发生实质性变化；
- "这个 Task" 可能指多个对象；
- cwd/工作树发生变化；
- 压缩移除了维持唯一引用所需的证据。

不要仅仅为了保持会话便利而将当前 Task 状态写入项目配置、Task frontmatter、WAL 或宿主元数据。

## 应用分配边界

运行时分配是上下文相关的，不由 Core 存储或执行：

- 分配给 Task X 的 Agent 可写入 X 及其后代。
- 父级和同级 Task 数据为只读。
- 分配给顶层 Task 的 Agent 可写入整个 Task 树。
- 此边界适用于 Task 数据，而非普通项目工作文件。
- 分配给子任务的子代理不写入父级 WAL；父 Agent 记录观察到的结果。

给被委派的 Agent 提供唯一的 Task 引用和最小必要的材料入口点。不要认为创建子代理就会创建子任务，反之亦然。

整合委派结果时，检查实际的项目和 Task 变更。在适当的 Task 级别记录结果，而非复制工具对话内容。

## 准备移交和压缩上下文

在结束未完成工作的会话前，追加一条简洁的 WAL 记录，涵盖：

- 已完成的工作；
- 当前阻塞项或未解决的决策；
- 下一步有用的操作。

当这不适合一条有用的 WAL 记录时，在 `task_dir` 下创建普通的移交/检查点文件，并从 `TASK.md` 或 WAL 链接它。包括未来 Agent 必须检查的当前代码/工作树引用，但不要复制完整的 diff 或源文件作为备份。

如果宿主暴露即将进行的上下文压缩，按当前可用信息尽力完成检查点。无法发出压缩信号的宿主不违反 Task 契约；正常的恢复必须仍能从当前事实、材料、WAL 和项目工作现场工作。

在记录复杂移交或修正历史活动前，阅读 [WAL](wal.md)。

## 处理困难恢复

如果 Task 状态与工作现场不一致：

1. 将文件、Git 和当前的 Core 读取视为事实。
2. 区分过时的文字与结构元数据和实际代码状态。
3. 不要重写 WAL 历史使其显得一致。
4. 更新 `TASK.md` body 为当前事实，然后追加一条 WAL 修正，解释观察到的变更。
5. 当多个 Task、分支或预期的继续方向仍然可能时，询问用户。

如果警告指示无效元数据、重复身份、丢失根目录或文件系统竞态，停止正常恢复并阅读[诊断与修复](diagnostics-and-repair.md)。
