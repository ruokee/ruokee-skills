## 定义持久对象

Task 是用户已决定推进的临时性工作，且能从跨会话、跨 Agent 或上下文压缩的持久状态中受益。它属于一个项目，且无需特定模型或宿主即可理解。

不要将 Task 用于：

- 想法、待办捕获或积压；
- 一次性回答或快速编辑；
- 宿主的临时待办/计划；
- 隐式当前 Task；
- 无项目的全局项；
- 代码快照、同步或外部问题跟踪。

Task 文件属于用户。让确定性 Core 拥有受管元数据、关系、生命周期、发现、锁定和 WAL 追加。使用普通文件存放灵活的 Task 自有材料。

## 创建顶层 Task 和子任务

顶层创建遵循有效的 `creation_policy`：

- 在 `strict` 下，仅当用户在当前对话中明确要求或确认顶层 Task 时才创建。仅对此事实传递 `user_confirmed: true`。
- 在 `permissive` 下，为明显持续性、多阶段或跨会话的工作创建，并在同一轮次告知用户。

不要为快速工作、未决定的想法、普通待办事项或尚未开始的工作自动创建。如果 Core 报告项目未初始化，停止并阅读[项目设置](project-setup.md)。

在已分配的 Task 树内创建子任务不需要新的顶层确认。对应一起创建的同级子任务使用单次批量创建。父级必须为 open 或 paused；批量包含 1..50 个同级，在正常验证失败时是全有或全无的。

不要认为子代理需要子任务，或每个子任务都需要子代理。当工作本身具有持久的独立状态或材料时使用子任务。

对于普通创建，省略 `created_at` 以使 Core 一致地使用实际创建时间。仅为具有可靠时区感知原始时刻的历史迁移设置它；在此之前阅读迁移指南。

## 使用稳定身份和人类引用

UUIDv7 是永久身份。名称、目录路径、目录名和分支是可变的入口点。

唯一操作接受：

- 完整 UUIDv7；
- 绝对 Task 路径或 `TASK.md` 路径；
- 相对于项目根目录的清晰路径；
- 确切名称；
- 确切目录名。

不要使用 UUID 前缀。当名称或目录引用有歧义时，展示候选项并询问用户。永远不要按相似度、最近使用时间或假定的当前 Task 解析。

结构化关系存储完整 ID，而非名称或路径。使用 Core 返回的绝对 `task_dir` 进行宿主文件操作。

## 维护名称和路径

Core 使用 NFC 规范化 Task 名称并修剪外部空白。名称必须非空、不含 NUL 或换行、适合 40 显示列内且适合 96 UTF-8 字节内。

Core 派生目录 slug。调用者从不提供 slug 或目标目录。目录冲突会失败而非接收随机后缀；选择更清晰的名称。

顶层 Task 通常位于：

```text
<task-root>/YYYY-MM/DD/NN--<slug>/
```

子任务通常位于：

```text
<parent>/subtasks/NN--<slug>/
```

当用户手动移动 Task 时，UUID 是权威的。Core 从最近的合法祖先重新派生父级，并可能发出 `name_slug_mismatch` 或 `noncanonical_task_path`；这些警告通常不阻塞读取/更新/日志。重复 UUID、跨项目路径、无效受管字段和目标冲突会阻塞写入。

不要使用 `task_update` 或仅编辑 frontmatter 来更改 `name`。在更改名称前阅读[重命名](rename.md)。

## 维护 `TASK.md`

`TASK.md` 包含受管 YAML frontmatter 和用户/Agent 维护的 body。

受管字段：

| 字段 | 含义 |
| --- | --- |
| `schema_version` | 当前 Task 数据模式 |
| `id` | 不可变 UUIDv7 |
| `name` | 显示名称 |
| `status` | `open`、`paused` 或 `closed` |
| `archived` | 布尔值；仅在已关闭时为 true；省略表示 false |
| `created_at` | 时区感知的 RFC 3339 时间戳 |
| `branch` | 可选分支名；存在性不保证 |
| `depends_on` | 完整同项目 Task ID；省略表示空 |
| `related_to` | 完整同项目 Task ID；省略表示空 |
| `last_transition_reason` | 最新生命周期原因 |
| `extra` | 可选的浅层扩展映射 |

不要将父级、路径、项目、Task 类型、分配人、会话、活动状态、修订、etag 或派生时间戳添加为新的一等语义。

当存在等效操作时，对受管字段使用 Core 操作。Core 保留未知顶层字段，并尝试保留 YAML 顺序、注释、引号、锚点和确切的 body 字节，同时更改受管字段。

新 Task 使用稀疏 frontmatter：Core 在 `archived` 为 false 时省略它，在空的 `depends_on` 或 `related_to` 时省略它们。归档写入 `archived: true`，取消归档移除该节点；添加第一个关系创建该字段，移除最后一个关系删除它。旧文件中显式的 `archived: false` 或 `[]` 仍然有效，无关更新不会顺便剥离这些默认值。读取视图仍将 `archived` 报告为布尔值、关系报告为数组，因此调用者永远看不到磁盘上的差异。

如果受管字段或 frontmatter 无效，显式路径读取可返回 `managed_valid: false`、验证错误和可读内容。该候选不能加入关系图、更新或日志，直到修复。不要让 Core 覆盖重复键、不安全的自定义标签或不可解析的 YAML；阅读[诊断与修复](diagnostics-and-repair.md)。

将 body 写为当前事实。将"我们用了 A 后来改为 B"替换为当前选择 B；将变更历史记录在 WAL 中。保持 body 足够简洁以解释当前目标、范围、状态和重要材料的入口点。

## 组织普通材料

将计划、研究、详细决策、检查点、移交、结果和其他持久的 Task 自有产物作为普通文件存储在 `task_dir` 下。从 body 或 WAL 链接重要的入口点。

Core 有意不盘点这些材料。使用宿主文件工具读取和编辑它们。不要将完整 diff 或源代码副本放入 Task 以备份代码库；Git 和项目文件仍然是代码事实。

Task 材料符号链接可能存在，但发现从不跟随目录符号链接，且 `TASK.md` 必须是常规文件。将不可移植或不明确的链接视为诊断问题。

## 建模关系

Task 有三种关系类型：

| 关系 | 表示 | 效果 |
| --- | --- | --- |
| 父/子 | 最近的合法祖先目录 | 仅拓扑；无自动依赖或状态级联 |
| 依赖 | `depends_on` 完整 ID | 有向；阻塞正常关闭直到解析并关闭 |
| 相关 | `related_to` 完整 ID | 非生命周期关联 |

关系保持在一个项目内。用文字表达跨项目的弱上下文，而非使用受管关系字段。

通过 `task_update` 使用添加/移除增量；不要从过期读取中替换关系数组。Core 拒绝自引用、未知新目标和依赖循环。`related_to` 存储为单向传出列表；读取可以派生 `related_from` 而无需写入反向边。

手动引入的缺失依赖会发出警告并阻塞正常关闭。缺失的相关目标会发出警告但不阻塞其他更新。不要仅仅为了消除警告而移除未解析的 ID，而不理解预期的关系。

## 应用生命周期转换

状态机为：

```text
open ⇄ paused
  │       │
  └───┬───┘
      ▼
    closed ──reopen──> open
```

为每次转换提供非空原因。Core 更新 `last_transition_reason` 并追加一条机械 WAL 记录。Reopen 始终返回 open。

在正常关闭前，Core 递归验证：

- 每个后代已关闭；
- 每个依赖已解析且已关闭；
- 不存在阻止满足条件的依赖循环。

已暂停或缺失的依赖不算完成。相关 Task 不影响关闭。

仅当用户明确要求绕过检查时使用 force close。Force 仅关闭目标；它不关闭后代或依赖。Core 记录被跳过的检查。已关闭的父级会隐藏后代，阻止普通发现，但完整 ID、显式路径或检查仍可到达它们。

当关闭被阻塞时，展示后代/依赖/循环证据。不要将 force 静默转换为清理机制。

## 处理归档

归档是与三个状态正交的布尔值，永远不会移动 Task 目录。

- 仅归档已关闭的 Task 并提供原因。
- 不要重新打开已归档的 Task。
- 仅在用户当前明确同意且有原因时取消归档。
- 取消归档后，Task 保持已关闭。如果工作应继续，重新打开它。

当保留旧完成边界更有用时，考虑创建新的相关 Task 而非重新打开。

## 通过正确接口更新

对以下操作使用 `task_update`：

- 分支设置/清除；
- 依赖和相关添加/移除增量；
- 浅层 `extra` 设置/移除；
- 最多一次生命周期操作。

使用宿主文件工具编辑 `TASK.md` body 和普通材料。使用重命名 CLI 更改名称。使用 `task_log` 记录持久工作活动。当更新返回 `changed: false` 时，不要为未发生的变更添加机械 WAL 记录。

如果已提交的更新报告 WAL 失败，不要重复更新。阅读 [WAL](wal.md) 并仅修复缺失的活动记录。
