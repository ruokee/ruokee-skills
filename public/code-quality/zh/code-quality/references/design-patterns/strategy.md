# 策略模式（Strategy）

## 意图（Intent）

定义一族可互换的算法，置于稳定接口之后，使得算法可以在配置时或运行时被选择和切换，而无需改变使用它的代码。

## 解决的问题

当单个操作有多个变体算法（排序、评分、定价、路由、重试、压缩的方式）时，将它们嵌入到不断增长的 `if/elif` 块中，会将调用方耦合到每个变体，并使添加新变体成为对共享代码的编辑。策略隔离每个算法，使调用方只依赖于"某种对项目进行评分的东西"，而不依赖于具体的评分规则。

## 结构和参与者

经典形式有一个**上下文（context）**，它持有一个对**策略（strategy）**接口的引用，并有**具体策略（concrete strategies）**实现该接口。上下文将其工作中可变的部分委托给注入的策略。上下文拥有稳定的工作流；策略拥有变化的部分。

## Python 形式

Python 函数是一等公民，因此策略模式通常*只是一个函数参数*。通常不需要策略类：

```python
def rank_items(items: list[Item], score: Callable[[Item], float]) -> list[Item]:
    return sorted(items, key=score, reverse=True)

def priority_score(item: Item) -> float:
    return item.priority

rank_by_priority = partial(rank_items, score=priority_score)
```

按重量级大致排序的形式：

- **普通函数或 `lambda`** 作为参数传递——最轻量的策略。
- **闭包** 捕获少量稳定配置，或 `functools.partial` 将参数预绑定到更窄的可调用对象。
- **`Protocol` 或可调用对象**：当策略需要状态、多个相关方法或一个能说明意图的名称时。
- **`functools.singledispatch`**：当策略由输入的*类型*而非配置值选择时。
- **分发映射 / `match`**：当选择依赖配置值或多个条件时。

## 何时使用

- 当前存在多个真实算法，且调用方不应知道运行的是哪一个。
- 算法必须在运行时可选（用户选择、配置、A/B 测试）或在测试中可替换。
- 变化沿着一个明确的轴（"如何评分"），具有小而稳定的接口。

## 何时不使用

- 只有一个算法，或者单个 `if` 就能覆盖两种情况。预防性的"策略化"引入接口却没有任何好处。
- 策略接口过于宽泛，调用方为了改变一个决策就必须组装一个复杂对象——抽象切错了方向。
- "策略"实际上在*需要什么数据*上不同，而不仅仅是*如何计算*——它们可能无法共享一个连贯的接口。

## 失败模式

- 一个 `Strategy` 类只有一个方法和一个实现，而函数能以更少的仪式表达相同内容。
- 接口过宽，迫使每个具体策略实现只有其中一个才需要的方法。
- 策略选择散落在代码库各处，而非在一个地方（工厂或分发映射）集中解析，导致添加变体时需要查找每个选择点。

## 与其他模式的关系

[factory.md](factory.md) 通常*选择*使用哪个策略。[command.md](command.md) 在结构上相似（行为作为对象），但其意图是捕获供稍后执行的请求，而非变换算法。[state.md](state.md) 模式看起来像策略，但其变体会在事件响应中*改变自身*，而非由调用方选择。`functools.singledispatch` 和模式匹配是 Python 中最常吸收策略模式的机制。
