# 仓库模式（Repository）

仓库模式在持久化机制之上提供一个类似集合的接口，将领域逻辑与存储细节隔离。它在领域模型和数据映射层之间进行中介，提供诸如 `get`、`add`、`remove` 和返回领域对象的查询方法。

其核心价值在于，领域代码通过干净的接口使用领域对象，而无需知道数据来自数据库、文件系统、API、缓存还是内存集合。这使得领域逻辑无需真实基础设施即可测试，并允许在不修改业务规则的情况下更改存储技术。

## 结构

- **仓库接口（Repository interface, Protocol 或 ABC）**：定义类似集合的方法（`get_by_id`、`add`、`remove`、`list_by_criteria`）。
- **具体仓库（Concrete Repository）**：使用特定技术（SQLAlchemy session、原生 SQL、HTTP 客户端、文件系统）实现接口。
- **领域对象（Domain objects）**：由仓库返回的纯领域实体或聚合。
- **应用/服务层（Application/service layer）**：使用仓库接口，而非具体实现。

依赖方向向内流动：领域代码依赖仓库*接口*，具体实现依赖它持久化的领域对象。这是[依赖反转](references/design-principles/dependency-inversion.md)在持久化中的应用。

## 模式适用时

- 领域逻辑足够复杂，将其与持久化隔离可提高清晰性和可测试性。
- 可能有多个存储后端（生产数据库、测试内存存储、从一种 ORM 迁移到另一种）。
- 应用程序采用分层或六边形架构，具有明确的边界。
- 业务规则应无需数据库 fixture 即可测试。
- 团队受益于一致、可发现的数据访问 API。

## 模式不适用时

- 应用程序是带有最少领域逻辑的薄 CRUD 层。在 ORM 之上添加仓库增加了间接性，却没有降低复杂性。
- ORM 已经提供了足够干净的抽象（例如，Django 的 Manager/QuerySet 用于简单应用）。
- 只会使用一种存储技术，且领域逻辑很简单。
- 查询高度动态或分析性。仓库接口成为复杂 SQL 上的泄漏抽象——在此类情况下，专用查询服务或 CQRS 分离更好。
- 项目是脚本或带有单一数据源的小工具。

## 常见实现问题

**接口膨胀。** 仓库增长出数十个查询方法后失去了其抽象价值。保持接口聚焦于[领域](references/design-principles/ddd.md)操作，而非通用查询构建器。

**泄漏的抽象。** 暴露 ORM 特定概念（session、flush、懒加载、查询构建器）的方法破坏了目的。返回完全加载的领域对象。如果调用方需要分页、过滤或排序，将这些设计为仓库参数，而不是泄露 ORM 查询链。

**事务边界。** 仓库通常不拥有事务。事务边界属于应用/服务层或[工作单元](unit-of-work.md)。内部提交的仓库使得跨多个聚合协调写入变得不可能。

**急切加载 vs 懒加载。** 如果 ORM 懒加载相关对象，在仓库边界外访问它们可能失败或触发意外查询。在仓库边界决定加载策略，使返回的对象对其预期用例是完整的。

**测试。** 为领域逻辑的单元测试提供内存实现。集成测试仍需要真实的数据库仓库来验证查询、约束和迁移。

## 与工作单元的关系

仓库通过类似集合的 API 处理单个聚合的持久化。[工作单元](unit-of-work.md)处理跨多个仓库的事务一致性。它们互为补充：仓库提供集合接口，工作单元提供提交/回滚边界。它们共同为应用层提供了对*持久化什么*和*何时持久化*的控制。

## 与外观模式的关系

仓库是一个面向领域的外观（[Facade](facade.md)），位于持久化子系统之上。区别在于意图：外观为任何消费者简化复杂子系统；仓库具体在领域对象和存储之间进行中介。如果你的"仓库"实际上只是简化一个没有领域对象的复杂外部 API，那么外观可能是更合适的名称。
