# 测试坏味道

测试套件是第二套代码库，它会以自己特有的方式退化。下面的坏味道来自 Gerard Meszaros 的 *xUnit Test Patterns*、Google 的测试指南和 Kent Beck 的测试期望属性集。和[代码坏味道](references/refactoring/code-smells.md)一样，每一条都是*值得调查的症状*，不是最终判决。一个读起来略有点重复的测试可能完全正确；一个跟内部调用耦合的测试可能是固定关键契约的唯一方式。判断标准始终是这个坏味道是否在消耗套件的价值——捕捉回归、记录行为、在重构中存活——并且每节末尾的误报边界跟检测信号同等重要。

几乎所有坏味道背后的统一原则是 Kent Beck 那句：**测试应该与行为耦合，与结构解耦。** 当测试在保留行为不变的重构上炸了，说明某种结构耦合渗了进来。当测试在行为已坏掉时仍通过，说明某种行为覆盖漏了出去。大多数测试坏味道就是这两种泄漏之一穿上特定外衣的样子。

## 脆弱测试

**它是什么。** 一种测试，生产代码改了但*被测行为*没变，它却炸了。测试跟结构耦合——方法签名、调用顺序、内部数据形状、周围上下文——而不是跟可观察契约耦合。Meszaros 归纳了四种敏感性：接口（重命名或改签名击穿测试）、行为（无关的行为变化波及进来）、数据（共享测试数据的变化破坏无关测试）、上下文（测试依赖全局或环境状态）。

**为什么重要。** 这是日常工作中最贵的测试坏味道，也是"改一个地方就得修一堆测试"这个抱怨的根源。脆弱的套件让每次重构的成本都比应有的高：本应让变更更放心的安全网，反倒成了变更的负担。更糟的是，当大规模机械修测试变成例行公事，团队就不再读测试在断言什么，而是按模式机械编辑，真正的回归就藏在噪音里。一个每次重命名都叫狼来了的套件，最终会被无视。

**检测信号。** 保留行为不变的重构迫使编辑多个测试文件。测试断言私有字段、内部方法名或内部调用的*顺序*，而不是返回值或可见效果。重命名内部辅助函数击穿了本不依赖它的测试。相同的字面量在几十个测试里重复，改一个产品行为就得全面扫一遍。你发现自己同时给很多测试做*同样的*修改——这是变更检测器的信号。

**解决方法。** 通过公共接口——真正的调用者用的那个表面——来测，断言可观察的结果：返回值、抛出的异常、持久化的状态、发往真正边界的消息。把重复的字面量抽到命名的 fixture 或构建器里，让契约变化只改一处（参见"测试代码重复"一节）。当测试必须引用某个内部时，问那个内部是否该被提升为真正的、有测试的接口。注意构造函数加了依赖时 setup 变更是正常的——坏味道在于*断言*追踪结构，不是 arrange 步骤跟了新签名。

**示例场景。** 一个服务被重构，一个方法拆成三个私有辅助函数；行为完全一样。四十个测试炸了，因为它们用 mock 框架断言旧方法以特定参数被调用了一次。没有检查服务实际产出了什么，四十个全得重写——而重写也证明不了任何新东西。

**误报边界。** 有些测试*本应*对某个实现选择敏感，因为那个选择本身就是行为——验证请求走了缓存而不是源站，或查询用了索引，是合理的实现断言。Google 允许这类测试，但要求它们很少见且被明确标注。没人主动选择的结构耦合才是坏味道；有意的、文档化的实现测试不是脆弱，是精确。

## 变更检测器测试

**它是什么。** 一种测试，只是把生产代码用第二种形式复述一遍——通常是全 mock 加调用顺序验证——以至于*任何*代码变化都会改变测试，但测试没有独立的正确性判断标准。Google 明确指出了这一点：期望值是实现的镜像的测试是变更检测器，不是正确性检查。

**为什么重要。** 这些测试往往趋向*负*价值。它们通过读代码然后断言代码做了它该做的事写成，没有独立的正确性判据——所以代码错了的时候，测试通常会忠实地跟着一起断言错的东西，然后依然保持通过。同时它们承担着完整的维护成本——每次重构都会炸掉它们——还挤占了套件空间，让真正的测试更难找。这是脆弱测试的极端：最大结构耦合，最小行为覆盖。

**检测信号。** 一个测试几乎全是 `mock(...)` setup 加 `verify(...)` / `assert_called_with(...)` 断言，没有对真实结果的断言。测试读起来像方法体的转录。删掉它不会损失任何输入-输出关系的覆盖。生产代码变了的时候，测试的 diff 和生产 diff *形状相同*。

**解决方法。** 删除或重写。问代码负责什么可观察行为，改测那个；协作者便宜且确定的就用真实对象。如果边界上的一次交互才是在乎的（扣款恰好扣了一次），在那个真实边界上保留一个聚焦的交互测试，扔掉内部镜像的断言。这是套件级别上的死重测试移除——但要挣得这个说法：删之前确认这个测试不固定任何独特行为、不杀死任何独特突变、也不提供其他测试没有的诊断价值。变更检测器测试通常达不到这些标准，但"通常"不是"总是"，逐个核实而不是按类型删除。

**示例场景。** 一个方法计算总额并调了 `logger.debug`、`repo.save`、`bus.publish`。测试 mock 了三个，断言按那个顺序、用那些参数调了。从不检查算出来的总额。总额计算里的 bug 照样以通过状态发布了；一行无害的日志重排让它失败。

**误报边界。** 真正系统边界上的交互测试不是变更检测器：验证邮件发出去了、或扣款确实发生了一次，是用户关心的可观察行为，虽然表达为交互。坏味道是镜像没有任何外部契约提到的*内部*交互，而不是验证代码真正要产生的副作用。

## 晦涩测试

**它是什么。** 一种测试，意图不能一眼看懂。Meszaros 把几个成因归在此类：Eager Test（一个测试验证多个行为，没有单一清晰意图）、Mystery Guest（测试依赖它没展示的外部数据或状态，读者看不出为什么通过）、General Fixture（过大的共享 setup 构建了整个世界，测试只用了一小片）、Irrelevant Information（噪音淹没了少数几个关键值）、Indirect Testing（测试通过多层其他对象来驱动目标单元）。

**为什么重要。** 测试是不能悄悄过期的文档，但前提是人读得懂。晦涩的测试解释不了它保护什么保证，所以炸了的时候没人说得清是生产代码还是测试本身错了。这就是 DAMP 应该优于 DRY 的地方：测试代码应该有描述性、有意义，哪怕重复一点，因为读者必须*用眼*验测试——没有给测试写的测试。过度压缩、过度抽象的测试代码，用测试最需要的属性（显而易见的正确）换来了它最不需要的（最少行数）。

**检测信号。** 一个测试里有好几个 act-assert 周期，分别测不同行为。决定结果的值来自文件、共享数据库行或远处定义的 fixture，测试体里看不出为什么通过。`setUp`/fixture 构造的东西远多于测试触及的。测试被包在循环和条件里，实际案例难以提取。读测试比读被测试代码还费时间。

**解决方法。** 每个测试一个行为，以为其保护的保证命名。把关键数据内联，读者一眼同时看到原因和预期结果——DAMP，不是 DRY。用小的、按用途命名的 fixture 或工厂辅助函数替换通用 fixture，让每个测试只拿自己需要的（Python 参见 `python-engineering` 的测试规范和 pytest 实践里的 fixture 设计与工厂即 fixture）。把说明性细节推进命名良好的辅助函数，去掉噪音但不隐藏驱动断言的值。

**示例场景。** `test_user_flow` 创建一个用户、更新邮箱、加两个订单、取消一个、然后断言最终余额——十二行 setup、四个不相关的断言。失败时只说 `test_user_flow` 炸了，读者得重建四个行为里哪个退化了。

**误报边界。** 真正的多步骤行为——端到端工作流测试——理所当然有更多 setup 和不止一个断言，因为工作流*就是*那个"单元"。少量可读的重复不是晦涩；DAMP 明确偏好它。坏味道是误把无关行为捆在一起或隐藏因果数据造成的意外晦涩，不是真实场景自身不可避免的规模。

## 断言轮盘赌

**它是什么。** 一个测试里一堆裸断言，每个没有描述，所以其中一条失败时你不知道是哪条——玩轮盘赌找凶手。Kent Beck 的期望属性里*具体*的正面形式就是：失败的测试应该精确指向什么坏了。

**为什么重要。** 诊断时间是真实成本，每次测试失败时都要付，而且经常在截止日期压力下。第 N 行"期望 true，得到 false"，周围十个看起来一模一样的断言——读者得插桩或二分才能搞清楚到底什么坏了。它也暗示晦涩测试：一堆没描述的断言通常意味着几个行为塞进了一个测试。

**检测信号。** 长串断言，没有消息，没有结构。失败输出只给行号，不给行为。几个各自都可以是有意义的独立检查的断言，无分隔堆在一个测试里。

**解决方法。** 每个测试一个逻辑断言（关于一个行为），测试名*就是*失败描述。多个断言确实该在一起时，给每条加消息，或用能识别主体的断言风格。断言是对不同输入的相同检查时用参数化，让每个案例单独报告（Python 参见 `python-engineering` 的 pytest 实践）。

**误报边界。** 对*一个*结果对象断言几个属性不是轮盘赌——检查解析记录的名字、年龄、id 是一个行为（"解析出了正确记录"），读起来清晰。坏味道是跨*不同*行为的无描述断言，不是单一结果上的几个相关检查。

## 不稳定 / 非确定性测试

**它是什么。** 一种测试，没有代码变化但有时过有时不过。Meszaros 的 Erratic Test 把原因归类为：交互测试（一个测试的遗留状态影响另一个）、测试运行战争（共享外部资源被并行运行争抢）、不可重复测试（过了一次再跑就挂）、资源泄漏、真正的非确定性逻辑（时钟、随机、网络、顺序、并发）。Fowler 关于根除非确定的文章把它视为套件的生存威胁。

**为什么重要。** 不稳定性摧毁*整个*套件——不只是那个不稳定测试——的信任。一旦失败既可能是真有问题也可能是运气不好，每次失败都成了判断题，人们开始重跑到通过为止，真正的回归就被当"可能只是不稳定"放过。一个你信不过的套件比没有好不到哪去。这个坏味道是 Kent Beck 的*确定性*和*隔离*期望属性、以及 Google"测试必须能任意顺序运行"这条规则的敌人。

**检测信号。** CI 上挂但本地过，或只在并行下挂，或只以某种顺序运行才挂。读写共享状态的测试——真实时钟、共享数据库、固定路径文件、模块级全局变量——且没做隔离。立即重跑就挂（它留下了残留状态）。用 sleep"等"异步工作。

**解决方法。** 隔离：每个测试拿干净的新状态、自己清理干净，在其声明支持的执行方式下，不因顺序或并行调度改变结果（刻意测试并发的测试仍要控制并发——用固定调度或播种的交错，而不是裸奔竞争）。用测试替身或注入控制替换不受控的边界——固定时钟、设种子的 RNG、内存假实现代替网络。用自动恢复状态的框架工具（临时目录、teardown 时还原的环境补丁），而不是测试失败就跳过的自制清理。绝不让已知道不稳定的测试留在主套件里当警示灯——隔离或修掉，因为一个被容忍的不稳定会训练所有人无视失败。

**示例场景。** `test_report_uses_todays_date` 调真实的 `date.today()` 并断言格式化字符串。几个月都过，然后在月份边界和 CI 的时区之间挂了。行为从来没出过错；测试把自己耦合到了墙上的时钟，而不是注入固定日期。

**误报边界。** 合法练习并发或随机性的测试，只要能控制它们——设种子的生成器或确定性调度器让结果可重复——就不是不稳定。触碰真实外部系统的集成测试本来就慢，正确做法是隔离到单独的套件里；只要它不是坐在快速套件里假装确定，就不是不稳定单元的坏味道。

## 测试代码重复

**它是什么。** 相同的 setup、断言或数据在多个测试间复制，以至于一个契约变化需要编辑每一份副本。这是 DRY 坏味道在测试上的表现——但它跟 DAMP 之间存在张力，所以解决方法比生产代码更微妙。

**为什么重要。** 结构型重复——那种一个契约变化迫使几十个测试同步编辑的重复——是维护负担，也是脆弱测试行为的来源。但修复方式*不是*像对生产代码那样猛搞 DRY，因为测试必须保持肉眼可读（参见晦涩测试）。Google 的指南明确：测试应该 DAMP，容忍可读的重复，只把提取留给*噪音*——那些掩盖意图的样板代码——而不是让每个案例可读的有意义的值。

**检测信号。** 相同的多行 setup 粘贴到多个测试里，改变构造对象的形状会触及所有测试。同一个魔数到处重复。多个文件里复制粘贴的 fixture 定义，微妙地或完全一样——"不同地方重复 fixture"的具体问题。一个能去掉纯模板但从未被提取的辅助函数。

**解决方法。** 提取*噪音*，把*意义*留在内联。把重复的构造收拢到最近公共作用域上的共享初始化位置（共享 fixture、构建器、初始化模块——用测试框架提供的任何机制），只有一个定义而不是五个近似副本——但每个测试仍展示自己具体的输入和预期输出，保持可读。每个测试各不相同的对象用工厂辅助函数或构建器。那条线：消除会迫使*同步结构编辑*的重复，保留读者*理解案例*所需的重复。

**示例场景。** 十二个测试各以同样的八行构建 `Config` 对象开头，只在一个字段上不同。新增了一个必填字段；十二个全部以同样方式炸。一个构建默认配置的 fixture，每个测试只覆盖自己关心的字段，本可以在一个地方吸收变化，同时保持每个测试的意图可见。

**误报边界。** 可读的、有意的重复不是这种坏味道——DAMP 说让测试自解释的一点重复，胜过迫使读者跳到别处的抽象。如果把 setup 提取到共享辅助函数会隐藏解释测试为何通过的值，那就不要提。坏味道是把许多测试耦合到同一个结构的重复，而不是让每个测试局部可理解的重复。

## 测试错误的东西

**它是什么。** 把精力花在测试那些没有独立行为值得保护的代码上——琐碎 getter、框架胶水、配置值，或通过人为直接测试而非通过使用它的行为来练习的通用工具函数。Meszaros 的 Indirect Testing 和更通用的"测试行为，而不是附带代码"原则都有关。

**为什么重要。** 每个测试既是资产也是负债：它要被阅读、维护、更新。固定一个没有逻辑的值（常量、配置默认值）或没有领域意义的工具函数的测试，增加了维护成本和覆盖率百分比带来的虚假安全感，实际上什么也没保护——谁也不会在那上面犯错。它还反转了覆盖信号：套件看起来全面，因为容易的、无行为的代码大量覆盖了，实际的决策逻辑可能很薄。覆盖率是找*未被测试*代码的工具，决不是要最大化的目标；为了涨数字测琐碎代码正是 Fowler 直接警告的反模式。

**检测信号。** 断言常量等于其字面量的测试，或断言配置键有默认值的测试——值变了就得同步改测试，测试从未表达任何意图行为。测只返回字段的 getter。用合成输入隔离测通用 `utils` 函数，而它唯一的真实调用者已经覆盖了它。"把覆盖率拉上去"的推进，为最容易的代码而不是风险最高的代码加测试。

**解决方法。** 在行为被决定的层级测试行为，让琐碎代码被使用它的行为附带覆盖。删掉只复述常量或配置默认值的测试；配置值如果是承重的，测试*它驱动的行为*，而不是值本身。对具有真实边界案例的可复用工具函数（日期解析器、棘手算法），直接的行为测试是合理的——界线是函数有没有*可能出错、可枚举的行为*，而不是它是不是在一个叫 `utils` 的文件里。

**示例场景。** 一个测试断言 `DEFAULT_TIMEOUT == 30`。有人把超时改成 45；测试挂掉，然后"修好了"——30 改成 45——现在它花两次编辑保护了空无一物：它从未检查 30 的超时产生任何特定行为。真正值得测的（请求在配置的超时后中止）从来没人写过。

**误报边界。** 有真实、易出错行为的工具函数，不管在哪都值得直接测——坏味道不是"永远不测 utils"，是"不要为没有行为值得保护的代码造测试"。同样，编码了真正契约的配置值（不能悄悄改的监管上限）可以配一个守卫测试，但应该以保护契约为框架，而不是镜像字面量。判断标准始终是测试能否保护有人可能搞错的行为，而不是代码属于哪一类。
