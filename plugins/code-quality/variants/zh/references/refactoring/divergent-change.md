# 发散式修改（Divergent Change）

## 什么是发散式修改

发散式修改是这样一种坏味：一个单一的模块因许多无关原因被反复修改。每次数据库模式变化，你都要编辑这个模块；每次报告格式变化，你又要编辑它；每次业务规则变动，你又回到同一个文件触碰不同的部分。这个模块成了几个独立关注点恰好共存的交汇点。一个文件，许多无关的打开理由。

它是一种*变更阻止者*，因为混合的关注点使每次变更都比应有的更冒险：编辑报告格式化代码意味着要绕过持久化代码和验证代码，对一个关注点的修改可能意外干扰另一个。该模块也抵抗理解，因为要了解"这个模块是关于什么的"，你必须在脑中同时持有多个不相关的主题。

## 信号

检测问题是：*这个模块因多少种不同的原因而被编辑？* 如果你能查看提交历史（或者只是想象未来请求的种类）并看到这一个文件因为"新税务规则"、"新导出格式"、"新存储后端"和"新验证策略"而变更，那就是四个原因、四个关注点、一个模块。一个文件内部的信号是，一个类或模块的各部分之间没有共同的数据或逻辑——方法聚集成组，彼此不互相调用，也不触碰相同的字段。

与 [shotgun-surgery.md](./shotgun-surgery.md) 的干净对比能使其更清晰：在那里，一个变更触及多个模块；在这里，多个变更触及一个模块。它们是镜像。

## 它在告诉你什么

发散式修改是**拆分职责**的信号。该模块在做多项工作，每项工作应位于自己的模块中，以便能独立和隔离地变更。重构通常是：

- [extract-function.md](./extract-function.md) 先隔离内聚的组，然后提取类/提取模块，给每个关注点自己的归属。
- [move-function.md](./move-function.md) 将行为迁移到拥有其关注点的模块。
- 分离被缠结的层——例如，将一个混合了业务规则、持久化和格式化的模块拆分为三个，这样格式化变更只触及格式化模块。

拆分后，每个生成的模块只有一个变更轴：你只在持久化变化时编辑持久化模块，只在规则变化时编辑规则模块。这是内聚的实际含义。

## 与 SRP 和霰弹式修改的关系

发散式修改本质上是**单一职责原则**作为坏味的表达。Uncle Martin 对 SRP 的阐述——"一个模块应该只有一个变更原因"——正是发散式修改所违反的：该模块有许多变更原因。因此疗法是 SRP 疗法：找到驱动变更的不同角色或力量，给每个自己的模块。参见 [solid](references/design-principles/solid.md)。

它也是**霰弹式修改的逆反**，两者必须相互平衡：

- **发散式修改** → 一个模块，多种变更原因 → **拆分**它。
- **霰弹式修改** → 一个变更原因，多个模块 → **汇聚**它们。

向任一方向过度修正都会产生另一个。拆分得太激进，单一变更就开始需要跨所有新片段的编辑（霰弹式修改）。汇聚得太激进，合并后的模块就开始因多种原因变更（发散式修改）。两个坏味指向的同一个目标是：模块边界应跟随变化的真实轴线，使一种变更映射到一个编辑地点。有疑问时，识别这段代码历史上实际变化的原因，让那些原因定义边界。
