# 分析 — Python 工程 (Analysis — Python Engineering)

协作探索，而非评分。当用户想要*决定该做什么*而非了解*哪里有问题*时使用此模式：头脑风暴方案、设计机制、分析现有代码的行为方式或规划重构。没有需要评分的 diff，也不需要生成发现列表。目标是达成一个共享的、经过充分推理的决策。

## 触发条件 (Trigger)

- "我该怎么组织这个？"、"最好的方式是什么…"、"我有哪些选择？"
- 机制设计：缓存、重试策略、插件系统或数据模型应如何工作。
- 代码分析：解释模块的行为方式、为何如此组织、其接缝在哪里。
- 重构讨论：在承诺实施之前规划变更。

## 前置条件 (Preconditions)

- 只读。分析从不自行修改代码。
- 收集足够的上下文以进行具体推理。不要对你未读过的代码进行理论推演。

## 收集上下文 (Gathering Context)

先阅读，再推理。上下文的多少取决于决策的规模：

- 项目事实：`pyproject.toml` 中的 `requires-python`、依赖和项目形态。版本下限决定了哪些语言特性可以被考虑。
- 讨论中的实际代码，及其调用者和测试，以便选项基于真实结构而非想象的结构。
- 偏好设置（`.agents/preferences/python-engineering.md` 或其目录形式）如果存在 — 但在分析中，偏好是供权衡的输入，而非要执行的规则。
- 相关机制对应的叶子文档（例如 [match-case](references/grammar/match-case.md)、[type-hint](references/spec/type-hint.md)），以便你呈现的权衡是已建立的，而非即兴发挥的。

## 组织分析 (Structuring The Analysis)

以建议开头，然后展示推理过程。有用的分析不是对所有可能性的事无巨细的调查；它是一条清晰的路径加上诚实的权衡。

1. **界定决策。** 重述实际问题和约束条件 — 版本下限、现有架构、依赖限制、代码必须存续的时长。如果在阅读过程中发现了隐藏约束，将其暴露出来。
2. **呈现真正有区别的选项。** 两到三个真正的替代方案，而非五个细枝末节的变体。对于每个方案：它是什么、代价是什么、何时胜出。当选项是沿着相同轴线的并行选择时，使用简短对比表。
3. **推荐。** 说明你会选择哪个选项以及*为什么*，基于第 1 步的约束条件。一个附有用户可以反驳的理由的建议，比一个中立的菜单更有用。
4. **说明权衡。** 每一个真正的选择都要有所舍弃。说明推荐路径的代价，以便用户心中有数地做决定。

## 推荐与询问 (Recommend Versus Ask)

- **推荐并进入细节**，当约束条件明确指向一个选项且错误代价低或可逆时。
- **停下来询问**，当决策依赖于只有用户掌握的信息（部署目标、团队约定、未来计划）时，当选项沿着只有用户能权衡的轴线取舍（速度 vs 简洁性、灵活性 vs 清晰度）时，或者当变更规模大或难以逆转时。
- 不要为了得到一个干净的答案而编造需求。如果正确的回应是"取决于 X"，明确指出 X 并询问。

## 输出 (Output)

散文体和表格，而非发现块。深度与问题匹配 — 小型设计问题用几段话，结构性决策给出选项和建议。以具体的下一步行动结束，并仅在用户同意后才进入实施。

## 停止规则 (Stop Rules)

- 不要修改代码；在用户接受计划前分析是只读的。
- 不要运行会写入文件、创建 `.venv`/缓存或更改锁文件的命令。
- 不要将真正的"取决于情况"压缩成虚假的确定性。
- 不要将偏好设置呈现为通用的 Python 真理 — 它们只是这个项目的选择。
- 不要用你永远不会推荐的选项来填充内容；三个真实选项胜过七个凑数的。
