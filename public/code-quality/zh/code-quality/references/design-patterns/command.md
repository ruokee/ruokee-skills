# 命令模式（Command）

## 意图（Intent）

将请求封装为对象，使得请求可以被传递、存储、排队、记录、重放和撤销。发出请求的调用方与知道如何执行请求的代码解耦。

## 解决的问题

直接调用——`service.send_email(to, template, context)`——立即执行且不留痕迹。你无法将其放入队列、为后续持久化、记录到审计日志、在崩溃后重放或撤销它。当请求需要超越其调用瞬间的生命周期时，它必须变成数据。命令将"做这个"转化为一个你可以持有和检查的值。

## 结构和参与者

- **命令（Command）**：携带执行请求所需的所有信息的对象——操作及其参数。
- **接收者（Receiver）**：知道如何执行工作的对象。
- **调用者（Invoker）**：持有并触发命令（队列、调度器、菜单、按键绑定）。
- **客户（Client）**：创建命令并使用接收者配置它们。

在经典形式中，命令暴露 `execute()` 有时还有 `undo()`。关键在于调用者统一对待所有命令，而不知道每个命令具体做什么。

## Python 惯用实现

当命令是需要排队、序列化或审计的数据时，冻结的数据类是自然的载体：

```python
@dataclass(frozen=True)
class SendEmail:
    to: EmailAddress
    template: str
    context: dict[str, object]
```

一个单独的处理函数（或按键值命令类型索引的 `dispatch` 映射）执行工作，使命令本身成为一个普通的、可序列化的记录。当命令不需要持久化且仅承载行为时，普通函数或 `functools.partial` 本身就*是*一个命令——Python 的一等函数吸收了最简单的场景。

对于撤销，命令必须捕获足够的前置状态来反转自身，或者与 [unit-of-work.md](unit-of-work.md) 或快照状态的备忘录（memento）配对。

## 何时使用

- **排队和调度**：任务放入作业队列、重试或稍后运行。
- **撤销/重做**：每个动作都是一个带有配对逆操作的命令；历史栈驱动撤销。
- **审计追踪**：每个状态变更作为命令记录，用于合规或调试。
- **宏录制/重放**：捕获并重新运行一系列命令。
- **解耦 UI 与动作**：菜单项、按钮和快捷键持有命令，而非直接调用服务。

## 何时不使用（过度设计）

- 简单的、即时的函数调用就足够了，且请求不需要存储、重放或反转。将其包装在命令类中是纯粹的仪式。
- 命令没有稳定的模式（schema），因此实际上无法持久化或重放——那么它无法提供你构建它所付出的成本对应的任何好处。
- 你为了"解耦"只有一个调用者和一个接收者的代码而使用命令对象。

## 失败模式

- **贫血命令加臃肿处理函数**：逻辑迁移到处理函数中，直到命令变成字段的袋子，而分发变成上帝函数。
- **不稳定的序列化**：持久化的命令在重构后无法反序列化，因为模式没有版本化。如果你持久化命令，请将其形态视为契约。
- **撤销漂移**：`undo()` 不能完美反转 `execute()`，会静默地破坏状态。撤销的正确性需要明确的测试。
- **构造中的隐藏副作用**：构建命令不应执行工作；只有调用者应触发它。

## 与其他模式的关系

命令和 [strategy.md](strategy.md) 都将行为包装在对象中，但策略参数化*如何*做某事（可互换的算法），而命令参数化*做什么*以及*何时做*。命令队列是 [observer.md](observer.md) 事件处理的常见伙伴，事件成为要处理的命令。撤销通常结合命令与备忘录或 [unit-of-work.md](unit-of-work.md)。在 Python 中，权衡每个命令类与普通可调用对象——只有对状态、序列化、排队、审计或撤销的需求才能证明对象的合理性。
