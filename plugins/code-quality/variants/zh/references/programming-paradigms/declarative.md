# 声明式编程（Declarative Programming）

## 什么是声明式编程

声明式编程（Declarative Programming）描述的是你*想要*什么——目标、约束、数据的形状——而将*如何做*留给框架、库或引擎。你不是编写步骤，而是编写描述，让其他东西来执行它。SQL 是典型的例子：你陈述你想要的结果集，查询计划器决定如何生成它。配置文件、模式、路由表、CLI 参数定义、验证规则和规则引擎都是声明式的。

在 Python 项目中，声明式风格出现得比人们注意到的多得多：`pyproject.toml`、`dataclass` 字段列表、`TypedDict`、`argparse` 解析器、Web 框架的路由装饰器、权限表、状态机的转换表。这些都是由某个引擎解释的数据，而不是你逐步执行的代码。

## 底层假设

- 某些结构作为*数据*来检查、组合和文档化，比作为命令式代码更容易。
- 框架或库可以拥有重复的执行逻辑，因此你的代码只表达与默认值不同的部分。
- 当规则稳定且执行模型很好理解时，声明它们可以消除样板代码并集中化事实来源。

## 何时适用

- 项目和工具配置：`pyproject.toml`、Ruff、pytest、pre-commit。
- 模式和类型定义：`dataclass` 字段、`TypedDict`、Pydantic/FastAPI 模型、序列化元数据。
- 由框架执行的结构定义：CLI 参数、Web 路由、权限表、映射表、转换表。
- 查询和转换：SQL、查询构建器、声明式数据映射。

共同点：一个稳定的结构，一个定义良好的执行引擎，以及从一个可检查的单一事实来源中受益。

一个从命令式到声明式转换的小例子。命令式分发：

```python
def handle(event: str) -> None:
    if event == "submit":
        do_submit()
    elif event == "pay":
        do_pay()
    elif event == "cancel":
        do_cancel()
    else:
        raise ValueError(event)
```

声明式分发——映射*就是*逻辑，一个小型引擎应用它：

```python
HANDLERS: dict[str, Callable[[], None]] = {
    "submit": do_submit,
    "pay": do_pay,
    "cancel": do_cancel,
}

def handle(event: str) -> None:
    try:
        HANDLERS[event]()
    except KeyError:
        raise ValueError(event) from None
```

声明式形式使所有情况在一个地方可见，易于扩展，并且可以作为数据进行检查、计数和测试。代价是"何时运行什么"现在多了一层间接——这正是下一节要讨论的权衡。

## 何时成为问题

- 声明背后隐藏着真实的控制流，但执行顺序和错误位置不可见。当声明式管道失败时，堆栈跟踪指向框架内部，而不是你的意图。
- 配置增长到实际上成为一种编程语言——用 YAML 或 TOML 表达的条件、循环和模板，却没有语言工具的支持。
- 为了"灵活性"构建了 DSL 或扩展点，但只有唯一的调用点。这是投机性的结构；参见 [yagni](references/design-principles/yagni.md)。
- 调试需要理解引擎的评估模型，而该模型文档不全或令人惊讶。

声明式风格用显式控制流换取简洁性。当引擎值得信赖且结构稳定时，这种权衡是值得的；当你需要看到并逐步跟踪实际发生的事情时，这种权衡是有害的。

一个实用的测试：如果新人问"这段代码运行时发生了什么？"，而诚实的回答需要在回答之前解释引擎的评估模型，那么声明可能吸收了太多逻辑。声明应该描述*事实和结构*；当它们开始描述*序列和决策*时，通常命令式形式更清晰。

## 在 Python 中

- 为每个声明式结构提供一个单一的事实来源。不要为类型检查器写一次模式，为运行时验证再写一次，为文档再写第三次——尽可能进行派生或生成。
- 在边界处将外部配置解析为类型化对象（`dataclass`、Pydantic 模型），然后让其余代码使用具体类型而不是原始字典。参见 [data-oriented.md](./data-oriented.md)。
- 复杂的规则表仍然需要测试。声明式并不意味着免测试；转换表或权限矩阵需要覆盖其行及其被拒绝的情况。
- 对于隐藏控制流的声明式机制——装饰器、路由注册、信号处理器——确保实际执行路径仍然可追踪。参见 [event-driven.md](./event-driven.md) 了解相关的不可见连接风险。
- 保留一个命令式逃生口。最好的声明式设计允许罕见的不规则情况回退到普通代码，而不是迫使每个异常都塞入声明的词汇表。将路径映射到处理器的路由表保持声明式；而增长出一个 `condition` 微型语言来表达"仅限周二的高级用户"的路由表，已经开始糟糕地重新发明编程语言了。声明常规情况，让普通函数处理不规则情况。

## 与其他范式的关系

- [state-machine.md](./state-machine.md) 的转换表是声明式的：合法转换是数据，由一个小型引擎应用它们。
- 声明式配置供给命令式外壳（[imperative.md](./imperative.md)），外壳读取配置并执行操作。
- 声明式和面向数据的设计有大量重叠：两者都将结构视为可检查的数据而不是行为。区别在于侧重点——声明式关注将执行交给引擎，面向数据关注良好地建模数据本身。
- "更声明式"本身从来不是目标。目标是让规则和数据形状成为中心、可检查和可文档化。当声明不再作为事实可读，而开始隐藏决策时，这就是退向显式代码的信号。
