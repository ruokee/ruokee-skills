# 项目结构

一个长期维护的 Python 项目绝不应仅仅是"一个碰巧能被当前工作目录导入的目录"。结构的存在是为了让可导入的包、测试、工具配置、运行入口点、依赖声明和部署边界各自都有一个明确的位置。正确的形态取决于项目如何使用以及预期维护多长时间；本文档涵盖所有常见形式以及推动项目从一种形态演进到下一种的信号。

## 单文件脚本

单个 `.py` 文件是一次性自动化任务、小工具和实验的正确形态。通过 PEP 723 内联脚本元数据，即使是独立文件也可以在注释块中声明其依赖和所需的 Python 版本，因此像 `uv run script.py` 这样的运行器可以在没有项目结构的情况下构建隔离环境：

```python
# /// script
# requires-python = ">=3.12"
# dependencies = ["httpx"]
# ///
import httpx
```

这适用于脚本确实很小、没有测试需求，并且不是长期维护的组件的情况。表明它已超出单文件范围的增长信号包括：积累了多个辅助函数、需要自己的测试、出现配置文件，或者其他代码想要导入其中的一部分。此时，应将其升级为真正的包，而不是让它变成一个永久追加的超大脚本。

## 扁平布局

扁平（或"no-src"）布局将导入包直接放在项目根目录：

```text
project/
├── pyproject.toml
├── mypkg/
│   ├── __init__.py
│   └── app.py
└── tests/
    └── test_app.py
```

它直观、启动成本低，常见于小型库、历史项目和简单的内部工具。其结构上的弱点是项目根目录自然进入 `sys.path`，因此本地测试和脚本倾向于导入*源码目录*而不是*已安装的产物*——掩盖了打包错误，例如构建的 wheel 中缺少文件。

真正的退化风险不是扁平布局本身，而是它退化为"大脚本目录"：`run.py`、`utils.py`、`db.py`、`api.py` 都散落在根目录下，通过当前目录相互导入，入口点和库逻辑纠缠在一起，导入时的副作用无人能追踪。如果你选择扁平布局，仍然要保持清晰的包目录，将入口脚本与可导入逻辑分离，并禁止严重的导入时副作用（读取环境变量、打开连接、启动线程）。

## Src 布局

Src 布局将预期的导入包移入 `src/` 子目录：

```text
project/
├── pyproject.toml
├── README.md
├── src/
│   └── mypkg/
│       ├── __init__.py
│       └── app.py
└── tests/
    └── test_app.py
```

这种分离的关键在于*导入安全*：由于 `src/` 默认不在 `sys.path` 上，你不会意外地从源码树导入包。测试针对*已安装*的包（通过可编辑安装）运行，因此它们使用了与实际用户相同的导入路径和打包边界。这能捕获一类扁平布局直到发布前都会隐藏的 bug——模块遗漏在构建中、数据文件未被打包。

对于任何将要发布、部署或长期维护的项目，选择 src 布局：库、SDK、框架、CLI、Web 服务和工作空间成员。代价是稍高的初始心智模型（你必须安装包，即使是可编辑安装，才能运行它），但换来的是名副其实的测试。本地开发使用可编辑安装或等效的 uv 管理环境；永远不要依赖手动编辑的 `PYTHONPATH`。

## 打包应用

打包应用是可安装、可运行、可部署的——CLI、Web/API 服务、后台工作进程或内部平台工具——通过 `[project]` 元数据描述，而不是作为松散脚本存在：

```text
project/
├── pyproject.toml
├── uv.lock
├── README.md
├── src/
│   └── app_name/
│       ├── __init__.py
│       ├── __main__.py
│       └── cli.py
└── tests/
    └── test_cli.py
```

`[project]` 表声明了 `name`、`version`（或 `dynamic`）、`requires-python` 和 `dependencies`。入口点属于 `[project.scripts]`，这样工具通过其名称调用，而不是通过 `python src/app_name/main.py`。非运行时依赖——开发、测试、lint、文档——放在依赖组中，保持在运行时集合之外（参见 [dependency-management](dependency-management.md)）。锁文件固定了部署环境。

常见的误解是"可安装"意味着"必须公开发布"。并非如此。私有应用被构建成 wheel 用于内部部署；防范措施是防止其被意外发布——CI 中没有公开的 PyPI token，默认没有发布步骤，以及一个不会冒充公共包的名称。

## 工作空间

工作空间是一个多包仓库，其中多个包或应用共享一个锁文件和一套工具入口点。uv 工作空间适用于应用加共享库、一组服务，或一个共同演进的内部包的单体仓库：

```text
repo/
├── pyproject.toml
├── uv.lock
├── packages/
│   ├── app-api/
│   │   ├── pyproject.toml
│   │   └── src/app_api/
│   └── shared-domain/
│       ├── pyproject.toml
│       └── src/shared_domain/
└── tests/
```

只有当确实存在至少两个具有不同、可描述职责的包，它们之间有清晰的依赖*方向*，并且共享包公开了稳定的 API（而不是临时从应用中划分出来的内部目录）时，才引入工作空间。工作空间根目录拥有共享的工具配置和整个仓库的命令；每个成员拥有自己的元数据和运行时依赖。应用成员不得深入共享包的内部——它们依赖于其公共 API，且仅遵循声明的方向。

不要仅仅因为有很多目录，或者为了按文件夹拆分一个大应用，或者当包之间没有稳定边界地相互导入内部时，就使用工作空间。如果没有全仓库的测试、类型检查和依赖升级策略，工作空间会放大耦合而不是遏制它。

## 决策表

| 项目类型 | 推荐布局 | 需要升级的信号 |
|-|-|-|
| 一次性自动化、实验 | 单文件脚本（PEP 723） | 多个辅助函数、测试需求、配置文件或其他模块想要导入它 |
| 小型库、简单内部工具 | 扁平布局 | 多个入口点、依赖组，或打包错误开始出现 |
| 发布的库、SDK、框架 | Src 布局 | （任何维护或发布的项目默认选项） |
| CLI、Web/API、服务、内部应用 | 打包应用（src + `[project]`） | 需要部署产物、入口点和锁文件 |
| 应用 + 共享库、服务组、单体仓库 | 工作空间 | 两个以上具有稳定 API 和清晰依赖方向的包 |

当对于任何将持续超过一周的项目在扁平布局和 src 布局之间犹豫时，优先选择 src：导入安全保证是防止打包意外的廉价保险。

## 测试目录、配置和入口点

在所有打包形式中，将测试放在顶层的 `tests/` 目录中，而不是放在生产包内部，除非生态系统有强烈的相反约定。根据行为和边界命名测试，而不是机械地镜像每个实现文件；对于大型框架和 SDK，对包树的松散镜像有助于导航。将测试运行器明确指向测试目录（`testpaths = ["tests"]`），这样它就不会进入临时目录或构建的文档。覆盖率源应指向实际的包路径，以便工具和脚本不会污染生产覆盖率数字。详细的测试约定见 [testing](references/spec/testing.md)。

工具配置集中在 `pyproject.toml` 的 `[tool.*]` 下，但工具真正需要自己的文件的情况除外（pre-commit 保留 `.pre-commit-config.yaml`）。已安装命令的入口点属于 `[project.scripts]`；`__main__.py` 支持 `python -m app_name`。指导原则是项目形态、依赖组、测试范围和类型检查范围都应该*在配置中显式声明*，而不是由当前目录碰巧可导入的内容来隐含。
