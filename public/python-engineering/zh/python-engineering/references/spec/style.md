# Code Style

Python 项目的 style 可以清晰分成两层：一层是工具可以机械决定并强制执行的部分，另一层是关于含义的人类判断。把二者混为一谈，会把 review 精力浪费在 formatter 早已解决的事情上；更糟的是，还会让真正重要的东西 - 名字、边界、抽象层次 - 因为大家在争论行宽而被放过。本文要做的，就是划清这条线，把判断力只花在最值得的地方。

## PEP 8 基线

PEP 8 是 Python 的默认 style 基底：布局、缩进、导入、命名、注释，以及少量编程建议。它是让陌生 Python 代码可读的共享基础。但它并不是完整的工程规范 - 它没有谈到类型边界、异常语义、工具链或架构，而且它的许多具体格式规则早于现代自动格式化器。采用 PEP 8 的正确方式是“默认遵循它，在本地一致性和可读性更好的地方允许胜出”，而不是“把所有历史格式条款逐字转成硬规则”。

## 自动化边界

style 管理中最有用的一步，是把一切机械性的东西交给工具，然后停止争论。

formatter（Ruff 的 formatter，或 Black）负责布局：换行、引号风格、空白、尾随逗号、括号位置。这些没有语义内容，因此唯一重要的是一致性，而 formatter 可以免费提供。只要 formatter 在 pre-commit 和 CI 中运行，格式就不再是 review 话题 - 没什么可争论的。

linter（Ruff）负责那些可以机械检测的内容：未使用的 import、import 顺序、明显死代码、被现代目标版本淘汰的过时语法，以及大量容易出 bug 的模式。linter 通过有意识地选择规则族来配置，而不是把所有规则都打开；新增的每条规则都应该在误报成本面前证明自己的价值。工具的具体信息 - 哪些规则族、如何配置 - 放在工具参考里；这里重要的是原则：如果规则可以决定，review 里就不该由人来决定。

因此，人类 review 该关注的是工具 _无法_ 判断的内容：一个名字是否传达了意图，一个 module 是否只承担一个职责，一个抽象是否物有所值，代码是否真的在表达它想表达的意思。style review 的注意力应当放在这里。

## 命名

名字是影响最大的 style 决策，因为它们被读取的次数远多于被写下的次数，而且没有任何工具能评估它们。除了 PEP 8 的机械约定（函数和变量用 `snake_case`，class 用 `PascalCase`，常量用 `UPPER_CASE`）之外，更重要的是名字在使用场景下是否传达了意图。

一个名字应当描述某物在领域中 _是什么_ 或 _做什么_，而不是它的类型或实现。`users` 比 `user_list` 好；`deadline` 比 `dt2` 好；`is_expired` 比 `flag` 好。作用域应当指导长度：循环索引可以是 `i`，但模块级函数不行。避免把读者已经能从 annotation 看出来的类型信息写进名字，也避免使用节省几个字符、却让每次阅读都要多花一瞬解码的缩写。布尔值最适合写成谓词形式（`is_`、`has_`、`should_`）；有副作用的函数最好写成动词。

## 模块边界

一个 module 应当有一句话就能说清楚的存在理由。当一个 module 塞进了互不相关的职责 - 请求处理、数据库访问、格式化 helper 混在一起 - 每次修改都会碰到它，每次 import 都会把整堆东西带进来。工具无法判断的是：module 里的这些东西是否真的该放在一起。它们是否为了同样的原因而变化，是否处于相同的抽象层级，读者为了找其中一个是否会自然想到另一个。判断标准是内聚，而不是文件大小。一个聚焦的 400 行 module 比一个叫 `utils`、被所有人 import、但没人真正拥有的 50 行杂物堆更健康。

## 抽象层次

好代码在任何单个 function 内都应保持一致的抽象层次：高层编排函数应该调用命名清晰的步骤，而不是把编排和字节级操作混在一起。混杂层次 - 一个 function 既在决定业务策略，又在操作字符串索引 - 会迫使读者不断切换思维档位。纠正方法不是“更多抽象”，而是 _一致的_ 抽象：把低层细节抽到一个名字后面，让高层函数读起来像一连串意图。这里的判断带有真实的反面风险（过早抽象、没有回报的间接层），这正是它无法被自动化的原因。

## 显式性

PEP 20 的“显式优于隐式”是一种 style value，而不是 lint rule，它体现在许多小决策里：把 dependency 作为参数传入，而不是去抓全局变量；返回一个命名类型，而不是让调用者自己记得如何解包的裸 tuple；抛出具体异常，而不是裸 `Exception`；把魔法数字命名为常量。显式性会牺牲一点简洁，换来大量可读性，在接口边界处几乎总是值得。与之相对的是，过度显式会变成噪音 - 把一个好名字或清晰类型已经表达出来的内容再展开一遍，什么也没增加。判断标准是：让某件事显式化是否 _减少_ 了读者的工作；如果减少了，就优先采用它。
