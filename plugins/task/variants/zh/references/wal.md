## 使用 WAL 记录活动

WAL 意为工作活动日志（Work Activity Log）。用它来保存 Task 推进时发生了什么，以及为什么该历史对未来工作重要。

不要将 WAL 视为：

- 当前 Task 状态的来源；
- 命令转录；
- 具有不可变性保证的审计日志；
- 计划、研究或大型移交文件的替代；
- 必须回放以构建当前状态的事件流。

`TASK.md` body 陈述当前事实。WAL 解释随时间推移的持久活动。普通文件存放详细分析、计划、检查点和输出。

## 理解文件格式

Core 基于系统当前本地日期追加到文件：

```text
<task-dir>/wal/YYYY-MM-DD.md
```

权威记录为：

```markdown
## 2026-07-21T15:04:05.123+08:00 · codex:gpt-5.6-sol

完成了 Task 发现和引用解析。

可选的多行细节。
```

只有带有时间戳和 actor 的严格 H2 行才能启动一条解析记录。时间戳必须包含时区。`task_log` 的 message 是非空单行摘要；可选的 `extra_body` 可包含多行。

Actor 必须为单行且不能包含 ` · `。从当前可用的最具体信息设置：先是精确的运行时 `model.id`，然后是 Agent 自身上下文中的精确模型，再是 Agent 能合理说明的最具体模型或模型系列，最后是 `<host>:unknown`。优先使用 `codex:<model>`、`claude:<model>`、`pi:<model>` 或子代理形式如 `codex:<subagent-name>:<model>`。Actor 是基于现有证据尽力准确填充的字段上下文，而非经过身份验证的运行时证明：没有理由时不要降级已知的精确模型，且不要发明 Agent 无法证明的模型。当没有模型上下文可用时，Core 使用 `<host>:unknown`。

不要创建记录 ID、JSON 负载或结束标记。当 `task_log` 可以安全追加时，不要手动构建 WAL。

## 区分 Core 和 Agent 记录

Core 自动记录机械的结构化变更：

- Task 创建；
- 分支、关系和 `extra` 变更；
- 生命周期转换和归档操作；
- 重命名；
- 批量子任务创建的父级摘要。

除非需要额外的持久推理，否则不要立即用 `task_log` 记录重复这些事实。

Agent 撰写的记录用于 Core 无法推断的语义活动：

- 研究发现和分析结果；
- 决策及其下游影响；
- 有意义的用户或外部工具变更；
- 子代理和移交结果；
- Task 文字与实际项目状态之间的差异；
- 未完成的会话状态和下一步。

## 决定记录什么

当在下次会话中丢失某条活动会迫使重新发现或可能导致错误的下一步行动时，记录该活动。

基于语义事件记录，而非基于计时器或固定记录计数。一旦形成持久的发现、决策、修正、可恢复的里程碑、验证结果、已验证的协作结果或改变下一步的阻塞项，就在开始下一个独立工作线或询问下一个实质性问题之前追加它。不要等待后续里程碑、测试或会话结束时才批量记录。

将这些视为不同事件，在继续前在各自边界上追加 WAL 记录：

- 研究或修正结论是一个事件；它驱动的编码修改是后续的独立事件；
- 可恢复的实现里程碑是一个事件；你随后开始的验证是后续的独立事件；
- 验证结果是一个事件；其后进行的移交或最终响应是后续的独立事件。

仅合并同一时刻或同一语义事件中共同形成的事实。稍后发生的里程碑或验证结果不得为了减少调用次数而折叠回前面的记录；当其变为持久时，它理应拥有自己的记录。

让会话结束检查点仅填补仍未被记录的内容，绝不将整个会话压缩到一条记录中。如果检查点将成为本会话的唯一 WAL 记录，说明事件边界记录被遗漏了：找出本应已写入的不同发现、修正、里程碑和验证结果，先分别补齐它们作为独立记录。即便如此，也不要为命令流、常规读取、进行中状态或 Core 的机械变更发明记录。

通常记录：

- 改变范围或实现方向的决策；
- 排除了某种方法的研究发现；
- 完成里程碑，其结果从当前文件中不明显；
- 解释为什么 Task/项目状态发生变化的外部变更；
- 阻塞项和恢复所需的证据；
- 父 Agent 验证后的委派结果；
- 未完成工作的简洁会话结束检查点。

通常不记录：

- 常规读取和搜索；
- 每个 shell 命令或工具调用；
- 临时待办列表；
- 未用于决策的推测性想法；
- Core 已自动记录的状态；
- 重复的"仍在工作"消息。

当细节属于普通文件时，记录持久的结论并链接该文件而非复制其内容。

## 编写有用的记录

使 `message` 成为紧凑的已完成事实或当前状况。仅在信息会改变解释或恢复时使用 `extra_body`。

好的模式：

```text
message: 确认分离注册表冲突来自旧项目路径。
extra_body: 注册表条目 ... 仍然是权威的。仅在用户确认项目移动后更新它。
```

弱模式包括"已取得进展"、命令列表或整段讨论转录。

每条连贯活动写一条记录。`task_log` 有意不接受批量记录或记录 ID。

## 通过追加修正历史

WAL 按约定和 Core 行为是只追加的。当旧记录有误时：

1. 保留旧文本不变。
2. 追加一条修正，指明受影响的结论。
3. 陈述新证据和替换结论。
4. 说明任何必须更改的下游工作。

不要重写昨天的 WAL 以使历史看起来没有错误。用户可以手动编辑文件，但 Agent 应保留活动记录，除非明确要求修复意外损坏。

## 在预算内读取

`task_read` 提供：

- `metadata`：无 body 或 WAL；
- `summary`：body 加上选定 WAL 记录的第一段；
- `detailed`：body 加上完整的选定 WAL 记录。

WAL 有独立的字符和记录预算。默认值来自有效配置。Core 从最新到最旧选择，并按时间顺序返回选定的记录。

`wal_truncated: true` 表示磁盘上还有更多内容。如果最新的单条记录超过字符预算，Core 可能返回标记为截断的记录。仅在当前决策需要更多上下文时增加预算或读取特定 WAL 文件。

不要将旧 WAL 复制到 `TASK.md` 中以规避预算。保持当前事实简洁，按需逐步加载历史。

## 处理移交和会话结束

在结束未完成的工作前，写一条简洁的 WAL 记录，涵盖：

- 已完成的内容；
- 当前阻塞项或重要决策；
- 下一步有用的操作。

如果这将成为一条长记录，在 `task_dir` 下创建普通的移交/检查点文件，从 `TASK.md` 或 WAL 链接它，并保持 WAL 摘要简洁。

分配给子任务的子代理仅在其分配边界内写入。它不写入父级 WAL；父 Agent 检查结果并记录父级级别的后果。

## 处理已关闭和已归档的 Task

对于已关闭和已归档的 Task，`task_log` 仍然可用，以便以后的事实、修正、迁移说明或外部结果可以保存下来。

记录不会重新打开或取消归档 Task。不要使用活动记录暗示生命周期转换；使用带必需原因和授权的 `task_update`。

在 Core 能追加 WAL 之前，受管 frontmatter 必须保持有效，即使 Task 已关闭。

## 从 WAL 失败中恢复

Core 要求 WAL 目录和目标文件是可写的常规文件系统对象。它拒绝符号链接/非目录的 WAL 根目录、符号链接/非常规的目标文件以及非 UTF-8 的现有内容。

未解析的手动文本可能在读取时产生 `wal_unparsed_text` 或时间戳警告。当目标是常规 UTF-8 文件时，Core 可能仍能追加；保留手动文本并诊断是否需要显式修正。

结构化 Task 状态在自动 WAL 追加前已提交。如果创建/更新/重命名结果成功并带有 `committed: true` 和 `wal_write_failed`：

1. 不要重复原始的状态更改操作。
2. 重新读取 Task 元数据以确认已提交的状态。
3. 检查 WAL 路径和确切警告。
4. 仅在数据可能被覆盖时，经用户授权修复文件条件。
5. 在 WAL 安全后追加一条简洁的缺失活动记录。

对于无效 UTF-8、符号链接、重复身份或不确定的暂存状态，在更改文件前阅读[诊断与修复](diagnostics-and-repair.md)。
