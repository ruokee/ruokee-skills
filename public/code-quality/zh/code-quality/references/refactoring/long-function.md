# Long Function

## 它是什么

Long Function 是最常见的 bloater，也是最容易被误判的一个。这个 smell 不是按行数来度量的 - 当一个函数 **混合了多个抽象层次、承担了不止一个职责，或者包含了必须通读才能理解的多个子段** 时，它才算太长。长度只是一个线索，提示其中某件事正在发生，而不是问题本身。一个四十行、但始终处于同一抽象层次、而且是一条清晰序列的函数没有问题；一个十二行、却把高层策略和底层字节操作交织在一起的函数则不行。

真正可靠的判断是：你能不能诚实地给这个函数命名。如果名字能忠实概括里面的一切，那它就是凝聚的。如果你诚实地只能把它叫成“验证订单、_并且_ 计算总额、_并且_ 格式化收据、_并且_ 发送邮件”，那它就是在做好几件事，而这些 “and” 就是它想要拆开的边界。

## 什么会让函数变长

- **抽象层次混杂。** 代码在“做什么”（编排业务流程）和“怎么做”（字符串格式化、索引运算、字典整理）之间来回跳。读者必须不断切换高度。每一块底层代码都适合提成一个命名函数，这样高层流程才能顺畅阅读。
- **多个职责。** 明确不同的阶段 - 解析、验证、转换、持久化 - 全塞进一个函数里。每个阶段都可以被提取，尤其当它们只通过少量局部变量通信，而这些变量又自然形成了接口时。
- **难以命名的子段。** 一个明显的信号是：某个块前面有注释，例如 `# now compute the discount`。注释其实是在给代码拒绝命名的概念命名。那一块通常就是一个等着被提取出来的函数，注释会变成函数名。
- **深层嵌套和长条件。** 箭头状代码可以通过 guard clause 或提取谓词来压平逻辑并显露意图。

## 提取信号

在以下情况下，可以考虑 [extract-function.md](./extract-function.md)：

- 你写了一段注释来解释接下来的几行 - 把它们提出来，让名字承载那段注释。
- 一段代码只使用了一小簇局部变量，而函数其他部分都不用这些变量 - 那簇变量就是一个函数边界。
- 你发现自己要滚动页面才能在脑中装下整个函数。
- 某个阶段本可以单独测试，但现在却因为埋在里面而做不到。

## 何时长度是可以接受的

并不是所有长函数都是 smell，强行拆开反而可能把一条线性故事切成很多小函数，让读者得自己重新拼回去。下面这些情况里，长度是可以接受的：

- **它是一个抽象层次上非常清晰的序列。** 一个函数做十个步骤，按顺序，全部处于同一层次，没有分支复杂度，从上到下像散文一样可读。硬拆只会增加跳转成本。这和 [thin-wrapper-function.md](./thin-wrapper-function.md) 的张力有关 - 提取一个只被用一次、没有抽象边界的步骤，通常会得到一个浅 helper，伤害比长度本身更大。
- **配置或数据。** 一个很长的字面量表、一个很大的映射、一组常量 - 这里的长度只是数据量，不是逻辑复杂度。
- **测试准备和测试主体。** 测试即使很长，通常也更适合显式、线性的 arrange-act-assert；过度提取 helper 反而会藏住测试到底在验证什么。
- **一个包含很多 case 的 `match` 或 dispatch**，其中每个 case 都很短，而且结构是平铺的。

## 在 Python 中

- 把阶段提成模块级函数或方法；它们共享的局部变量会变成参数和返回值，这通常能把数据流说清楚。
- Guard clause（`if not valid: raise`）能有效压平嵌套，而且读起来很自然。
- 注意每个提取出来的函数都会增加一次调用帧；在真正的热点循环里，这一点是可测量的，不过对普通代码来说，清晰度通常更重要。相关权衡见 [thin-wrapper-function.md](./thin-wrapper-function.md)。
- comprehension 或 generator 可以替代累积式循环块，在不遮蔽逻辑的前提下缩短函数。

## 与其他 smells 和 refactorings 的关系

Long Function 是 [extract-function.md](./extract-function.md) 的主要触发器。它经常和 Long Parameter List、Data Clumps 一起出现 - 当提取出来的函数需要同样五个参数时，这些参数其实是在要求自己变成一个 parameter object。它也可能是在提示缺少一个抽象：在 agent-assisted code 里，函数会变长，常常是因为每来一个新需求就再加一个分支，把函数变成变化的垃圾桶；这种情况更接近 [divergent-change.md](./divergent-change.md)，正确修法是找变化轴，而不是继续拆函数。
