# KISS — Keep It Simple

## 它是什么

KISS 说的是：选择足以满足实际目标的最简单设计。陷阱在于把“简单”读成“短”。简单不是按行数来衡量的。它衡量的是读者为了理解并安全修改代码，必须在脑中同时持有多少概念：依赖数量、隐藏控制流的多少、状态的宽度，以及一次变更会传播多远。

一个更可操作的说法是：尽量减少 _不必要的_ 复杂度——也就是你额外加上的复杂度，而不是问题本身固有的复杂度。

## 必要复杂度 vs 偶然复杂度

这个区分就是整个原则的核心。

- **必要（本质）复杂度** 来自问题本身。正确的分布式一致性协议之所以复杂，是因为一致性本来就难。税务引擎之所以复杂，是因为税法本来就复杂。你无法把它简单化掉。
- **偶然复杂度** 来自解决方案：投机性的抽象层、没人要求的配置开关、为了一个调用点而存在的框架、只是为了显得灵活而引入的间接层、读十分钟才能懂的花哨一行代码。

KISS 针对的是偶然复杂度。它不是要求你假装难题很容易。面对真正复杂的领域，目标是把复杂度封装进一个边界清晰、interface 简单的 module 里（见 [deep-modules.md](./deep-modules.md)），而不是把它稀释到整个系统里。

## 简单是“更少的概念”，不是“更少的行数”

两份清晰、稍有重复的拷贝，往往比一个带三个 boolean flag 和一个 callback 的 generic function 更简单。一个扁平的语句序列，往往比一串逼读者在文件中来回跳转的小 helper 更简单。一个普通的 `dict` 查找，往往比一个带 plugin protocol 的 registry 更简单。当你把行数压缩了，却增加了概念数量，那只是让代码更短、更难——这正与 KISS 相反。

这也是 KISS 与天真的 [dry.md](./dry.md) 以及“函数越小越好”的笼统偏好之间存在张力的原因：二者都可能把具体、可读的重复，换成抽象但更难掌握的结构。

## 过度设计的信号

- 只有一个 implementation 的 abstraction、interface 或 base class。
- 当前没有任何 caller 使用的配置选项、flag 或 extension point（这也是违反 [yagni.md](./yagni.md) 的表现）。
- 只有“灵活性”或“以后也许能替换”作为理由的多层间接。
- 为了单一具体用途而构建的 framework 或 plugin system。
- 很多只重命名表达式、却没有隐藏复杂度的一行 wrapper function。
- 当直接调用就够时，却引入了花哨的 metaprogramming、dynamic dispatch 或 generic machinery。

## 什么时候复杂度是合理的

KISS 不是不许多做。增加的复杂度在服务以下目标时，才算赚到了自己的位置：

- **Correctness**——处理问题真实存在的边界情况、并发风险或 invariants。
- **Performance**——在热点路径上经过测量、确有必要的优化（同时要把复杂度局部化，并记录原因）。
- **Safety**——系统边界的输入校验、资源清理、以及确实会发生的失败处理。

为了“简单”而删掉这些内容，并不是 KISS；那是在制造脆弱代码。真正的能力，是区分哪些复杂度能换来正确性、性能或安全，哪些复杂度只是在制造“看起来高级”的假象。

## 在 Python 中

- 默认使用清晰的数据结构、直接控制流、标准库，以及少量命名良好的函数。
- 当复杂度不可避免时，把它隔离在一个 module、adapter 或 deep module 中，让调用方保持简单。
- 不要为了单一调用点创建 abstraction，也不要把一切都塞进一个巨型函数里——这两种都违背了简单性，只是方向相反。
- 让 formatter 和 linter 处理样式，让人的注意力集中在边界和行为上。

## 与其他原则的关系

- [yagni.md](./yagni.md)：YAGNI 去掉投机性特性；KISS 去掉你保留下来的特性里不必要的复杂度。二者共同抵制过度设计。
- [deep-modules.md](./deep-modules.md)：应对真实复杂度的长期简单做法，是把它藏在一个简单 interface 后面，而不是散落成浅薄的 helper。
- [dry.md](./dry.md)：当去重会增加比它减少更多的概念时，KISS 胜出——保留清晰的重复即可。
