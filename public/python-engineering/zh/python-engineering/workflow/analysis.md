# Analysis — Python Engineering

协作式探索，不是评分。用户想要的是“决定该怎么做”，而不是“知道哪里错了”时，就使用这个模式：头脑风暴一种方案、设计一个机制、分析现有代码的行为，或规划一次重构。这里没有 diff 要打分，也没有 findings 列表要产出。目标是达成一个共同、充分论证过的决定。

## 触发条件

- “我应该怎么组织这个？”，“什么是最好的方式……”，“我有哪些选项？”
- 机制设计：缓存、retry policy、plugin system 或 data model 应该如何工作。
- 代码分析：解释一个 module 如何运行、为什么会长成这样、它的边界在哪里。
- 重构讨论：在真正改动之前先规划变更。

## 前提

- 只读。analysis 不会自行修改代码。
- 收集足够上下文，以便做出具体推理。不要对你没有读过的代码进行理论化猜测。

## 收集上下文

先读，再推理。所需上下文的多少取决于决策的规模：

- 项目事实：读取 `pyproject.toml`，看 `requires-python`、依赖和项目形态。版本 floor 决定了哪些语言特性根本还在桌上。
- 相关讨论的实际代码，以及它的调用者和测试，这样你的选项就建立在真实结构上，而不是想象中的结构上。
- 偏好（`.agents/preferences/python-engineering.md` 或目录形式）如果存在，也应读取 - 但在 analysis 中，偏好是要权衡的输入，而不是必须执行的规则。
- 与当前机制相关的叶子文档（例如 [match-case](../references/grammar/match-case.md)、[type-hint](../references/spec/type-hint.md)），这样你给出的取舍就是既定的，而不是临时想出来的。

## 如何组织分析

先给建议，再给理由。好的 analysis 不是把所有可能性穷举一遍；它应该给出清晰路径，并诚实地讲出取舍。

1. **框定决策。** 复述真正的问题以及限制它的约束 - 版本 floor、现有架构、依赖限制、代码需要存活多久。如果你读到隐藏约束，就把它说出来。
2. **提出真正不同的选项。** 两到三个真实可比的方案，而不是五个微小变体。对每个方案说明：它是什么、代价是什么、何时更优。若选项在同一维度上并列，可使用简短比较表。
3. **给出推荐。** 说明你会选哪个方案，以及为什么，理由要落在第 1 步的约束上。一个带着可被用户反驳的理由的推荐，比中立菜单更有价值。
4. **说明代价。** 每个真实选择都会失去些什么。说清推荐路径的代价，让用户在知情的情况下做决定。

## Recommend 与 Ask 之间

- **推荐并继续细化**：当约束明显指向一个选项，且出错成本低或可逆时。
- **停下来询问**：当决定取决于只有用户知道的信息（部署目标、团队约定、未来计划），当选项在只有用户能权衡的轴上取舍（速度 vs 简洁、灵活 vs 清晰），或者变更很大且难以逆转时。
- 不要为了得到一个整洁答案而虚构需求。如果正确回答是“这取决于 X”，就直接说出 X 并询问。

## 输出

使用 prose 和表格，而不是 findings block。深度应与问题相匹配 - 小设计问题给几段话，大结构决策给选项和推荐。最后要落到一个具体的下一步，只有在用户同意后才进入实现。

## 停止规则

- 不要修改代码；analysis 在用户接受计划前始终只读。
- 不要运行会写文件、创建 `.venv` / 缓存，或改变 lockfile 的命令。
- 不要把真实的“这取决于”压成虚假的确定性。
- 不要把偏好说成 Python 的普适真理 - 它们只是这个项目的选择。
- 不要为了凑选项而塞进我不会推荐的方案；三个真实方案比七个凑数方案更好。
