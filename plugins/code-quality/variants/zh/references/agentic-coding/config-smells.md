# 智能体配置坏味道（Agent Configuration Smells）

配置 AI 智能体的文件——`AGENTS.md`、`CLAUDE.md`、SKILL 文件、提示规则、权限清单、工作流文档——是工程制品，它们会以特有的方式退化。就像 [code-smells](references/refactoring/code-smells.md) 中的代码坏味道一样，下面的每个条目都是一个*需要调查的症状*，而不是一个定论。一个很长的配置文件可能完全正确；一条重复的规则可能是有意的提醒。判断始终在于：这个坏味道是否在消耗智能体的准确性、令牌或一致性——而每节末尾的误报边界与检测信号同样重要。

这些坏味道与平台无关。无论智能体是 Claude、Codex、Cursor 还是其他什么，底层机制都是相同的：有限的上下文窗口、作为优先级读取的指令、模型可能遵循也可能不遵循的引用，以及与其所描述的代码库逐渐偏离的配置。

## 上下文膨胀（Context Bloat）

**是什么。** 低频、冗长或与任务无关的信息在每个轮次中永久加载到智能体的上下文中。典型形状是一个巨大的系统提示，将整个风格指南、完整的工具文档和详尽的示例内联粘贴，使它们无论当前任务是否需要都存在。

**为什么重要。** 上下文是与实际任务共享的固定预算。每个花在常驻参考材料上的令牌都是用于正在编辑的文件、错误输出或推理链的一个不可用的令牌。除了原始预算外，信号也会被稀释：与此任务相关的规则埋藏在数百行无关内容中，而模型对它们大致同等加权。膨胀的上下文也会在每一轮中增加成本和延迟，而不仅仅是需要该信息的轮次。

**检测信号。** 以数千令牌计量的系统提示或始终加载的配置。内联粘贴而不是链接的整篇文档。无论任务如何都存在的参考材料（API 规范、完整风格指南、长枚举）。尽管大多数会话从不触及某个主题，但相同的、跨每个会话加载的大块内容。

**解决方案。** 渐进式披露：保持常驻层小巧，并按需路由到详细信息。用一行摘要加一个链接（参见 [盲目引用](#blind-reference) 了解如何编写链接）替换内联转储，智能体仅在触发条件适用时才读取该链接。条件加载——将领域特定上下文门控在需要它的任务之后。本技能自身的结构就是一个例子：SKILL 文件保持简洁，并指向像本文档这样的叶子文档，这些文档仅在相关信号出现时加载。

**示例场景。** `AGENTS.md` 增长到 1,200 行，因为每个库的约定、每个过去的事件以及完整的部署手册都被粘贴进来"以便智能体始终拥有它们"。一个修复拼写错误的任务现在读取了全部 1,200 行，而部署部分挤占了实际正在编辑的函数。

**误报边界。** 小型、高频、真正跨领域的规则属于常驻上下文——那不是膨胀，那是配置的职责。一个每个任务都依赖的 30 行 `AGENTS.md` 是大小合适的。长度本身不是坏味道；*与不相关性加权的长度*才是。按使用频率审计，而不是按行数。

## 技能泄漏（Skill Leakage）

**是什么。** 任务特定知识被放置在错误的抽象层级——要么太高，使全局上下文对每个人都变得杂乱；要么太低，需要它的智能体永远不会到达它。这是一个分层错误：正确的知识放在了错误的层级。

**为什么重要。** 过高层级的知识是 [上下文膨胀](#context-bloat) 的一种形式——一个专门规则让每个不相关的任务都付出代价。过低层级的知识以更安静的方式更糟：智能体永远不会加载它，默默地违反了一个它本来会遵守的约束，没有人注意到直到输出出错。两者都是同一信息的错误路由。

**检测信号。** 位于全局用户级配置中的项目特定或仓库特定规则。一个关键的高频约束埋在三层深的、很少加载的叶子中。专家级的边缘情况指导放在面向初学者的入门提示中，反之亦然。一条自然受众是"仅触及子系统 X 的任务"的规则，却存在于所有任务都为其付费的地方。

**解决方案。** 按频率和范围分层。全局配置仅存放对所有项目都成立的内容；项目配置存放项目真相；技能或叶子存放仅其触发任务需要的内容。将加载条件与知识的真实受众匹配：高频且广泛的上移，低频且狭窄的下移——但永远不要低到其触发任务无法找到它。

**示例场景。** 用户的全局 `~/.claude/CLAUDE.md` 包含"在此仓库中，迁移通过 `make db-upgrade` 运行"。用户接触的每个项目现在都携带一个适用于一个仓库的指令，而仓库自身的配置——它应该属于的地方——保持沉默。

**误报边界。** 某些知识确实是跨领域的，即使看起来很具体也应该放在高层（一个绝不能遗漏的安全规则可能理应在常驻上下文中）。而且有意将罕见的、繁重的细节推迟到深层叶子是正确的设计，而不是泄漏，*只要*一个可靠的触发器在需要时将智能体路由到那里。

## Lint 泄漏（Lint Leakage）

**是什么。** 格式化工具、linter 或类型检查器已经机械性强制执行的规则，以自然语言提示规则的形式重复陈述。"始终使用 4 空格缩进"、"按字母顺序排序导入"、"没有未使用的变量"、"每行不超过 88 个字符"——所有这些已经是别人的工作了。

**为什么重要。** 确定性工具完美且免费地强制执行这些规则；提示规则以概率方式强制执行它们，代价是上下文预算和注意力。更糟的是，以提示陈述的格式化规则会诱使智能体将推理能力花在手工格式化代码上，而格式化工具无论如何都会重写这些代码，并且一旦工具配置发生变化，它们就会过时——现在提示和 `.ruff.toml` 不一致，智能体有了两个主人。用散文表达的机械规则稀释了那些真正需要人类语言解释的规则。

**检测信号。** 任何可以用 `ruff`、`eslint`、`prettier`、`black`、`gofmt` 或 `mypy` 配置表达的提示规则。在 `AGENTS.md` 中陈述的格式化、导入顺序、空白、命名大小写、未使用符号或行长度规则。重复仓库中已存在的 `.editorconfig` 或 pre-commit 钩子的指令。

**解决方案。** 从提示中删除它们并信任工具。将智能体指向命令（"在提交前运行 `ruff check --fix`"），而不是用散文重新实现规则集。为工具*不能*检查的内容保留提示空间：架构意图、何时抽象、领域规则，以及任何会让只读了工具配置的人感到惊讶的行为。

**示例场景。** `AGENTS.md` 列出了十五条风格规则——缩进、引号风格、尾随逗号、导入分组——每一条都已经在仓库的 `ruff` 和 `prettier` 配置中。智能体尽职地重新推导它们，偶尔与格式化工具冲突，而这十五行挤掉了唯一重要的那条规则："绝不要在请求处理器中调用计费 API。"

**误报边界。** *因为工具的行为会出乎智能体的意料*而提及一条规则是合理的——例如，指出非标准格式化工具配置做了不寻常的事情，或某个特定的 lint 规则被有意禁用及其原因。坏味道是重复陈述工具已经静默且正确地做的事情，而不是解释工具与预期偏离的地方。

## 盲目引用（Blind Reference）

**是什么。** 指向另一个文件、URL 或文档的指针，但没有说明*何时*阅读它、*它确立了什么*或*它不涵盖什么*。"另见：`architecture.md`"没有任何上下文就是盲目引用；没有路由逻辑的扁平链接列表也是。

**为什么重要。** 智能体在决定是否花费工具调用和上下文预算来打开引用时需要理由。没有触发条件，智能体要么忽略链接（错过必要的上下文），要么防御性地打开所有链接（以另一种方式重新引入 [上下文膨胀](#context-bloat)）。没有目的陈述的引用也无法被推理：智能体无法判断文档是权威的、说明性的还是仅仅相关的，因此无法权衡在那里读到的东西。

**检测信号。** 没有附带条件的"另见"或"参考"行。标题下的链接列表但没有逐项路由。不说明目标证明什么的引用，因此智能体无法判断找到的答案是否权威。指向外部 URL 的指针，没有说明要提取什么或为什么。

**解决方案。** 每个引用通过触发条件和目的来赢得其位置：智能体*何时*应该跟随它以及它在*那里会得到什么*。本技能中其他地方的索引和路由表就是模型——每一行将代码中的信号与解决问题的文档配对。一个好的引用应该是"当 X 时，阅读 Y，它确立了 Z"，而不是"见 Y"。

**示例场景。** 工作流文档以"相关：`deploy.md`、`rollback.md`、`secrets.md`"结尾。智能体在代码更改任务中途，无法判断其中任何一个是否现在相关，因此它要么全部读取（浪费预算），要么一个都不读（错过 `deploy.md` 本会揭示的部署前提条件）。

**误报边界。** 一个精选的索引，其周围的散文已经提供了路由逻辑，不需要在每一行上重复陈述触发条件——上下文为该节一次性确立。而一个真正是可选丰富内容的"进一步阅读"指针，在标明为可选时是没问题的，这样智能体就知道可以在没有风险的情况下跳过它。

## 初始固化（Init Fossilization）

**是什么。** 由初始化或脚手架模板产生的配置，现已偏离实际的代码库。配置描述的是一个不再存在的项目：引用已删除文件的规则、针对已废弃流程的工作流步骤、针对不再使用的工具的权限、与树不再匹配的目录结构描述。

**为什么重要。** 过时的配置比没有配置更具误导性。智能体将每条指令视为当前为真，因此指向已删除模块的规则会让它寻找已经不存在的东西，针对已退役 CI 管道的工作流步骤会让它执行或预期没有效果的操作。固化的权限是一个更安静的風險：它们可能授予本应撤销的工具访问权限，或阻塞项目现在依赖的工具。智能体无法知道配置在说谎；它自信地在虚构之上推理。

**检测信号。** 命名不再存在的文件或目录的规则（将配置中的路径与树进行 grep 匹配）。引用团队已废弃的流程、服务或命令的工作流步骤。没人再使用的工具的权限条目。与 `ls` 不匹配的"项目结构"部分。初始化模板特有的样板措辞，从未为此项目编辑过。

**解决方案。** 定期审计，理想情况下与重复性触发器（发布、依赖升级、季度审查）绑定。对照当前现实解析每条路径和工具引用，删除不再成立的内容。将配置视为需要维护的代码，而不是一次性写入的制品。当文件或过程在一次变更中被删除时，在同一次变更中清理配置。

**示例场景。** 仓库的 `CLAUDE.md` 仍然说"使用 `make test` 运行测试并在 `htmlcov/` 中检查覆盖率"，但项目已直接迁移到 `pytest` 并在六个月前删除了 Makefile。智能体运行 `make test`，得到"没有此目标"，浪费一轮来重新发现项目实际是如何构建的。

**误报边界。** 并非每个不匹配都是固化——配置可能*有意*描述代码库正在迁移的目标状态，或记录一个前瞻性的、从今往后强制执行的约定。在删除引用之前，确认它意在描述当前现实；进行中的迁移合理地拥有超前于代码的配置。

## 冲突指令（Conflicting Instructions）

**是什么。** 两条规则相互矛盾，没有机制决定哪条获胜——没有说明范围，没有优先级顺序。全局规则说"总是做 X"，项目规则说"永不做 X"，没有任何东西告诉智能体如何协调它们。

**为什么重要。** 面对矛盾且没有解决规则，智能体会非确定性地选择一条，或在各轮之间摇摆，或遵循它最近读到的那条。行为变得不可预测且不可审计——你无法推理智能体会做什么，因为配置不能决定它。矛盾也会侵蚀对整个配置的信任：一旦智能体看到两条规则冲突，每条规则都成为"也许这一条也被覆盖了"的候选。

**检测信号。** 跨文件或跨节关于同一主题的总是/永不配对。范围重叠而不说明哪个范围更窄或优先级更高的规则。通过堆叠例外来解决矛盾的配置（"总是 X，除非 Y，除非 Z"）而不是重构。层级化配置（全局、项目、任务）且没有说明层级之间优先级的配置。

**解决方案。** 使范围和优先级显式化。说明当层级不一致时哪个层级获胜（通常是最具体的：任务覆盖项目，项目覆盖全局）。为每条规则赋予清晰的范围，使两条规则不会同时声称同一情况。最重要的是，从根源解决矛盾而不是掩饰它——如果两条规则真正冲突，其中一条是错误的，或者它们属于不同的范围，修复方法是纠正或分离它们，而不是添加第三条规则来调解前两条。本技能自身的配置正是基于此原因声明了显式的优先级顺序。

**示例场景。** 全局配置说"始终为新函数编写测试"。项目的配置说"这是原型，跳过测试"。两者都没有引用对方。智能体为某些函数编写测试，而其他函数不写，取决于哪个指令在特定轮次中更显著，测试套件最终半途而废且不连贯。

**误报边界。** *看起来*矛盾但操作在不同、清晰声明的范围的规则并非冲突——"默认 X"加上"在子系统 Y 中，改为做 Z"是正确的通用规则加例外结构，不是坏味道，只要范围是显式的且不模糊地重叠。坏味道是未解决的矛盾，而不是规则及其范围的例外共存。
