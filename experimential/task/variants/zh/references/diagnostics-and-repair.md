# 诊断 Task 数据并安全修复

## 先保留证据

不要仅为了让警告消失而"修复"Task。在任何修复前：

1. 停止更改状态的 Task 操作。
2. 记录确切的结构化错误、警告、路径和候选 ID，不暴露无关数据。
3. 当受影响的 Task 数据无法以其他方式恢复时，检查 Git 状态或创建可恢复副本。
4. 尽可能使用 `metadata` 或 `summary` 读取显式 Task 路径。
5. 确定哪些字段/文件是权威的，以及预期身份是否唯一。
6. 在删除、覆盖、移动或解决有歧义的身份前询问用户。

优先选择最小本地修复而非广泛规范化。保留未知 frontmatter、body 字节、WAL 历史、普通材料和无关 Task。

## 使用可用的诊断工具

从预期的项目/工作树目录运行 `task-core check`。当前检查报告：

- 已解析的项目和 Task 根目录；
- Task 候选总数；
- 重复的 UUID；
- 无效的 Task 候选；
- 残留的 `.*.tmp` 暂存文件；
- 发现警告，如非权威路径或名称/slug 不匹配。

它不会自动修复任何内容。它不是模式迁移器，也不会穷尽验证每个关系或 WAL 记录。

在特定 Task 上使用 `task_read` 检查：

- `managed_valid` 和验证错误；
- 拓扑和缺失的关系；
- 名称/路径警告；
- WAL 解析警告；
- 当前 body 和近期活动。

使用 `task_find` 警告定位隐藏在常规候选列表之外的无效候选。当已关闭的父级会隐藏后代时，使用完整 ID 或显式路径。

## 区分警告与阻塞错误

常见的非阻塞警告：

- `name_slug_mismatch`；
- `noncanonical_task_path`；
- 缺失的 `related_to` 目标；
- 未解析的手动 WAL 文本；
- WAL 预算截断。

这些需要解释，但通常不构成移动或重写数据的理由。

阻塞条件包括：

- 重复 UUID 或有歧义的身份；
- 无效的受管字段/模式/frontmatter；
- 跨项目 Task 路径；
- 对未知/自身/循环目标的关系添加；
- 生命周期前置条件；
- 非常规或非 UTF-8 的 WAL 目标；
- 外部写入竞态；
- 项目根目录/存储冲突。

除非预期的修正状态已得到验证，否则不要通过手动编辑将阻塞错误转换为警告。

## 诊断项目和根目录故障

对于 `project_not_initialized`、`root_conflict`、`task_root_missing`、注册表、配置或 Git 策略失败，加载[项目设置](project-setup.md)。

检查最近的项目配置、嵌入根目录、分离注册表条目和当前绝对 cwd。不要根据哪个包含更近的文件来选择冲突的根目录。

在确定原始数据是否存在于其他地方之前，不要为缺失的分离根目录创建新的空根目录。未经用户同意不要删除过时的注册表条目。

## 诊断 Task 身份和元数据

对于无效候选，使用其显式路径。即使候选无法加入拓扑或接受写入，Core 也可能返回 body 和验证错误。

检查：

- 确切的完整 UUIDv7 及其在项目中的重复出现；
- 支持的 `schema_version`；
- 名称、状态、归档、时间戳、关系列表、分支、原因和 `extra` 类型；
- 重复的 YAML 键；
- 不安全的自定义 YAML 标签；
- frontmatter 分隔符和 UTF-8；
- `TASK.md` 是否为常规文件。

未知的顶层字段是有效的，应予以保留。不要将其作为清理操作删除。

重复 UUID 是身份冲突，而非格式问题。确定哪个 Task（如果有）拥有原始身份，并询问用户另一个应如何重建。永远不要分配手写替换的 UUID 同时假装来自 Core。

对于名称和路径不一致的有效 Task，当名称应变更时使用重命名。如果用户有意移动了目录，保留 UUID 身份并将路径警告视为信息性，除非明确请求了权威移动。

## 诊断关系和生命周期

读取 `metadata` 以检查已解析的拓扑、依赖、被依赖方、相关 Task 和缺失的 ID。

- 缺失的依赖会阻塞正常关闭。仅当用户确认其为过时时，恢复预期的 Task 或移除该关系。
- 缺失的相关目标会发出警告但不阻塞无关工作。当它仍然是有用的来源信息时，保留它。
- 依赖循环必须通过更改实际的依赖模型来解决，而非强制关闭任意 Task。
- 打开的后代或依赖会正确阻塞正常关闭。

强制关闭是用户拥有的生命周期绕过机制，而非修复命令。不要用它来让 `check` 变绿或隐藏损坏的后代。

## 诊断 WAL

检查确切的 `wal/` 目录和目标日期文件。Core 要求：

- `wal/` 是真实目录，非符号链接；
- 日期文件是常规文件，非符号链接；
- 现有内容为 UTF-8；
- 权威记录具有时区感知的 H2 标题。

除非用户要求规范化，否则保留未解析的手动文本。添加修正而非重写有效历史。

如果在自动 WAL 失败前结构化状态已提交，重新读取状态并仅修复 WAL。不要重复原始更新。参见 [WAL](wal.md) 了解已提交警告序列。

## 诊断竞态和部分写入

`external_write_race` 表示 Core 检测到文件在读取后被更改。重新读取最新文件，比较预期的语义变更，并仅在协调外部编辑后重试。不要用过时的字节覆盖。

`lock_target_changed` 表示用于锁定的目录在锁获取前消失或变更。再次解析当前路径和身份。

残留的 `.*.tmp` 暂存文件可能来自中断的原子写入。它们的存在不能证明哪个内容应胜出。将暂存文件与权威文件、时间戳、Git 历史和操作结果进行比较后，再决定是保留还是移除。移除具有破坏性，需要用户批准。

多文件重命名或批量放置期间的崩溃可能留下部分工作现场，即使正常验证路径对预期使用足够原子化。在进行任何后续重命名前，盘点整个受影响的 Task 树和引用。

## 仅修复已验证的问题

使用以下顺序：

1. 恢复项目/根目录发现，不必要移动 Task 数据。
2. 建立唯一的 Task 身份。
3. 仅修复 Core 读取 Task 所需的无效受管元数据。
4. 通过 Core 重新读取并检查拓扑/警告。
5. 修复 WAL 文件系统条件，不重写历史。
6. 再次运行 `task-core check`。
7. 使用 `task_read` 验证 Task，然后检查 Git 状态和普通材料。
8. 在日志记录安全后，追加 WAL 修正或修复说明。

对 Core 可以表达的任何修复使用 Core 操作。手动 frontmatter 修复是最后手段，仅用于 Core 在无效状态下无法写入的数据；保留 YAML/body 结构，并在多个值都合理时请求用户指示。

## 识别不支持的恢复

Core 不提供：

- 自动修复或 `doctor` 命令；
- 模式迁移；
- 嵌入/分离或 Git 策略迁移；
- 分布式冲突解决；
- 崩溃安全的多文件事务重放；
- 自动 UUID 冲突解决；
- 项目级重命名回滚；
- 对任意手动损坏的穷尽恢复。

诚实报告这些边界。保留证据并提供范围明确的恢复计划，而非发明命令或重写大范围的 Task 数据。
