# Agent Configuration Smells

用于配置 AI agent 的文件——`AGENTS.md`、`CLAUDE.md`、SKILL 文件、prompt rules、permission manifests、workflow docs——本身也是工程产物，也会以典型方式退化。像 [../refactoring/code-smells.md](../refactoring/code-smells.md) 里的 code smells 一样，下面每一项都是一个 _值得调查的症状_，不是自动定罪。一个很长的配置文件完全可能是正确的；一个重复的规则也可能是有意的提醒。最终判断始终是：这个 smell 是否在消耗 agent 的准确性、token 或一致性——而每一节最后的误报边界与检测信号同样重要。

这些 smell 与平台无关。不管 agent 是 Claude、Codex、Cursor 还是别的什么，底层机制都相同：有限的 context window、被当作优先级读取的指令、模型可能会也可能不会跟随的 references，以及与代码库本身漂移的配置。

## Context Bloat

**它是什么。** 每一轮都永久加载进 agent context 的低频、冗长或与任务无关的信息。典型形状是一个巨大的 system prompt，把完整风格指南、所有工具文档和详尽示例全都内联进去，于是不管当前任务是否需要，它们都在上下文里。

**为什么重要。** context 是和真实任务共享的固定预算。每一个花在常驻参考材料上的 token，都会少一个给正在编辑的文件、错误输出或推理链使用。除了预算本身，信号也会被稀释：这次任务真正相关的规则，被埋在大量无关内容之间，而模型会大致同等看待它们。臃肿的 context 还会在每一次对话中提高成本和延迟，不只是需要这些信息的那些轮次。

**检测信号。** 一个 system prompt 或常驻配置长达数千 token。整份文档被内联而不是通过链接引用。参考材料（API spec、完整风格指南、长列表）无论任务是否相关都被加载。每次会话都带着同一大段内容，尽管大多数会话根本不会碰到它的主题。

**解决方式。** 渐进式披露：保持常驻层小而精，在需要时再路由到细节。用一句摘要加链接代替内联大段内容，只有触发条件成立时才读取（关于如何写链接，见 [Blind Reference](#blind-reference)）。按需加载——把领域相关上下文放到真正需要它的任务后面。这份 Skill 自己的结构就是例子：SKILL 文件保持精简，只指向像本文这样的叶子文档，而叶子文档只有在相关信号出现时才加载。

**示例场景。** 某个 `AGENTS.md` 长到 1,200 行，因为所有库约定、所有历史事故、完整部署 runbook 都被贴进去了，理由是“这样 agent 永远都有”。结果一次修 typo 的任务也把这 1,200 行全读了，部署部分还把真正要编辑的函数挤出了注意力。

**误报边界。** 小而高频、确实跨任务的规则，应该放在常驻 context 中——那不是 bloat，那就是配置的职责。一个所有任务都依赖的 30 行 `AGENTS.md` 就是合适大小。问题不在总行数，而在 _与任务相关性加权后的长度_。应按使用频率审计，而不是按行数审计。

## Skill Leakage

**它是什么。** 任务特定知识放在了错误的抽象层级上——要么太高，以至于污染了所有人的全局 context；要么太低，以至于真正需要它的 agent 根本读不到。它是一种分层错误：知识是对的，但层级不对。

**为什么重要。** 放得太高的知识是一种 [Context Bloat](#context-bloat)——某个小众规则给所有无关任务都增加税负。放得太低更糟糕，因为它悄无声息：agent 根本不会加载它，结果违反了本来会遵守的约束，直到输出出错才被发现。二者本质上都是同一份信息被路由错了。

**检测信号。** 项目级或仓库级规则被放进了全局、用户级配置。关键且高频的约束被埋在很少加载的叶子里，层级太深。专家级边角指南放进了面向初学者的 onboarding prompt，或者反过来。一条自然受众是“只针对 subsystem X 的任务”的规则，却放在所有任务都要付费的地方。

**解决方式。** 按频率和范围分层。全局配置只保留对所有项目都成立的内容；项目配置放项目事实；skill 或 leaf 放只在触发任务中才需要的内容。让加载条件与知识真正的受众相匹配：高频且广泛的放上层，低频且狭窄的放下层——但也不要低到触发任务都找不到。

**示例场景。** 用户的全局 `~/.claude/CLAUDE.md` 里写着“在这个 repo 里，migration 用 `make db-upgrade`”。于是用户碰到的每个项目都带上了一条只适用于一个 repo 的指令，而真正属于那个 repo 的配置却保持沉默。

**误报边界。** 有些知识确实是跨切面的，即使看起来很具体也应该放在高层（比如必须绝不能漏掉的安全规则，放在常驻 context 中是合理的）。而把罕见且沉重的细节有意下沉到更深的 leaf 里，也是正确设计，不算 leakage，_只要_ 有可靠的触发条件在需要时把 agent 路由过去。

## Lint Leakage

**它是什么。** 一些规则本来就能被 formatter、linter 或 type checker 机械地强制，却又被重写成自然语言 prompt rule。比如“始终用 4 空格缩进”、“导入按字母排序”、“不要有未使用变量”、“行宽小于 88 字符”——这些都是别的工具本来就能做的事。

**为什么重要。** 确定性工具能完美、免费地执行这些规则；prompt rule 只是概率性地执行，而且要消耗 context budget 和注意力。更糟糕的是，把格式化规则写进 prompt，会诱使 agent 花推理去手工排版那些 formatter 本来会重写的内容；一旦工具配置变化，prompt 也会过时——现在 prompt 和 `.ruff.toml` 不一致，agent 两边都要照顾。用自然语言写机械规则，会稀释掉真正需要人类语言解释的那些规则。

**检测信号。** 任何 `ruff`、`eslint`、`prettier`、`black`、`gofmt` 或 `mypy` 配置已经能够表达的 prompt rule。`AGENTS.md` 里写的格式、导入顺序、空白、命名风格、未使用符号或行长规则。与仓库里已有的 `.editorconfig` 或 pre-commit hook 重复的指令。

**解决方式。** 从 prompt 里删掉这些规则，交给工具。与其在文中重写规则集，不如直接告诉 agent 去跑哪个命令（例如“提交前运行 `ruff check --fix`”）。把 prompt 空间留给工具做不到的事：架构意图、何时抽象、领域规则，以及任何只看工具配置不会想到会令人意外的行为。

**示例场景。** `AGENTS.md` 列了十五条风格规则——缩进、引号风格、尾随逗号、导入分组——而这些内容早已写在 repo 的 `ruff` 和 `prettier` 配置里。agent 费力重建这些规则，偶尔还会和 formatter 冲突，而这十五行反而挤掉了真正重要的规则：“不要在 request handler 里调用 billing API”。

**误报边界。** 当你之所以提到某条规则，是因为工具的行为会让 agent 感到意外，这当然合理——例如指出某个非标准 formatter 配置有不寻常的行为，或者某个特定 lint rule 被有意关闭并说明原因。这个 smell 指的是重述工具已经静默、正确执行的内容，而不是解释工具与预期不一致的地方。

## Blind Reference

**它是什么。** 一个指向另一个文件、URL 或文档的引用，却没有说明 _什么时候_ 该读、_它能证明什么_、以及 _它不覆盖什么_。比如没有上下文的“另见：`architecture.md`”就是 blind reference；只有一串链接、没有路由逻辑的列表也是。

**为什么重要。** 决定是否要花一次工具调用和 context budget 去打开一个 reference 的 agent，需要知道理由。没有触发条件，agent 要么忽略链接（漏掉必要上下文），要么防御性地全读（又把 [Context Bloat](#context-bloat) 带回来了）。没有用途说明的 reference 也无法推理：agent 不知道读到的文档是权威的、示例性的，还是仅仅相关度很低，因此无法判断自己读到的答案该怎么加权。

**检测信号。** 没有条件说明的 “see also” 或 “refer to”。在某个 heading 下摆出一堆链接，却没有逐项路由逻辑。引用没有说明目标文件能证明什么，因此 agent 无法判断找到的答案是否权威。指向外部 URL 却没说要提取什么或为什么的指针。

**解决方式。** 每个 reference 都要配上触发条件和用途：_什么时候_ 该跟过去，_到那里_ 能得到什么。这个 skill 里的 index 和 routing table 就是范例——每一行都把代码里的一个 signal 对应到处理它的文档。好的 reference 读起来像“当 X 发生时，读 Y，它能建立 Z”，而不是“见 Y”。

**示例场景。** 一个 workflow 文档最后写着 “Related: `deploy.md`, `rollback.md`, `secrets.md`”。agent 在做代码修改时，无法判断这些文件现在是否相关，于是要么全读一遍（浪费 budget），要么一个都不读（漏掉 `deploy.md` 会说明的部署前置条件）。

**误报边界。** 一个 curated index 如果周围的文字已经提供了路由逻辑，就不必在每一行都重复触发条件——上下文在这一节里已经建立好了。真正的可选延伸阅读，只要明确标成 optional，让 agent 知道可以安全跳过，也没有问题。

## Init Fossilization

**它是什么。** 由初始化或 scaffolding template 生成、但后来已经与真实代码库漂移的配置。这个配置描述的是一个早已不存在的项目：引用已删除文件的规则、针对废弃流程的工作流步骤、对不再使用的工具的权限、以及与当前目录树不符的目录结构说明。

**为什么重要。** 陈旧配置会以一种比“没有配置”更危险的方式误导人。agent 会把每条指令都当成当前仍然有效，所以一条指向已删除模块的规则会让它去寻找不存在的东西，一条针对已退役 CI pipeline 的工作流步骤会让它执行或期待毫无作用的动作。僵化的权限则更隐蔽：它们可能授予本该撤销的工具访问权，或者阻止项目现在依赖的工具。agent 无法知道配置在说谎；它只会在虚构上自信推理。

**检测信号。** 规则里写的文件或目录已经不存在（把配置里的路径和实际树比对一下）。工作流步骤引用了团队已经放弃的流程、服务或命令。权限项指向没人再用的工具。一个“project structure” 部分与 `ls` 的结果对不上。典型的 `init` 模板措辞，却从未为这个项目真正编辑过。

**解决方式。** 定期审计，最好绑定到一个固定触发点（发布、依赖升级、季度回顾）。把每个路径和工具引用都与当前现实进行核对，删掉不再成立的部分。当某次变更删掉了某个文件或流程时，应在同一次变更里同步清理配置。把配置当作需要维护的代码，而不是一次写好就不动的文档。

**示例场景。** 某个 repo 的 `CLAUDE.md` 仍然写着“用 `make test` 跑测试，并在 `htmlcov/` 查看 coverage”，但项目六个月前就已经改成直接用 `pytest`，同时删掉了 Makefile。agent 运行 `make test` 后得到 “no such target”，结果浪费了一次轮次去重新发现项目到底怎么构建。

**误报边界。** 并非所有不一致都是 fossilization——配置可能 _故意_ 描述一个正在迁移中的目标状态，或者文档化一条将来要执行的约定。删除之前，先确认这条引用是否本来就应该描述当前现实；一个进行中的迁移，确实可能出现配置先于代码。

## Conflicting Instructions

**它是什么。** 两条规则在同一件事上互相矛盾，却没有机制说明谁优先——没有 scope，也没有优先级顺序。全局规则说“总是做 X”，项目规则说“永远不要做 X”，但没有任何东西告诉 agent 怎么调和它们。

**为什么重要。** 面对冲突却没有解决规则时，agent 可能随机选一条，或者在不同轮次之间来回摇摆，或者跟随它最近读到的那条。结果就是行为不可预测、不可审计——你无法推断 agent 会做什么，因为配置本身没有决定它。冲突也会侵蚀对整套配置的信任：一旦 agent 见过两条规则冲突，所有规则都会变成“也许这条也被覆盖了”的候选对象。

**检测信号。** 在不同文件或不同 section 中，对同一主题出现 always/never 配对。规则的适用范围重叠，却没有说明哪个范围更窄或优先级更高。配置用堆砌例外（“always X, except when Y, unless Z”）来解决冲突，而不是重构结构。全局、项目、任务等分层配置之间没有明确优先级说明。

**解决方式。** 明确 scope 和 priority。声明当不同层级发生冲突时，哪一层优先（通常是越具体的越优先：task over project over global）。给每条规则清晰的适用范围，避免两条规则都声称适用于同一情境。最重要的是，从源头解决冲突，而不是用第三条规则去调停前两条——如果两条规则真的冲突，那么要么其中一条是错的，要么它们属于不同的 scope，修复方式是纠正或拆分它们，而不是再加一条中介规则。这份 skill 自己的配置之所以这样写，就是为了避免这种问题。

**示例场景。** 全局配置说“为新函数总是写测试”。某个项目配置说“这是原型，跳过测试”。两者都没有互相引用。结果 agent 有时会给部分函数写测试，有时不会，完全取决于哪条指令在当前回合更显眼，最终 test suite 变得一半完整、一半混乱。

**误报边界。** 看起来冲突、但作用在不同且明确说明过的 scope 上的规则，不算冲突——“默认用 X” 再加上“在 subsystem Y 中改用 Z”，只要范围明确且不含糊，就是合理的总则加局部例外结构。真正的 smell 是没有解决的冲突，而不是一条规则和它的 scoped exception 共存。
