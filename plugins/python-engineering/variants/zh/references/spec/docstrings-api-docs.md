# 文档字符串与 API 文档

一旦类型注解承载了静态契约，文档字符串的职责就发生了变化。它不再是记录参数类型和返回类型的地方——签名已经做到了这一点，而且是机器可检查的，重复它只会创建一个会不同步的副本。文档字符串的职责是签名*无法*表达的一切：函数在领域中的含义、它承诺和假设了什么、它触及了外部世界的哪些部分，以及如何正确使用它。核心技能是将每条信息放在它所属的位置，并且只在文档字符串为读者添加了他们还看不到的东西时才编写它。

## 信息所属的位置

现代 Python API 将其文档分布在多个表面上，而大多数文档问题实际上是放置问题——正确的事实放错了位置，或者相同的事实放在三个地方。对于任何一条信息，首要问题不是"我如何措辞"，而是"哪个表面拥有它"。

| 信息 | 所属表面 | 原因 |
|-|-|-|
| 参数和返回值的*类型* | 签名/注解 | 机器检查、IDE 可读；在文档字符串中复制会不同步 |
| 领域含义、业务规则 | 文档字符串 | 类型系统无法表达一个值*意味着*什么 |
| 值范围、单位、形状、数据类型 | 文档字符串或 `Annotated`/`Field` | 取决于机器是否需要读取它 |
| 跨参数约束 | 文档字符串或验证器 | 单个注解无法关联两个参数 |
| 副作用、资源生命周期 | 文档字符串 | 调用者必须了解 IO、网络、数据库、全局状态 |
| 异常语义 | 文档字符串 | Python 类型不表达"抛出" |
| 机器可读的字段约束 | 模式元数据（`Field(...)`） | 为验证和生成的 API 文档提供输入 |
| 工作示例 | 文档字符串、README 或测试 | 最好也经过执行，这样它们不会过时 |
| 版本变更、弃用 | 发布说明、`warnings.deprecated()` | 同时服务于读者、检查器和运行时 |

最常见的错误是重复更权威的表面已经拥有的事实——在 `Field(min_length=3)` 已经声明了的情况下用散文写 `min_length=3`，或者重述注解已经给出的类型。

## 文档字符串何时增加价值

当文档字符串告诉读者签名无法表达的东西时，它才值得存在。最清晰的信号：

- **语义。** 返回值*意味着*什么？`-> str` 说明它是一个字符串；文档字符串说明它是"适用于不区分大小写查找的规范化形式，而不是显示名称"。
- **类型无法持有的约束。** 必须为奇数的窗口、必须非空的列表、必须为 UTC 的时间戳——注解说 `int`、`list`、`datetime`，文档字符串承载其余部分。
- **副作用。** 返回值之外任何可观察的事情：数据库写入、网络调用、修改的参数、填充的缓存、创建的文件。调用者需要这些才能安全地使用函数，而且没有注解能揭示它。
- **异常语义。** 调用者应处理的异常及其含义——不是所有可能传播的异常的详尽列表，而是作为契约一部分的那些。

```python
def normalize_username(raw: str) -> str:
    """返回用于账号查找的规范化用户名。

    首尾空白被去除，结果进行大小写折叠，因此适用于不区分大小写的比较。
    不保证结果是有效的显示名称。
    """
    return raw.strip().casefold()
```

这里 `raw: str -> str` 已经在签名中；文档字符串添加了查找语义、大小写折叠规则和明确的限制（"不是显示名称"）。糟糕的版本则重述函数名称（"规范化用户名"）或重新列出签名类型（"raw 是 str，返回 str"）。

## 文档字符串何时是噪音

重复签名的文档字符串比没有更糟糕——它增加了维护成本和一个最终会与第一个矛盾的第二个真理来源。在以下情况下跳过或精简文档字符串：

- 它只会重述注解中已有的类型（`user_id (str): 用户 ID`）。
- 它只会回显函数名称（`def save_user(...): """保存用户。"""`）。
- 函数是一个小型私有辅助函数，其名称和签名已经说明了一切。
- 它会重复模式元数据中已声明的约束。

覆盖率驱动的文档字符串命令（"每个函数都必须有文档字符串"）倾向于制造这种噪音。Linter 可以检查文档字符串*存在*并且其格式良好；它无法检查它是否说了任何有用的东西。将文档字符串 lint 视为低级卫生门控，决不能作为文档良好的证据。

## 文档字符串风格

三种格式被广泛使用，该技能不强制使用某一种——选择取决于项目的文档需求，而非时尚。

- **Google 风格**使用 `Args:`、`Returns:`、`Raises:` 部分。它轻量、可读，非常适合通用工程 API 和后端服务。在有完整类型注解的情况下，从部分中省略类型（`host: 绑定目标`，而不是 `host (str): 绑定目标`）。
- **NumPy / numpydoc 风格**使用带下划线的部分（`Parameters`、`Returns`、`Notes`、`Examples`）。它结构更强，适合科学和数据 API，其中形状、数据类型、单位、数学定义和示例有真正的分量。对于简短的业务辅助函数来说过于繁重。
- **reStructuredText / Sphinx**使用字段列表（`:param x:`、`:returns:`），并与 Sphinx 文档站点、交叉引用和版本指令集成。适用于发布的库和框架；对于内部代码，其语法噪音超过收益。

无论何种风格，在有注解的情况下规则是不变的：不要将类型写入散文。让签名拥有类型，让文档字符串拥有含义。

## 模式驱动的文档

对于基于 Pydantic 和 FastAPI 等工具构建的 Web API 和数据模型，类型注解和字段元数据成为验证、序列化*和*生成文档的单一事实来源。字段的类型、其约束（`Field(gt=0, le=100)`）和其描述（`Field(description=...)`）流入生成的 OpenAPI/JSON 模式。在这个世界中，文档字符串的领域缩小到模式无法表达的部分——更高级的业务语义、事务行为和跨字段关系：

```python
class CreateOrderRequest(BaseModel):
    sku: Annotated[str, Field(min_length=3, description="公共库存单位。")]
    quantity: Annotated[int, Field(gt=0, le=100, description="请求数量。")]
```

约束和每个字段的描述位于 `Field(...)` 中，直接输入模式。然后在端点上的文档字符串解释操作*做什么*——"在写入订单前预留库存；预留失败则不会创建订单"——这是任何字段元数据都无法捕获的。需要防范的失败模式是同时维护 `Field(description=...)`、模型文档字符串和手写的 OpenAPI 描述中的同一句话。选择一个事实来源，让其他引用它。

## 文档站点与文档字符串

文档字符串记录一个对象；它不能替代文档站点。教程、操作指南、设计原理和迁移说明是叙述性文档，再多的逐函数文档字符串也无法替代。像 Sphinx 配合 `autodoc` 这样的工具可以将文档字符串拉入 API 参考，但自动生成的参考不等同于好的文档——叙述性内容仍然需要手动编写。一个实际的注意事项：`autodoc` 会导入模块来读取它，因此任何导入时的副作用都会在文档构建期间运行；保持模块导入干净（与 [structure](references/project/structure.md) 要求的相同纪律），文档构建就能保持可预测。
