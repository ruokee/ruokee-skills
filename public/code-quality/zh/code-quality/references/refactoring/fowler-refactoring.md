# Fowler 风格重构（Fowler-Style Refactoring）

## 什么是重构

在 Martin Fowler 的精确意义上，重构是**在不改变外部可观察行为的前提下重组现有代码**。输入产生相同输出；调用方依赖的副作用被保留。变化的是内部结构：名称更清晰，职责转移到更合适的所有者，长函数拆分为命名阶段，重复知识合并到一处。

这比随意的"我重构了代码"（通常意味着"我重写了代码并顺便改变了一些行为"）更严格。当可观察行为发生变化时，这不是重构——而是特性变更、缺陷修复或性能优化。将这两类活动分开正是这一学科的核心。

Fowler 的《重构》将**代码坏味**目录（暗示结构问题的症状）与**命名重构**目录（处理这些问题的、保持行为不变的小型变换）配对。坏味告诉你*在哪里*找问题；重构告诉你*如何*以可控步骤修复。坏味方面请参见 [code-smells.md](./code-smells.md)，变换方面的独立技术文档如 [extract-function.md](./extract-function.md)。

## 纪律：小步骤，一次只做一件事

决定性的实践是每一步的大小。重构是一系列微小的变换，每一步都足够小以至于显而易见是正确的，并且每一步之后测试再次变绿。你重命名一个变量，运行测试。提取一个函数，运行测试。将其移动到另一个类，运行测试。如果某一步出了问题，你确切知道是哪个变更引起的，因为你只做了一个变更。

这与"让我重组整个模块然后看看它是否还能工作"截然相反。未经验证的大规模重组正是重构出问题的地方：当测试在最后失败时，你无法判断是引入了一个缺陷还是五个，安全的做法是放弃这整个工作。小步骤让你持续处于可工作状态，因此你可以在任何时刻停止，仍然拥有可发布的代码。

一次只做一件事也意味着**绝不在同一步中混合重构和特性变更**。Fowler 描述了"戴两顶帽子"：添加特性时，不清除结构；重构时，不添加能力。你刻意地、频繁地切换帽子，但你始终知道戴的是哪一顶。混合两者很危险，因为一次组合变更后的绿色测试不说明任何问题——你无法将结果归因于任一活动。实际上，这意味着重构和行为变更应分属不同的提交，或至少被明确区分。安全机制请参见 [safe-refactoring.md](./safe-refactoring.md)。

## 何时重构

重构不是你安排到以后的一个独立阶段。它融入日常工作，有三个自然时机：

- **预备性重构。** 在添加特性之前，重塑代码以便特性能够干净地适配。"让变更变得容易，然后做容易的变更。"如果一个函数需要一个新参数，而它的三个调用方各自需要笨拙的处理，先重构出一个接缝，然后通过它添加特性。这是价值最高的时机，因为重组能立即得到回报。
- **理解性重构。** 当你终于理解了一段令人困惑的代码时，将这份理解沉淀回代码中——重命名误导性的变量，提取未命名的概念，添加缺失的结构。反正你也要花精力理解它；捕获结果意味着下一个读者不必再这么做。
- **清理性重构。** 当你接触比必要更混乱的代码时，让它比你发现时好一点点。童子军规则。这需要判断力：清理挡在你面前的部分，而不是整个模块。

## 何时不该重构

- **没有测试，也没有其他安全网。** 重构依赖于一种快速确认行为得到保留的方式。没有测试、特征测试、类型检查或某种可重复的验证，你并不是在重构——而是在编辑并祈祷。要么先添加安全网（参见 [safe-refactoring.md](./safe-refactoring.md)），要么不要动结构。
- **截止日期压力且没有安全网。** 在真正的截止日期前，大规模的投机性重组是个糟糕的赌注。小的、安全的步骤没问题甚至有帮助；但可能完不成的有风险的重构则不可以。
- **即将被删除或整体替换的代码。** 打磨下周就要丢弃的结构是浪费精力。YAGNI 也适用于清理。
- **当它会改变你无法验证的行为时。** 如果代码涉及公共 API、序列化模式或 CLI 参数，"只是整理一下"可能悄无声息地破坏消费者。先定义可观察的契约，否则将该工作视为行为变更而非重构。

## 坏味指方向，重构下刀子

两个目录之间的关系是诊断性的。一个坏味——长函数、重复代码、依恋情节的方法——是一个*症状*，而不是一个判决。它让你仔细看，而不是机械地行动。一旦你确认了底层的结构问题，你选择一个命名的重构来处理它，并以小步骤应用它。坏味是问题；重构是可能的答案之一。这防止你"修复"实际上并未损坏的东西，也防止你在重命名就够了的情况下使用重型变换。

在代理辅助开发中，坏味然后再重构的循环更重要，而非更不重要：代码积累更快，因此结构问题出现得更快，重写而非变换的诱惑也更强。保持行为不变的小步骤的纪律，正是防止快速移动的代码漂移到无人能验证状态的关键。
