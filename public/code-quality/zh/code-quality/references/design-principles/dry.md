# DRY — 不要重复自己（Don't Repeat Yourself）

## 它是什么

DRY 是软件中最容易被误解的原则之一。其原始表述（Hunt 和 Thomas，《程序员修炼之道》）是关于*知识*的："每一条知识在系统中必须拥有单一、明确、权威的表示。"它不是指"不能有两段看起来相似的代码。"真正值得去重的是一条领域规则、一个模式、一个协议、一个转换、一个错误处理策略——一个如果发生变化，就必须在所有地方同步修改的决策。

DRY 所警告的危险是：同一知识分散在多处，可能逐渐不同步。当一条税务规则、一个权限检查、一个状态转换或一个验证条件存在于三个文件中时，未来的修改可能会漏掉其中之一，系统就会自相矛盾。

## 背后的假设

- 当同一知识出现在多个地方时，未来的编辑最终会更新一部分而遗漏另一部分。
- 单一的权威表示让变更可以在一个地方发生。
- 但是，看起来相似的代码不一定表达相同的知识。两段代码今天可能因完全无关的原因具有相同的形态，明天则可能被不同的力量驱动。

最后一点将好的 DRY 与坏的 DRY 区分开来。该原则是关于知识的，而不是关于文本的。

## 知识重复 vs 巧合相似

问：如果一个副本背后的需求发生了变化，另一个副本*必须*跟着改变吗？如果是，它们编码的是同一知识——去重。如果不是，它们只是碰巧当前相似——保持原样。

值得消除的真正知识重复示例：

- 同一条业务规则（价格计算、资格判定、状态机转换）在多个地方实现。
- 同一个 schema、API 合约或数据模型在客户端、服务器和测试中重复。
- 同一个重试或错误处理策略在多个调用点复制粘贴。
- 同一个数据转换规则在多个函数中独立表达。

通常可以保持分离的巧合相似示例：

- 两个验证函数碰巧有相同的结构，但验证不同的概念，且将独立演进。
- 两个请求处理器有相似的模板代码，但业务逻辑无关。
- 看起来重复的测试设置，但锁定的是不同的场景。

## 何时抽象，何时容忍重复

当重复的东西确实是一条规则时，立即去重，尤其是高风险知识：安全、权限、金钱、协议合约、数据一致性，以及任何 schema 的唯一真实来源。对于这些情况，不要等待——分歧的成本太高。关于例外结构，请参见 [rule-of-three.md](./rule-of-three.md)。

当两个片段相似但有不同变更原因，且共享抽象的形态尚不明确时，容忍重复。过早的统一会选错分割点。从两个案例中提取的"通用"函数往往会随着第三、第四个案例的到来而增加布尔标志、模式参数、回调、特殊分支和隐式前提条件——最终比当初的重复更难维护。这与 [kiss.md](./kiss.md) 存在直接张力：一个带有许多参数和分支的抽象，其认知负荷可能超过两个清晰、独立的副本。

## 错误的 DRY

两种主要失效模式，在智能体生成的代码中都很常见：

1. **过早抽象（Premature abstraction）**。在了解变化方向之前将两个巧合相似的片段统一。结果是一个参数化怪物。当你发现这种情况时，修复方法是内联该抽象，再次复制代码，等待真正的分割点显现。

2. **参数化/薄包装器泛滥（Parametric / thin-wrapper proliferation）**。以"消除重复"为名将一两行代码提取为许多小型包装函数。这很少隐藏真正的复杂度。它增加了命名负担、跳转成本、更深的调用栈，以及——在 Python 中——真实的每次调用的开销（因为每次调用都会构建一个帧）。将一个单一表达式重命名为同义词的包装器是浅辅助函数，而非去重。参见 `references/refactoring/thin-wrapper-function.md` 了解如何识别和展开这些包装器。

在提取辅助函数之前，问：它是否提供了一个稳定的语义边界、隐藏了真正的复杂度、或携带了可复用的策略？如果唯一的答案是"它被用了两次"，那还不是理由——这正是三的原则（Rule of Three）的作用。

## 在 Python 中

- 首先去重领域概念、schema 和协议，而不是局部代码形态。
- 对于重复的 schema/API/模型定义，使用单一真实来源：`dataclass`、`TypedDict`、Pydantic、OpenAPI 规范或代码生成。
- 对于实现相同但有不同业务原因的两个函数，允许重复直到变化方向明确。
- 优先使用模块级函数、表驱动映射、`Protocol` 和策略函数，而不是构建类层次结构来共享代码。
- 表（table）和分发映射（dispatch map）是折叠真正重复知识（每个案例一行）的绝佳方式，无需发明继承树。
