# 观察者模式（Observer）/ 发布-订阅（Pub-Sub）

## 意图（Intent）

让一个对象（主题）在状态改变或事件发生时通知多个依赖者（观察者），而主题无需知道它们是谁。它建立了一种可在运行时改变的一对多依赖关系。

## 解决的问题

当系统的多个部分必须对某些事件做出反应——缓存必须失效、UI 必须刷新、审计日志必须记录——从源对象直接调用每个反应者的做法将源对象与所有反应者耦合在一起。添加新的反应者意味着编辑源对象。观察者模式反转了这种关系：反应者订阅，源对象只负责发布。

## 结构和参与者

- **主题 / 发布者（Subject / publisher）**：持有订阅列表并发出通知。
- **观察者 / 订阅者（Observer / subscriber）**：注册兴趣并处理通知。
- **事件（Event）**：描述所发生事情的载荷。

在 Python 中，主题通常是一个按键值事件类型索引的事件总线：

```python
class EventBus:
    def __init__(self) -> None:
        self._subscribers: dict[type[Event], list[Callable[[Event], None]]] = {}

    def subscribe(self, event_type: type[Event], handler: Callable[[Event], None]) -> None:
        self._subscribers.setdefault(event_type, []).append(handler)

    def publish(self, event: Event) -> None:
        for handler in list(self._subscribers.get(type(event), [])):
            handler(event)
```

## 需要明确决定的设计关注点

以下是观察者模式实现容易出错的地方。没有单一的正确答案，但每个问题都必须是有意选择，而非偶然。

- **订阅者生命周期**：谁取消订阅，何时取消？比订阅者存活更久的订阅会泄漏内存并使已死亡的对象保持活跃。考虑对处理器使用 `weakref`，或返回明确的取消订阅句柄 / 使用上下文管理器。
- **错误传播**：如果一个处理器抛出异常，后续处理器是否仍继续运行？吞掉错误会隐藏问题；让错误传播则允许一个坏的订阅者破坏发布。做出决定并记录下来——常见做法是：隔离每个处理器的错误，记录日志，继续执行。
- **顺序**：订阅者按注册顺序运行，还是顺序未定义？不要让调用方依赖顺序，除非你保证它。
- **可重入性**：处理器能否在分发期间发布、订阅或取消订阅？遍历订阅者列表的副本（如上所示）可以避免迭代期间变动的错误。
- **同步 vs 异步**：同步进程内通知最简单。对于 `async` 处理器，你必须选择顺序 `await` 还是 `asyncio.gather`，以及如何处理阻塞或永不完成的处理器。

## 何时使用

- 多个独立的反应者必须响应事件，且反应者集合会变化或对源对象不可知。
- 你想将领域操作与其副作用（邮件、指标、缓存）解耦。
- 关系确实是一对多且动态的。

## 何时不使用

- 只有一个固定的反应者——直接调用它。事件总线增加间接性并隐藏控制流，却没有解耦的好处。
- 严格的顺序和清晰的顺序工作流比解耦更重要——显式的管道可读性更好。
- 需要跨进程投递：进程内观察者模式不提供投递、重试或持久性保证。使用具有明确定义语义的真正的消息系统（队列、代理）。

## 失败模式

- **隐藏的控制流**：一次 `publish` 触发了级联效应，没人能通过阅读调用点追踪到。
- **内存泄漏**：遗忘的订阅永远持有引用。
- **意外的可重入性**：处理器在分发期间修改了主题，破坏了迭代。
- **静默失败**：吞掉的异常意味着事件"已发生"，但其效果从未执行。

## 与其他模式的关系

当事件是被具体化的对象，可以被排队或重放时，观察者模式与 [command.md](command.md) 有重叠。[mediator] 协调是相反的姿态：中介者集中协调逻辑，而观察者模式将其分发。对于跨进程扇出，此模式让位于消息代理基础设施而非内存中的总线。
