# Fowler-Style Refactoring

## 它是什么

在 Martin Fowler 的精确定义里，refactoring 是 **在不改变外部可观察行为的前提下重组现有代码**。输入依然产生相同输出；调用方依赖的副作用依然发生。改变的是内部结构：名字更清楚了，职责被移到更合适的拥有者那里，长函数拆成了命名阶段，重复知识收敛到一个地方。

这比平时口语里那种“我重构了代码”要窄得多，后者常常其实意味着“我重写了一遍，还顺手改了点行为”。一旦可观察行为变了，那就不叫 refactoring - 那是功能变更、修 bug，或者性能优化。把这些活动分开，正是这套纪律的核心。

Fowler 的 _Refactoring_ 把 **code smells**（暗示结构问题的症状）和 **named refactorings**（以小步、保行为方式修正问题的变换）配成一套目录。smell 告诉你 _往哪看_；refactoring 告诉你 _如何在受控步骤中修_。关于 smell 的一侧，见 [code-smells.md](./code-smells.md)；关于具体技术的一侧，见像 [extract-function.md](./extract-function.md) 这样的单项文档。

## 纪律：小步，一次只做一件事

这套实践的关键在于每一步的大小。一次 refactoring 应该由一串很小的变换组成，每一步都小到明显正确，而且每做一步，测试都要重新变绿。你改一个变量名，跑测试。你提一个函数，跑测试。你把它移到另一个 class，跑测试。如果哪一步坏了，你会精确知道是哪次变更引起的，因为你一次只做了一件事。

这和“让我把整个模块都重新组织一下，之后再看它能不能跑”正相反。大规模、未验证的重组，正是 refactoring 出问题的地方：当最后测试失败时，你根本不知道自己引入了一个 bug 还是五个 bug，于是最安全的选择就变成把整段工作丢掉。小步做能让你始终停留在可工作的状态，因此可以随时停下，而且代码仍然可交付。

一次只做一件事还意味着 **绝不在同一步里混合 refactoring 和功能变更**。Fowler 说的是“穿两顶帽子”：加功能时不要顺手整理结构；做 refactoring 时不要顺手加能力。你可以有意识地、频繁地切换帽子，但每一刻都要清楚自己戴的是哪一顶。混着做很危险，因为一次混合变更即使测试是绿的，也说明不了任何东西 - 结果无法归因。实践上，这意味着 refactoring 和行为变更应该分成不同 commit，至少也要分别说明。安全机制见 [safe-refactoring.md](./safe-refactoring.md)。

## 何时重构

Refactoring 不是一个安排在后面的独立阶段。它本来就缝在日常工作里，最自然的时机有三种：

- **Preparatory refactoring.** 在加功能之前，先把代码整理到这个功能能自然落进去。“先把修改变简单，再做简单的修改。”如果某个函数需要一个新参数，而它的三个调用者都要分别做别扭处理，那就先重构出一个 seam，再从那里加功能。这是价值最高的时机，因为结构调整会立刻见效。
- **Comprehension refactoring.** 当你终于理解了一段难懂的代码，就把这种理解折回到代码里 - 改掉误导性的变量名，提取那个一直没有名字的概念，补上缺失的结构。你本来也已经为理解它付出了代价；把结果留下，下一位读者就不用再付一次。
- **Cleanup refactoring.** 当你触到一段比它该有状态更乱的代码时，顺手让它比你接手时更好一点。也就是 boy-scout rule。但这要受判断约束：只清理挡住你的部分，不是整个模块。

## 何时不要重构

- **没有测试，也没有别的安全网。** refactoring 依赖一种快速确认行为是否被保留的方法。没有测试、characterization tests、类型检查，或者某种可重复验证方式时，你做的不是 refactoring - 你是在改代码并祈祷。要么先把安全网补上（见 [safe-refactoring.md](./safe-refactoring.md)），要么就不要碰结构。
- **赶工且没有安全网。** 真正的 deadline 压力下，做大范围、投机性的重组并不是好赌注。小而安全的步骤是有帮助的；可能做不完的高风险重组不是。
- **你马上就要删掉或整块替换的代码。** 给下周就会扔掉的结构打磨表面，是浪费精力。YAGNI 也适用于清理。
- **它会改变你无法验证的行为。** 如果代码碰到 public API、serialization schema 或 CLI arguments，“只是整理一下”也可能悄悄破坏调用方。要么先定义可观察契约，要么就把这件事当成行为变更，而不是 refactoring。

## smell 指路，refactoring 动刀

两套目录之间的关系是诊断性的。一个 smell - 长函数、重复代码、特征嫉妒的方法 - 只是一个 _症状_，不是判决。它只是在提醒你认真看看，而不是机械地动手。一旦你确认了底层结构问题，就选择一个能处理它的命名 refactoring，并以小步应用。smell 是问题；refactoring 是某种答案。这样就不会“修复”那些其实没坏的东西，也不会为了一个只需要改名的地方上来就做重型变换。

在 agent-assisted development 里，这个“先看 smell 再 refactor”的循环更重要，不是更不重要：代码生成更快，所以结构问题出现得更快；而且人更容易冲动地直接重写，而不是循序变换。保住行为不变的小步纪律，正是让快速前进的代码不至于漂到无法验证状态的关键。
